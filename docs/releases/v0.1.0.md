# v0.1.0 — Index Core

**Released**: 2026-05-30 | **Git tag**: `v0.1.0` | **Tests**: 76

First milestone — the in-memory search index engine. No filesystem, no daemon, no CLI yet.

## What was built

| File | Purpose |
|------|---------|
| `FileRecord.swift` | Codable + Sendable data model (id, name, path, size, dates, extension) |
| `Trie.swift` | O(k) prefix matching with Unicode scalar granularity |
| `FullSubstringMap.swift` | O(1) arbitrary substring lookup for names <= 64 chars |
| `TrigramIndex.swift` | Trigram -> posting list fallback for names > 64 chars |
| `PinyinIndex.swift` | Chinese filename search via CFStringTokenizer + CFStringTransform |
| `InMemoryIndex.swift` | Actor unifying all structures with thread-safe reads/writes |
| `ProductConfig.swift` | Product name configuration from PRODUCT.toml |

## Design decisions

- **NFC normalization** on all filenames (`precomposedStringWithCanonicalMapping`)
- **Case-insensitive** search by default, original case preserved for display
- **Zero external dependencies** — pure Swift + Foundation/CoreServices
- **Speed over memory** — M4 unified memory is the target platform

## Known issues

- No persistence — index rebuilt from scratch on every launch
- No filesystem integration — FileRecord objects created only in tests
- TrigramIndex is a correctness fallback, not optimized for performance

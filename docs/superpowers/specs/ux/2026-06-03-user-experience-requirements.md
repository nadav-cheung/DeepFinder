# DeepFinder — User Experience Requirements

**版本**：1.0
**日期**：2026-06-03
**状态**：✅ 已批准
**范围**：用户生命周期 UX 需求清单，覆盖发现到推荐的完整旅程

---

## 1. 概述

本文档从用户视角定义 DeepFinder 的体验需求。现有 158 项技术 REQ 描述「系统能做什么」（what），本补充覆盖「用户怎么感知、怎么交互、怎么流失」（how it feels）。

### 1.1 方法论

基于竞品 UX 调研（Everything、Spotlight、Alfred、Raycast、Recoll/DocFetcher、FileMinutes、ProFind）、macOS HIG 分析、桌面应用用户旅程最佳实践，按 **用户生命周期阶段** 组织需求。

**用户旅程（User Journey）vs 用户流程（User Flow）**：
- **用户旅程（NNGroup 方法论）**：高层级、跨渠道、关注情感体验、覆盖长期关系。回答「用户从发现到推荐的整体感受是什么」。
- **用户流程（User Flow）**：产品内部的具体交互步骤，关注单一任务完成路径。回答「用户点击这个按钮后发生了什么」。
- 本文档覆盖两者：生命周期阶段（§3-§10）的「关键洞察」侧重用户旅程，各阶段内的 UX 需求表侧重用户流程。

**NNGroup 框架补充**：用户旅程（Journey）描述用户跨渠道、长时间的高层次体验，包含情绪和想法；用户流程（Flow）描述产品内完成特定任务的具体交互步骤。本需求文档聚焦旅程层面，具体交互流程见 docs/ 用户指南。

### 1.2 目标受众

- **开发者**：作为后续开发的指导，评估哪些 UX 功能缺失、优先级怎么排
- **OSS 用户/社区**：作为面向用户的体验承诺，让用户知道产品能做什么、怎么用

### 1.3 用户画像

| 画像 | 描述 | 核心诉求 | 每日使用频率 |
|------|------|---------|------------|
| **开发者** | 写代码、管项目、找配置文件和日志 | 速度、精确、可脚本化 | 50-200 次 |
| **知识工作者** | 管文档、PDF、设计稿、媒体文件 | 可靠、直观、Quick Look | 10-30 次 |
| **高级 Mac 用户** | 管大量文件、追求效率工具 | 快捷键、自定义、无阻碍 | 30-80 次 |

### 1.4 生命周期总览

```
发现 → 安装 → 首次运行 → 核心使用 → 探索配置 → 高级功能 → 更新维护 → 推荐
 5%      15%     25%       30%        10%       8%       4%      3%
```

（百分比 = 各阶段 UX 投入权重）

### 1.5 优先级定义

| 优先级 | 含义 |
|--------|------|
| P0 | 发布阻断 — 没有它用户会流失 |
| P1 | 重要 — 显著影响用户满意度 |
| P2 | 增强 — 锦上添花，提升口碑 |
| P3 | 未来 — 后续版本考虑 |

## 2. 竞品 UX 基准

### 2.1 竞品对标矩阵

| 维度 | Everything (Win) | Spotlight (macOS) | Alfred | Raycast | FileMinutes | Fenn | ProFind | **DeepFinder 目标** |
|------|-----------------|-------------------|--------|---------|-------------|------|---------|-------------------|
| 查询延迟 | < 1ms | 50-100ms | ~100ms | ~200ms | ~100ms（依赖 Spotlight） | ~100ms | ~50ms（原生自定义索引） | **< 50ms** |
| 首次索引 | < 5s（读 MFT） | 后台静默 | 首次需等 | 首次需等 | 依赖 Spotlight（无独立索引） | 需建索引 | 需手动建索引（可调度） | **< 30s 增量可见** |
| 权限需求 | 无（读 MFT） | 无（系统级） | 无 | 无 | FDA（内容搜索需） | FDA | FDA（全盘搜索） | **FDA（引导获取）** |
| 全局热键 | Win+Q / 自定义 | Cmd+Space | ⌥Space / 自定义 | 自定义 | 有（键盘优先设计） | 有 | 有 | **⌃⌘K + 自定义** |
| 内容搜索 | 无（极慢） | 有（不可靠） | 需 Powerpack | 有 | 有（PDF/DOCX/XLSX 等 12+ 类型） | ✅ AI 深度内容 | 有（文本 + 元数据） | **有（高亮匹配）** |
| Quick Look | N/A | ✅ Space 键 | ✅ | ✅ | ✅ Space 键 | ❌ | ✅ | **✅ Space 键** |
| AI 语义搜索 | 无 | 有（Apple Intelligence） | 无 | 有（Pro） | 无 | ✅ AI 深度内容 | AI 图像搜索（macOS 12+，自然语言描述图片内容） | **有（多模型，含 Vision Tagging）** |
| 更新机制 | 手动检查 | 系统更新 | Sparkle | 内置 | 网站下载 | 内置 | 手动检查 | **Sparkle + Homebrew** |
| 安装方式 | 便携 EXE | 系统内置 | DMG 拖拽 | DMG 拖拽 | DMG | DMG | DMG | **DMG + Homebrew** |
| 内存占用 | 50-150 MB | 系统进程 | 30-50 MB | 80-120 MB | 50-100 MB（原生 Swift） | 80-120 MB | 50-100 MB | **< 200 MB** |

### 2.2 核心竞品教训

**Everything 教训**：
- 速度是唯一的第一印象 — 用户会原谅几乎所有 UI 缺陷，只要结果即时出现
- 空搜索展示全部文件是极其有效的 onboarding 手法
- 全局热键是效率工具的命脉

**Spotlight 教训**：
- 索引不可靠是信任杀手 — 搜不到存在的文件 = 产品失败
- 用户没有索引健康感知 — Spotlight 不告诉用户索引不完整
- 文件搜索专精比「什么都能搜」更有价值

**Alfred/Raycast 教训**：
- 100ms vs 200ms 的启动差异用户能感知
- 免费层的丰富度直接决定采用率（Raycast 免费版的功能甚至超过 Alfred 付费版）
- 单次买断（Alfred $34）vs 订阅（Raycast $96/yr）各有市场

**DocFetcher 教训**：
- 外部依赖（Java）是安装阶段的最大流失点 — 用户反馈中 Java 安装问题是 #1 流失原因，大量 macOS 用户在看到「需要安装 Java 运行时」提示后直接放弃安装
- DeepFinder 的「零外部依赖、纯 Swift + Apple 框架」是对此痛点的直接回应：无需安装任何运行时或第三方包管理器，下载即用，卸载即拖入废纸篓
- 实时搜索是用户期望的基线，不是加分项
- 开源社区的「零冗余」（crap-free）设计哲学有强大吸引力

**FileMinutes 教训**：
- Spotlight 依赖是致命弱点：FileMinutes 无独立索引，继承 Spotlight 的不可靠性 — DeepFinder 的最大竞争论点就是独立索引引擎，不遗漏任何文件
- 键盘优先设计是效率工具的基线：FileMinutes 的所有操作可键盘完成 — 这证明全键盘操作不是「加分项」而是「用户期望」
- 文件夹范围搜索是独立需求：FileMinutes 支持限定目录内搜索 — 这验证了全局搜索之外，用户有明确的目录限定需求

**ProFind 教训**：
- AI 图像搜索验证 Vision Tagging 投资方向：ProFind 是 7 款竞品中唯一支持 AI 图像内容搜索的产品，证明市场存在需求 — DeepFinder 的 v3.0 Vision Tagging 直接对标此能力
- 定时索引满足高级用户控制欲：ProFind 的调度索引功能表明部分用户希望精确控制索引时机（磁盘 I/O 顾虑、笔记本电池）
- 手动建索引是摩擦点：ProFind 需用户手动触发索引构建 — DeepFinder 的自动增量索引是关键的 UX 优势，应在宣传中强调

**FileMinutes/Fenn/ProFind 教训**：Mac 文件搜索市场正在细分——FileMinutes 证明键盘优先是桌面应用的基线期望，Fenn 证明 AI 语义搜索可以支撑 $9-29/月的定价，ProFind 证明"Windows Everything 迁移用户"是一个高意图、服务不足的细分市场。DeepFinder 是唯一同时覆盖这三个方向的工具。

## 3. 阶段 1：发现（Discovery）

### 3.1 用户目标

快速理解 DeepFinder 是什么、为什么比 Spotlight 好。

### 3.2 关键洞察

- Everything 的口碑完全建立在「速度」上
- Spotlight 的信任已被「找不到文件」破坏 — 最容易攻破的痛点
- 开源项目的 GitHub Star、Hacker News、博客评测是主要传播渠道
- FileMinutes 依赖 Spotlight 索引 — 这是 DeepFinder 独立索引引擎的天然对比素材，在发现阶段突出「不依赖 Spotlight」
- ProFind 的 AI 图像搜索是 7 款竞品中的独家能力 — 验证了媒体搜索的市场需求，可在技术社区内容中作为差异化叙事

### 3.3 UX 需求

| ID | 需求 | 优先级 | 验证标准 |
|----|------|--------|---------|
| UX-D01 | GitHub README 首屏 3 秒内传达「比 Spotlight 快且准」 | P0 | 5 名非用户 3 秒内能描述产品价值 |
| UX-D02 | 展示 benchmark 数据：查询延迟 vs Spotlight | P1 | 包含可复现的测试方法和数据 |
| UX-D03 | 提供截图/GIF 展示搜索交互 | P1 | 捕获从热键唤起到结果操作的完整流程 |
| UX-D04 | Homebrew 安装命令一目了然 | P0 | 单行命令，零解释即可执行 |
| UX-D05 | README 明确标注「零外部依赖、纯 Swift」 | P1 | 与 DocFetcher 的 Java 依赖形成对比 |
| UX-D06 | 免费/开源声明须出现在所有用户文档前三句 | P0 | 新用户在前 3 秒阅读中看到 'free and open source' |
| UX-D07 | 提供竞品对比页面（含可复现 benchmark） | P1 | 至少覆盖 8 款竞品，包含速度 benchmark 数据和测试方法 |
| UX-D08 | README/网站明确传达「独立索引，不依赖 Spotlight」 | P1 | 潜在用户 5 秒内理解 DeepFinder 拥有自己的索引引擎，不继承 Spotlight 的遗漏问题；引用 FileMinutes 的 Spotlight 依赖作为反面案例 |

## 4. 阶段 2：安装（Installation）

### 4.1 用户目标

30 秒内完成安装并启动。

### 4.2 关键洞察

- Alfred/Raycast 都是「拖到 Applications → 完成」的两步安装
- Homebrew cask 是开发者的首选安装方式
- 安装包体积传递「轻量」信号（Everything 仅 1.6 MB）

### 4.3 UX 需求

| ID | 需求 | 优先级 | 验证标准 |
|----|------|--------|---------|
| UX-I01 | DMG 拖拽安装 ≤ 2 步 | P0 | 拖到 Applications → 双击启动 |
| UX-I02 | Homebrew tap 一键安装 | P0 | `brew install nadav/deepfinder/deepfinder` 完成 |
| UX-I03 | 首次启动无 Gatekeeper 拦截提示 | P1 | 正确 code sign + notarize |
| UX-I04 | 安装包体积 ≤ 10 MB | P2 | 压缩后 DMG 体积 |
| UX-I05 | CLI 可独立安装（不含 GUI） | P1 | `brew install nadav/deepfinder/deepfinder` 默认仅 CLI |

### 4.4 流失风险

- Gatekeeper 拦截（未 code sign / notarize）
- 安装包过大传递「重」的信号
- 安装过程需要终端命令（非开发者用户）

## 5. 阶段 3：首次运行与引导（First Run & Onboarding）

### 5.1 用户目标

60 秒内完成首次搜索，体验「Aha Moment」。

### 5.2 关键洞察

- Everything 的「空搜索展示全部文件」是极其有效的 onboarding — 用户自然开始打字
- 权限请求是 macOS 应用最大的流失点（Full Disk Access + Accessibility）
- 超过 2 步权限引导，流失率陡增

### 5.2.1 权限引导设计参考

Alfred 2026.4 引入统一权限面板，是当前 macOS 非沙盒应用的权限引导 UX 基准：

- **单一面板集中展示**：所有需要的权限（FDA、Accessibility、通知等）展示在同一面板，避免多次弹窗引导导致的流失
- **实时状态反馈**：每个权限旁有勾选图标（✅ 已授权 / ⚠️ 待授权），用户一眼可知当前状态
- **一键跳转系统设置**：点击未授权权限自动打开对应系统设置页面，降低用户在 macOS 设置迷宫中导航的认知负担
- **内嵌截图指引**：权限面板中嵌入逐步骤截图 +「已打开系统设置」确认按钮，减少用户在应用和系统设置间反复切换

### 5.3 UX 需求

| ID | 需求 | 优先级 | 验证标准 |
|----|------|--------|---------|
| UX-O01 | 首次启动自动构建索引，显示实时进度 | P0 | 显示「正在索引... N 个文件」 |
| UX-O02 | 索引完成前允许搜索已有结果 | P0 | 增量索引可用，不需等完成 |
| UX-O03 | 权限引导分步进行 | P0 | 先 FDA（必需），后 Accessibility（可选）；采用统一权限面板设计（见 §5.2.1） |
| UX-O04 | FDA 权限缺失时显示具体操作指引 | P0 | 引导到系统设置 → 隐私 → 完全磁盘访问；面板内嵌截图指引（见 §5.2.1） |
| UX-O05 | 无 FDA 时仍可搜索用户可访问目录 | P1 | 优雅降级，不阻断核心功能 |
| UX-O06 | 首次搜索面板打开时显示欢迎提示 | P1 | 提示快捷键和基本操作 |
| UX-O07 | 「Aha Moment」在首次查询 < 100ms 内到达 | P0 | 第一次搜索结果即时出现 |
| UX-O08 | 引导流程可跳过 | P1 | 「跳过」按钮始终可见 |
| UX-O09 | 全局热键设置在 onboarding 中引导 | P1 | 展示默认热键，允许自定义 |
| UX-O10 | Daemon 自动启动，无需手动管理 | P0 | 首次运行自动拉起 daemon |
| UX-O11 | 提供统一权限状态面板，单一面板展示所有权限及实时授权状态 | P1 | 面板显示所有必需/可选权限（FDA、Accessibility、通知），每个权限旁有 ✅已授权 / ⚠️待授权 状态，点击一键跳转对应系统设置页（对标 Alfred 2026.4 设计） |

**2026.4 更新（参考 Alfred onboarding 改进）**：权限引导应采用统一的权限面板，显示每个权限的实时授权状态（勾选/未勾选），提供一键跳转到对应系统设置页面的按钮，并在 macOS 大版本升级后自动重新检查权限状态。

### 5.4 流失风险

- 权限步骤 > 2 步（用户放弃授权 → 功能受限 → 流失）
- 索引时间 > 30 秒且无进度反馈（用户以为卡死）
- 首次搜索延迟 > 200ms（未能超越 Spotlight 的即时感）

### 5.5 Onboarding 流程设计

```
启动 App
  ├─ Daemon 自动启动（后台，透明）
  ├─ 索引开始构建（显示进度条 + 文件计数）
  ├─ 权限检查
  │   ├─ FDA 已授权 → 继续
  │   ├─ FDA 未授权 → 弹窗引导（可跳过）
  │   │   └─ 点击「打开系统设置」→ 跳转 FDA 页面
  │   └─ Accessibility（仅热键需要，稍后引导）
  ├─ 索引首次可用（≥ 1000 文件时）
  │   └─ 搜索面板自动弹出 + 欢迎提示
  └─ 用户开始搜索 → Aha Moment
```

## 6. 阶段 4：核心使用循环（Core Loop）

### 6.1 用户目标

唤起 → 输入 → 扫描结果 → 操作，≤ 2 秒完成。

**这是产品的命脉。80% 的用户体验发生在这个循环里。**

### 6.2 核心循环流程

```
⌃⌘K 唤起 → 输入查询 → 实时结果更新 → ↑↓ 选择 → Enter 操作
                                              ↓
                                    Space 预览 / ⌘C 复制路径 / 拖拽
                                              ↓
                                    Esc 关闭 → 焦点回到原应用
```

### 6.3 关键洞察

- Everything 的核心循环：Enter 开最常用文件、Ctrl+Enter 打开文件夹
- Spotlight 的实时结果在 50-100ms 内更新
- Alfred 比 Raycast 快约 100ms（100ms vs 200ms），用户能感知差异
- Quick Look（Space 预览）是 macOS 用户期望的基准功能
- 键盘导航是效率工具的核心能力，不是可选增强

### 6.4 UX 需求

| ID | 需求 | 优先级 | 验证标准 |
|----|------|--------|---------|
| UX-C01 | 全局热键唤起搜索面板 ≤ 100ms | P0 | 从按键到面板可见 < 100ms |
| UX-C02 | 输入每个字符后结果更新 ≤ 50ms | P0 | 从击键到结果渲染 < 50ms |
| UX-C03 | 结果默认按相关度排序 | P0 | 文件名匹配 > 路径匹配 > 内容匹配 |
| UX-C04 | ↑↓ 箭头键导航结果列表 | P0 | 无需鼠标即可浏览所有结果 |
| UX-C05 | Enter 打开文件，⌘+Enter 在 Finder 中显示 | P0 | Everything 的 Ctrl+Enter 模式 |
| UX-C06 | Space 键触发 Quick Look 预览 | P0 | macOS 标准预览交互 |
| UX-C07 | Esc 关闭面板，空搜索时 Esc 清空 | P0 | Spotlight 标准行为 |
| UX-C08 | Tab 键切换搜索框/结果列表焦点 | P1 | Everything 交互模式 |
| UX-C09 | 面板关闭后焦点回到原应用 | P0 | 不打断用户工作流 |
| UX-C10 | ↑ 键在空搜索框时调出搜索历史 | P1 | 复用之前的查询 |
| UX-C11 | 结果行显示：文件名 + 路径 + 大小 + 修改时间 | P0 | 一眼判断是否为目标文件 |
| UX-C12 | 右键上下文菜单（打开/显示/复制路径/Quick Look） | P0 | macOS 标准文件操作集 |
| UX-C13 | 支持拖拽文件到其他应用 | P1 | Finder 标准交互 |
| UX-C14 | ⌘+C 复制选中文件的完整路径 | P1 | 开发者高频操作 |
| UX-C15 | 无结果时显示友好提示 + 搜索建议 | P1 | 避免空白页面体验 |
| UX-C16 | 中文拼音搜索在 GUI 中畅通无阻 | P1 | PinyinIndex 在 GUI 搜索路径中工作 |
| UX-C17 | 所有搜索操作可通过键盘完成，无鼠标可达 | P0 | 搜索、导航、筛选、文件操作（打开/预览/复制路径/在 Finder 中显示）全部支持键盘快捷键；覆盖 FileMinutes 的键盘优先标准 |

### 6.5 流失风险

- 响应慢于 Spotlight（用户会直接切回去）
- 键盘操作不流畅（效率工具的核心承诺被打破）
- 搜索结果不准确（漏掉文件 = 信任崩塌）

## 7. 阶段 5：配置与个性化（Configuration）

### 7.1 用户目标

按自己的习惯定制，但不被设置项淹没。

### 7.2 关键洞察

- Everything 设置项多但组织清晰（左侧树形菜单）
- Alfred 的 Preferences 是独立窗口，分板块展示
- Spotlight 几乎不可配置（仅排除列表和快捷键）
- Raycast 的设置是命令面板式，搜索即达

### 7.3 UX 需求

| ID | 需求 | 优先级 | 验证标准 |
|----|------|--------|---------|
| UX-S01 | 设置面板通过菜单栏图标访问 | P0 | 点击状态栏图标 → 设置入口 |
| UX-S02 | 全局热键可自定义 | P0 | 避免与其他工具冲突 |
| UX-S03 | 索引排除目录可配置 | P0 | 如 node_modules、.git |
| UX-S04 | AI 搜索开关 + API Key 配置 | P1 | v3.0 AI 功能的配置入口 |
| UX-S05 | 搜索结果默认排序方式可选 | P2 | 名称/日期/大小/类型 |
| UX-S06 | 面板外观（主题/透明度）可调 | P2 | Alfred 有 100+ 主题 |
| UX-S07 | 设置变更即时生效，无需重启 | P1 | 不存在需要重启的设置项 |
| UX-S08 | 「恢复默认设置」一键重置 | P2 | 安全网机制 |
| UX-S09 | 用户可限制搜索范围为特定文件夹，支持添加/移除/切换搜索范围 | P1 | 设置面板提供搜索范围配置界面；用户在搜索框中可通过前缀语法（如 `in:~/Documents`）临时限定范围；范围变更即时生效 |

### 7.4 流失风险

- 设置项过多导致决策疲劳（Everything 的教训：组织清晰比数量更重要）
- 热键冲突未解决（用户无法设置自己习惯的热键 → 切回其他工具）
- 排除目录配置不生效（索引包含不该索引的目录 → 性能/隐私顾虑）

## 8. 阶段 6：高级功能（Advanced Features）

### 8.1 用户目标

发现并使用 AI 语义搜索、内容搜索、重复查找等进阶能力。

### 8.2 关键洞察

- Everything 没有内容索引 → DeepFinder 有内容搜索是差异化
- Raycast Pro 的 AI 是最大卖点 → DeepFinder 的 AI 搜索有同样潜力
- ProFind 的 AI 图像搜索（自然语言描述图片内容，如「海边的日落」）验证了 Vision Tagging 是有价值的产品差异化方向 — 用户愿意为 AI 驱动的媒体搜索付费，且该功能是 ProFind 在 6 款竞品中的独家卖点
- 高级功能的 discoverability 是关键 — 用户不会主动找隐藏功能

### 8.3 UX 需求

| ID | 需求 | 优先级 | 验证标准 |
|----|------|--------|---------|
| UX-A01 | 语义搜索用自然语言触发 | P1 | 如「上周的 PDF」无需特殊语法 |
| UX-A02 | 内容搜索结果高亮匹配片段 | P1 | 类似 DocFetcher 的黄色高亮 |
| UX-A03 | 重复文件查找有独立入口和结果视图 | P2 | 非核心但差异化 |
| UX-A04 | AI 功能首次使用时引导配置 | P1 | API Key 输入 + 连接测试 |
| UX-A05 | 语音搜索有明确的状态指示 | P2 | SpeechOverlayView 的视觉反馈 |
| UX-A06 | 搜索建议/自动补全在输入时显示 | P2 | 降低高级语法的认知负担 |
| UX-A07 | 分类筛选器（文件类型/日期/大小）在结果上方显示 | P1 | Spotlight category chips 模式 |
| UX-A08 | 支持自然语言描述搜索图像内容（如「海边的日落」「包含代码的截图」） | P2 | 用户输入自然语言描述后返回匹配的图片结果；Vision Tagging 标签作为匹配依据；搜索结果标注匹配理由 |
| UX-A09 | 用户可配置索引的定时全量重建计划（如每日凌晨 2 点、手动、或仅在空闲时） | P2 | 设置面板提供调度选项；索引按计划自动触发；笔记本电脑接通电源时才执行（避免耗电）；手动触发始终可用 |

### 8.4 流失风险

- 高级功能不可发现（用户不知道 AI 搜索/内容搜索存在 → 按普通文件搜索工具评价产品）
- AI 功能配置门槛过高（API Key 输入 + 模型选择 + 测试连接 → 放弃尝试）
- 内容搜索结果不准确（误报率高 → 用户认为功能不可靠，退回 Spotlight）

## 9. 阶段 7：更新与维护（Update & Maintenance）

### 9.1 用户目标

保持最新版本，无需手动干预。

### 9.2 关键洞察

- macOS 非 App Store 应用的标准方案是 Sparkle 框架
- Homebrew cask 用户通过 `brew upgrade` 更新
- Alfred 15 年无重大更新事故 — 静默更新是最佳实践

### 9.3 UX 需求

| ID | 需求 | 优先级 | 验证标准 |
|----|------|--------|---------|
| UX-U01 | 支持 Sparkle 自动更新检查 | P1 | 非 App Store macOS 应用的事实标准 |
| UX-U02 | 更新可后台下载，下次启动自动安装 | P1 | 减少用户打断 |
| UX-U03 | Homebrew 用户通过 `brew upgrade` 更新 | P0 | 与安装方式保持一致 |
| UX-U04 | Daemon 更新后自动重启 | P1 | 无需用户手动管理 daemon |
| UX-U05 | 版本号在关于/设置中可见 | P2 | 调试和用户反馈时需要 |
| UX-U06 | 索引损坏时自动检测并触发重建 | P1 | Spotlight 核心弱点的对标优势 |

### 9.4 流失风险

- 更新打断工作流（弹出更新提示 → 用户推迟更新 → 安全风险 + 功能落后）
- 更新后索引重建时间过长（用户感知为「更新后变慢」）
- Daemon 更新失败静默（daemon 崩溃但用户不知 → 搜索失效 → 流失）

## 附录 A：竞品调研详情

### Everything (voidtools) — Windows

**安装体验**：1.6 MB 便携 EXE 或安装版。无需安装依赖。首次启动即时可用（读 NTFS MFT）。仅 Windows/NTFS。

**核心循环**：全局热键 → 输入即时出结果（sub-ms）→ Enter 打开最常用 → Ctrl+Enter 打开文件夹 → Alt+Enter 属性。

**强项**：速度就是品牌。实时 USN Change Journal 追踪。极小内存占用（50-150 MB for 百万文件）。

**弱项**：仅 Windows/NTFS。无内容索引。UI 过时（Win 7 风格）。无 AI。

**DeepFinder 教训**：速度是第一印象的全部。空搜索展示所有文件是极好的 onboarding。全局热键是命脉。

### Spotlight — macOS

**安装体验**：系统内置，零安装。Cmd+Space 即用。

**核心循环**：Cmd+Space → 输入 → 实时结果（50-100ms）→ ↑↓ 选择 → Enter → Space 预览。

**强项**：零摩擦唤起。统一搜索入口。Quick Look 集成。Browse Mode（macOS Tahoe）。

**弱项**：索引不可靠（最大弱点）。无索引健康指示。结果排序不可预测。文件搜索不是专精。

**DeepFinder 教训**：可靠性是头号差异化。索引完整性应是可验证的。文件专精比什么都搜更有价值。

### Alfred — macOS

**安装体验**：DMG 拖拽安装。首次启动引导热键设置（⌥Space 或自定义）。2026.4 引入统一权限面板：单一面板展示所有权限需求，每个权限旁勾选反馈（✅已授权 / ⚠️待授权），一键跳转对应系统设置页，面板内嵌截图操作指引。

**核心循环**：热键唤起（~100ms）→ 输入 → 实时结果 → 操作。$34 终身买断（Powerpack 解锁高级功能）。

**强项**：速度（100ms 启动）。轻量（30-50 MB）。单次买断定价模型。2026.4 统一权限面板是 macOS 非沙盒应用的 UX 基准。15 年稳定更新记录。

**弱项**：无 AI 功能。无窗口管理。需 Powerpack 才有剪贴板/工作流。功能集受限于「启动器」定位。

**DeepFinder 教训**：100ms 启动是用户可感知的门槛。统一权限面板是最佳实践参考。单次买断有市场。

### Raycast — macOS

**安装体验**：DMG 拖拽安装。首次启动引导设置。免费层功能丰富。

**核心循环**：热键唤起（~200ms）→ 命令面板式交互 → 结果/扩展 → 操作。$96/yr Pro 订阅（含 AI）。

**强项**：2000+ 免费扩展。内置 AI（Pro 订阅）。命令面板式交互降低认知负担。活跃的社区生态。免费层功能超过 Alfred 付费版。

**弱项**：启动慢（~200ms，用户可感知 vs Alfred 100ms）。重（80-120 MB 内存）。订阅定价（$96/yr）。非开源。

**DeepFinder 教训**：免费层丰富度直接影响采用率。100ms vs 200ms 差异用户能感知。扩展生态是网络效应的核心。

### FileMinutes — macOS

**安装体验**：DMG 安装，50-100 MB 原生 Swift 应用。无外部依赖。需 FDA 才能内容搜索。

**核心循环**：键盘优先设计 — 全局热键唤起 → 输入查询 → 结果列表 → Enter 打开 / Space 预览 / 其他操作均可键盘完成。

**强项**：12+ 文件类型内容搜索（PDF/DOCX/XLSX 等）。键盘优先操作。文件夹范围搜索。原生 Swift，轻量（50-100 MB）。

**弱项**：依赖 Spotlight 索引（最大弱点）— 继承 Spotlight 的遗漏和不一致。无独立索引引擎。~100ms 延迟受限于 Spotlight 速度。无 AI 功能。

**DeepFinder 教训**：独立索引起点是核心差异化。键盘优先是基线期望而非高级功能。内容搜索的广度是用户可见的价值。

### ProFind — macOS

**安装体验**：DMG 安装，50-100 MB。首次使用需手动触发索引构建。支持定时调度索引重建。需 FDA 用于全盘搜索。

**核心循环**：全局热键唤起 → 输入查询 → 实时结果（~50ms，自定义原生索引）→ 文本+元数据双模式搜索。结果可按名称/日期/大小/类型排序。

**强项**：自定义原生索引（~50ms，不依赖 Spotlight）。AI 图像搜索（macOS 12+，自然语言描述图片内容 — 7 款竞品中的独家能力）。支持文本+元数据双模式搜索。

**弱项**：需手动建索引（首次使用摩擦大）。功能集窄（专注于文件搜索，无工作流/扩展生态）。收费软件。无内容文本高亮。社区小。

**DeepFinder 教训**：AI 图像搜索是有市场验证的差异化方向。自动索引 > 手动索引是核心 UX 优势。定时调度是值得考虑的高级功能。

### Recoll & DocFetcher — 跨平台

**安装体验**：DocFetcher 便携 ZIP 但需 Java（macOS 上是巨大摩擦）。Recoll 需外部 helper 程序。

**核心循环**：手动建索引 → 输入查询 → 结果列表 → 文本预览（高亮匹配）。

**强项**：「crap-free」设计哲学。便携模式（USB 随身带）。实时输入搜索。

**弱项**：Java 依赖是 dealbreaker。UI 过时。大文件集内存问题。项目几乎停止维护。

**DeepFinder 教训**：零依赖安装是巨大优势。实时搜索是基线期望，不是加分项。OSS 社区重视最小侵入。

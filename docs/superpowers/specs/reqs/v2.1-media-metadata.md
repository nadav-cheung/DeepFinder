### REQ-2.1-01 MetadataExtractor 协议 ✅ 🖥️ P0

**用户故事**

作为「系统维护者」，我希望媒体元数据提取由统一的协议与注册表调度，以便新增媒体类型（如未来 RAW 图片）时只需新增一个 Extractor 文件，无需改动索引或搜索层。

**验收标准（Given / When / Then）**

- **AC1 — 协议定义**
  - **Given** `MetadataExtractor` 协议
  - **When** 审阅其接口
  - **Then** 包含 `supportedExtensions: Set<String>` 与 `extract(url: URL) async -> ExtractedMetadata?`

- **AC2 — 元数据结构**
  - **Given** `ExtractedMetadata` 结构体
  - **When** 持有任意字段
  - **Then** 以 `[String: MetadataValue]` 字典存储，`MetadataValue` 为字符串/整数/浮点数/日期的枚举

- **AC3 — 注册表分发**
  - **Given** `MetadataExtractorRegistry` 管理所有 extractor 实例
  - **When** 收到一个待提取文件
  - **Then** 按扩展名分发到对应 extractor

- **AC4 — 并发安全**
  - **Given** Swift 6 严格并发模式
  - **When** 多任务并发调用 extractor
  - **Then** 类型声明为 `Sendable`，编译通过且无并发告警

**实现**：`Sources/Media/MetadataExtractor.swift`、`Sources/Media/ExtractedMetadata.swift` · **测试**：`Tests/MediaTests/MetadataExtractorRegistryTests.swift`、`Tests/MediaTests/ExtractedMetadataTests.swift`

### REQ-2.1-02 ImageMetadataExtractor（图片元数据）✅ 🖥️ P0

**用户故事**

作为「用户」，我希望通过 `width:>2560` 或 `height:>1440` 搜索到所有高分辨率图片，以便无需记住文件名即可按尺寸定位图片。

**验收标准（Given / When / Then）**

- **AC1 — 提取来源**
  - **Given** 一张图片文件
  - **When** 调用 extractor
  - **Then** 通过 ImageIO/CGImageSource 提取 EXIF 元数据

- **AC2 — 字段完整**
  - **Given** 含完整 EXIF 的图片
  - **When** 提取
  - **Then** 返回 width（像素宽）、height（像素高）、dpi、colorSpace、orientation、dateTaken

- **AC3 — 格式覆盖**
  - **Given** JPG/PNG/HEIC/GIF/TIFF/BMP/WebP 图片
  - **When** 提取
  - **Then** 均被支持并返回元数据

- **AC4 — 损坏/无 EXIF 容错**
  - **Given** 文件损坏或无 EXIF 的图片
  - **When** 提取
  - **Then** 返回 nil（不崩溃）

- **AC5 — 大文件异步**
  - **Given** 大文件（> 50 MB）
  - **When** 提取
  - **Then** 异步执行，不阻塞索引主流程

**实现**：`Sources/Media/ImageMetadataExtractor.swift` · **测试**：`Tests/MediaTests/ImageMetadataExtractorTests.swift`

### REQ-2.1-06 MediaMetadataIndex（元数据索引）✅ 🖥️ P1

**用户故事**

作为「用户」，我希望索引构建时后台提取的媒体元数据被存入内存索引与 SQLite 并支持毫秒级过滤，以便按元数据字段搜索时不感知延迟。

**验收标准（Given / When / Then）**

- **AC1 — 提取路由**
  - **Given** 待提取的文件
  - **When** 触发提取
  - **Then** 元数据通过 `MetadataExtractorRegistry` 路由到对应 extractor（替代单一 actor）

- **AC2 — 数据载体**
  - **Given** 已提取的元数据
  - **When** 存入索引
  - **Then** 存入 `FileRecord.metadata: ExtractedMetadata?`（结构化字段）

- **AC3 — 持久化**
  - **Given** 元数据需要跨进程保留
  - **When** 写入 SQLite
  - **Then** 经 `metadata_json` 列以 JSON 编码存储，复用 IndexPersistence 批量逻辑

- **AC4 — 重复提取避免**
  - **Given** 已提取过元数据的文件
  - **When** 文件未改动
  - **Then** 基于 FileRecord 级别 mtime 检查，不重复提取

- **AC5 — 后台异步提取（延期）**
  - **Given** 索引构建阶段
  - **When** 后台低优先级 TaskGroup 异步提取
  - **Then** 延期至 v3.1 实现

- **AC6 — 进度上报（延期）**
  - **Given** 提取进行中
  - **When** 用户通过 `:stats` 查询进度
  - **Then** 进度经 IPC 上报，延期至 v3.1 实现

**实现**：`Sources/Media/MetadataExtractorRegistry.swift`、`Sources/Index/FileRecord.swift` · **测试**：—

> 备注：原 `MediaMetadataIndex` actor 在 commit 93d3e08 中因零调用者被移除。其功能由 `MetadataExtractorRegistry`（提取路由）、`FileRecord.metadata`（数据载体）、`IndexPersistence.metadata_json`（持久化）和 `SearchFilter.metadataMin/Max/Range/Match`（搜索过滤）共同承担。异步批量提取和进度上报延期至 v3.1。

### REQ-2.1-07 元数据搜索过滤器 ✅ 🖥️ P1

**用户故事**

作为「用户」，我希望使用 `width:>2560`、`artist:周杰伦`、`duration:>300`、`pdf-author:xxx` 等过滤语法搜索媒体文件，以便用结构化字段而非文件名定位媒体内容。

**验收标准（Given / When / Then）**

- **AC1 — 过滤器扩展**
  - **Given** SearchFilter
  - **When** 扩展元数据字段
  - **Then** 支持元数据字段过滤器

- **AC2 — 字段覆盖**
  - **Given** 元数据字段集合
  - **When** 列出可过滤字段
  - **Then** 包含 width、height、duration、pages、artist、album、title、genre、pdf-author、pdf-title、fps、bitRate、codec

- **AC3 — 比较运算符**
  - **Given** 数值字段
  - **When** 使用比较语法
  - **Then** 支持 `:`（等于）、`:>`（大于）、`:<`（小于）、`:>=`、`:<=`

- **AC4 — 字符串子串匹配**
  - **Given** 字符串字段（如 artist）
  - **When** 输入 `artist:周`
  - **Then** 匹配包含「周」的值（如「周杰伦」）

- **AC5 — 无元数据排除**
  - **Given** 无元数据的文件
  - **When** 应用元数据过滤器
  - **Then** 自动排除该文件

- **AC6 — CLI 帮助文档**
  - **Given** 用户输入 `:help filters`
  - **When** 查询帮助
  - **Then** 列出所有可用的元数据过滤器

**实现**：`Sources/Search/SearchFilter.swift` · **测试**：`Tests/MediaTests/MetadataFilterTests.swift`

# v1.2 — 元数据过滤

8 项 REQ (P0: 7, P1: 1)

对标 Everything 的 `size:`, `dm:`, `dc:`, `ext:`, `file:`, `folder:`, `depth:` 等过滤函数。用户可以在搜索查询中混入元数据过滤器，缩小结果范围。

**设计原则**：过滤器作为搜索查询的修饰词，与 v1.1 高级搜索语法（布尔、通配符、正则）正交组合。过滤在 SearchCoordinator 层执行，先从索引获取文本匹配结果，再用 FilterExpression 管线过滤。

**前置依赖**：v1.1 高级搜索语法（SearchQuery 解析管线）。v1.2 的过滤器解析集成到同一管线中。

### REQ-1.2-01 Size filter (size:>1mb, size:100kb..10mb) ✅ 🖥️ P0

**用户场景**：用户在清理磁盘时想找出大文件——输入 `size:>1gb` 立即看到所有超过 1GB 的文件。或者缩小范围 `size:100mb..500mb` 找特定大小的文件。在搜索结果中混入文件名关键词：`video size:>500mb` 找到名称含 "video" 且超过 500MB 的文件。

**操作流程**：
1. 用户输入 `size:>1mb` → 返回所有大于 1MB 的文件
2. 用户输入 `size:100kb..10mb` → 返回大小在 100KB 到 10MB 之间的文件
3. 用户输入 `report size:<1mb` → 名称含 "report" 且小于 1MB 的文件
4. 用户输入 `size:empty` → 返回大小为 0 的文件

**语法**（对标 Everything `size:` 函数）：

```
size:<value>          精确匹配（极少使用）
size:<=<value>        小于等于
size:<<value>         小于
size:>=<value>        大于等于
size:><value>         大于
size:<start>..<end>   范围（包含两端）
```

**单位**：

| 单位 | 后缀 | 换算 |
|------|------|------|
| 字节 | (无后缀) | 1 |
| KB | `kb` | 1024 |
| MB | `mb` | 1024 * 1024 |
| GB | `gb` | 1024 * 1024 * 1024 |

单位不区分大小写：`mb`、`MB`、`Mb` 等效。

**大小常量**（对标 Everything）：

| 常量 | 含义 |
|------|------|
| `empty` | size == 0 |
| `tiny` | 0 < size <= 10 KB |
| `small` | 10 KB < size <= 100 KB |
| `medium` | 100 KB < size <= 1 MB |
| `large` | 1 MB < size <= 16 MB |
| `huge` | 16 MB < size <= 128 MB |
| `gigantic` | size > 128 MB |

**验收标准**：
- [ ] 支持 `size:><value>`, `size:<<value>`, `size:>=<value>`, `size:<=<value>`, `size:<value>`, `size:<start>..<end>` 所有比较形式
- [ ] 支持 kb/mb/gb 单位（不区分大小写），无后缀 = 字节
- [ ] 支持 7 个大小常量：empty, tiny, small, medium, large, huge, gigantic
- [ ] 数值解析为 Int64 字节值，与 FileRecord.size (Int64) 直接比较
- [ ] 语法错误时返回明确错误信息：`"无效的 size 语法: '<input>'。用法: size:>10mb 或 size:100kb..1gb"`
- [ ] 过滤基于 FileRecord.size 字段，零 I/O（内存中直接比较）
- [ ] 文件夹的 size 值取决于 FileRecord.isDirectory：目录的 size 为 0（不参与大小过滤，除非用户显式搜索 `folder: size:empty`）
- [ ] 单元测试覆盖：所有比较运算符、所有单位、每个常量、范围语法、边界值（0、1 字节、Int64.max 附近）、无效语法错误

### REQ-1.2-02 Date filter (dm:today, dc:thisweek, dm:2026-01-01..2026-03-31) ✅ 🖥️ P0

**用户场景**：用户想找今天修改过的文件——输入 `dm:today`。找上周创建的文档——`dc:lastweek`。找特定日期范围内修改的报告——`report dm:2026-01-01..2026-03-31`。

**操作流程**：
1. 用户输入 `dm:today` → 返回今天修改过的所有文件
2. 用户输入 `dc:thisweek` → 返回本周创建的文件
3. 用户输入 `dm:2026-01-01..2026-03-31` → 返回修改日期在指定范围内的文件
4. 用户输入 `report dm:>2026-05-01` → 名称含 "report" 且 5 月 1 日后修改的文件

**语法**（对标 Everything `dm:` / `dc:` 函数）：

```
dm:<date>         修改日期匹配
dc:<date>         创建日期匹配
```

两个前缀支持完全相同的日期语法。

**比较运算符**：

```
dm:<date>           精确匹配（同一天）
dm:<=<date>         早于等于
dm:<<date>          早于
dm:>=<date>          晚于等于
dm:><date>          晚于
dm:<start>..<end>   范围（包含两端）
```

**日期格式**：

| 格式 | 示例 | 说明 |
|------|------|------|
| `YYYY-MM-DD` | `2026-05-29` | 完整日期 |
| `YYYYMMDD` | `20260529` | 紧凑日期 |
| `YYYY-MM-DDThh:mm` | `2026-05-29T14:30` | 日期+时间（分钟精度） |
| `YYYY-MM-DDThh:mm:ss` | `2026-05-29T14:30:00` | 日期+时间（秒精度） |

**日期常量**（对标 Everything）：

| 常量 | 含义 |
|------|------|
| `today` | 当天（00:00:00 - 23:59:59） |
| `yesterday` | 昨天 |
| `thisweek` | 本周（周一开始） |
| `lastweek` | 上周 |
| `thismonth` | 本月 |
| `lastmonth` | 上月 |
| `thisyear` | 今年 |
| `lastyear` | 去年 |

所有日期常量使用系统本地时区（`TimeZone.current`），不涉及 UTC 转换。

**日期范围常量的边界语义**：
- `today`：`Date().startOfDay ... Date().endOfDay`
- `thisweek`：本周一 `startOfDay` ... 本周日 `endOfDay`
- `thismonth`：本月 1 日 `startOfDay` ... 本月最后一天 `endOfDay`
- 范围语法 `dm:2026-01-01..2026-03-31`：start `startOfDay` ... end `endOfDay`（包含 3 月 31 日全天）

**验收标准**：
- [ ] 支持 `dm:` 和 `dc:` 两个前缀（修改日期 / 创建日期）
- [ ] 支持 4 种日期格式：YYYY-MM-DD, YYYYMMDD, YYYY-MM-DDThh:mm, YYYY-MM-DDThh:mm:ss
- [ ] 支持 8 个日期常量：today, yesterday, thisweek, lastweek, thismonth, lastmonth, thisyear, lastyear
- [ ] 支持 5 种比较运算符 + 范围语法
- [ ] 日期常量使用本地时区，使用 Foundation `Calendar` API 计算
- [ ] 过滤基于 FileRecord.modifiedAt / FileRecord.createdAt (Date)
- [ ] 精确匹配（无运算符）匹配整天（startOfDay ... endOfDay）
- [ ] 语法错误时返回明确错误信息：`"无效的日期语法: '<input>'。用法: dm:today 或 dm:2026-01-01..2026-03-31"`
- [ ] 单元测试覆盖：所有日期常量、所有比较运算符、范围语法、各日期格式、跨时区边界（mock 时区）、无效输入

### REQ-1.2-03 Extension filter (ext:pdf;doc;xlsx) ✅ 🖥️ P0

**用户场景**：用户只想在 PDF 和 Word 文档中搜索——`report ext:pdf;doc;xlsx`。或者只找图片格式——`screenshot ext:png;jpg;gif`。分号分隔多个扩展名，一次过滤多种文件类型。

**操作流程**：
1. 用户输入 `ext:pdf` → 只返回 PDF 文件
2. 用户输入 `ext:pdf;doc;xlsx` → 返回 PDF、Word、Excel 文件
3. 用户输入 `report ext:pdf` → 名称含 "report" 且为 PDF 的文件
4. 用户输入 `ext:pdf ext:doc` → 报错：重复过滤器不叠加，提示 `"ext: 过滤器只能出现一次，如需多扩展名请用分号: ext:pdf;doc"`

**语法**（对标 Everything `ext:` 函数）：

```
ext:<extension>              单个扩展名
ext:<ext1>;<ext2>;<ext3>     分号分隔多个扩展名（OR 语义）
```

**规则**：
- 扩展名不含前导点：`ext:pdf`（正确）而非 `ext:.pdf`
- 扩展名不区分大小写：`ext:PDF` 等同 `ext:pdf`
- 分号分隔的多个扩展名之间是 OR 关系
- 同一查询中 `ext:` 只能出现一次，重复出现报错
- 空扩展名用 `ext:` 表示（无扩展名的文件）

**匹配逻辑**：
- FileRecord.extension 为 `Optional<String>`，NFC 统一化 + lowercased
- `ext:pdf` 匹配 `FileRecord.extension == "pdf"`
- `ext:` (空值) 匹配 `FileRecord.extension == nil`（无扩展名文件）
- 文件夹：`FileRecord.isDirectory == true` 时 extension 可为 nil 或有值（".app" 等），过滤器按实际 extension 字段匹配

**验收标准**：
- [ ] 支持单个扩展名 `ext:pdf` 和分号分隔多扩展名 `ext:pdf;doc;xlsx`
- [ ] 扩展名不区分大小写
- [ ] 空扩展名 `ext:` 匹配无扩展名文件
- [ ] 同一查询中 `ext:` 重复出现时报错，不静默覆盖
- [ ] 匹配基于 FileRecord.extension 字段，不涉及文件系统 I/O
- [ ] 语法错误时返回明确错误信息
- [ ] 单元测试覆盖：单扩展名、多扩展名、空扩展名、大小写混合、重复过滤器错误、无效字符

### REQ-1.2-04 File type macros (audio:, video:, pic:, doc:) ✅ 🖥️ P0

**用户场景**：用户想找所有视频文件——输入 `video:` 即可，不需要记住 `ext:mp4;avi;mkv;mov;wmv;flv;webm...`。输入 `pic: sunset` 找到所有图片格式中名称含 "sunset" 的文件。

**操作流程**：
1. 用户输入 `video:` → 返回所有视频文件
2. 用户输入 `audio: podcast` → 名称含 "podcast" 的音频文件
3. 用户输入 `doc: report` → 名称含 "report" 的文档文件
4. 用户输入 `pic:` → 返回所有图片文件

**语法**（对标 Everything macros：`audio:`, `video:`, `pic:`, `doc:`）：

```
audio:    音频文件
video:    视频文件
pic:      图片文件
doc:      文档文件
```

每个宏展开为一组扩展名的 `ext:` 过滤器。宏可以与文本搜索组合：`video: movie` = 视频文件 AND 名称含 "movie"。

**预定义扩展名映射**：

| 宏 | 扩展名 |
|------|--------|
| `audio:` | mp3, wav, flac, aac, ogg, wma, m4a, aiff, alac, opus, mid, midi |
| `video:` | mp4, avi, mkv, mov, wmv, flv, webm, m4v, mpg, mpeg, m2ts, ts, vob, 3gp |
| `pic:` | jpg, jpeg, png, gif, bmp, tiff, tif, svg, webp, ico, heic, heif, raw, psd, eps |
| `doc:` | pdf, doc, docx, xls, xlsx, ppt, pptx, txt, md, rtf, pages, numbers, key, csv, odt, ods, odp |

**设计决策**：
- 扩展名列表硬编码在 `FileTypeMacros` 常量中，不依赖 Uniform Type Identifier (UTI) 查询（避免 I/O，确保确定性）
- 未来可通过 ConfigStore 扩展（v1.3 自定义过滤器），v1.2 只提供内置列表
- `exe:` 宏对标 Everything 但 macOS 不适用（无 .exe），不实现
- `zip:` 宏对标 Everything 的 "compressed"，延迟到 v1.3（与自定义过滤器一起设计）

**验收标准**：
- [ ] 支持 4 个内置宏：`audio:`, `video:`, `pic:`, `doc:`
- [ ] 每个宏展开为 `ext:<ext1>;<ext2>;...` 形式，等价于手动指定所有扩展名
- [ ] 扩展名不区分大小写（与 ext: 行为一致）
- [ ] 宏与 `ext:` 可同时使用（AND 语义：结果必须同时满足两个过滤器）
- [ ] 未知宏（如 `code:`）报错：`"未知的文件类型宏: 'code:'。可用: audio:, video:, pic:, doc:"`
- [ ] 扩展名列表可测试：通过 `FileTypeMacros.extensions(for:)` 公开访问
- [ ] 单元测试覆盖：每个宏的正确展开、与文本搜索组合、未知宏错误、与 ext: 同时使用

### REQ-1.2-05 File/folder filter (file:, folder:) ✅ 🖥️ P0

**用户场景**：用户只想看文件（不要文件夹）——`file: report`。或者只想在文件夹中搜索——`folder: project`。在 Everything 中这是最常用的修饰符之一。

**操作流程**：
1. 用户输入 `file:` → 只返回文件（排除所有文件夹）
2. 用户输入 `folder:` → 只返回文件夹（排除所有文件）
3. 用户输入 `file: report` → 名称含 "report" 的文件
4. 用户输入 `folder: project` → 名称含 "project" 的文件夹

**语法**（对标 Everything modifiers `file:` / `folder:`）：

```
file:      只匹配文件（isDirectory == false）
folder:    只匹配文件夹（isDirectory == true）
```

**规则**：
- `file:` 和 `folder:` 互斥：同一查询中同时出现报错 `"file: 和 folder: 不能同时使用"`
- 可以单独使用（无文本搜索）：`file:` 返回所有文件
- 与 IPC 协议中的 `IPCFileType` 对齐：CLI 的 `--type file` 和查询中的 `file:` 语义一致

**与现有 IPC 的关系**：
- v0.5 CLI 的 `--type file|folder` flag 通过 `IPCQueryRequest.fileType` 字段传递
- v1.2 的 `file:` / `folder:` 在查询文本中解析，效果相同但位置不同
- 两者同时使用时以查询文本中的 `file:`/`folder:` 为准（查询文本优先级高于 flag）
- IPC 协议的 `IPCQueryRequest.fileType` 保留用于 CLI flag 兼容

**验收标准**：
- [ ] `file:` 过滤：只返回 `FileRecord.isDirectory == false` 的记录
- [ ] `folder:` 过滤：只返回 `FileRecord.isDirectory == true` 的记录
- [ ] 同时使用 `file:` 和 `folder:` 报错
- [ ] 无文本搜索时 `file:` / `folder:` 可单独使用（返回所有文件或所有文件夹）
- [ ] 单元测试覆盖：file-only、folder-only、互斥错误、与文本搜索组合、单独使用

### REQ-1.2-06 Path depth filter (depth:3) ✅ 🖥️ P0

**用户场景**：用户想在 home 目录的前 3 层搜索——`depth:3 report`。避免搜索到深层嵌套的 `node_modules` 或 `.build` 目录中的文件。限定搜索深度让结果更精准。

**操作流程**：
1. 用户输入 `depth:3` → 只返回路径深度 <= 3 的文件和文件夹
2. 用户输入 `depth:1` → 只返回根目录下直接的文件和文件夹
3. 用户输入 `report depth:2` → 名称含 "report" 且路径深度 <= 2

**语法**（对标 Everything `depth:` / `parents:` 函数）：

```
depth:<N>        路径深度 <= N
depth:<=<N>      路径深度 <= N（显式）
depth:<<N>       路径深度 < N
depth:>=<N>      路径深度 >= N
depth:><N>       路径深度 > N
depth:<start>..<end>   路径深度范围
```

**路径深度计算**：
- 根路径 `/` 深度 = 0
- `/Users/` 深度 = 1
- `/Users/nadav/` 深度 = 2
- `/Users/nadav/Documents/` 深度 = 3
- `/Users/nadav/Documents/report.pdf` 深度 = 3
- 计算方式：`path.components(separatedBy: "/").filter { !$0.isEmpty }.count`
- 符号链接解析后的实际路径参与计算（FileRecord.path 存储解析后路径）

**规则**：
- N 必须是非负整数（0, 1, 2, ...）
- 无比较运算符时默认 `<=N`（对标 Everything `depth:` 语义）
- N 为 0 时只返回根路径项

**验收标准**：
- [ ] 支持所有比较运算符 + 范围语法
- [ ] 无运算符时默认 `<=`
- [ ] 深度基于 FileRecord.path 计算，不涉及文件系统 I/O
- [ ] 无效输入（非整数、负数）报错：`"无效的 depth 值: '<input>'。用法: depth:3"`
- [ ] 单元测试覆盖：各比较运算符、范围语法、边界值（0、1）、深层路径、符号链接路径

### REQ-1.2-07 FilterExpression model ✅ 🖥️ P1

**用户场景**：用户无直接感知。FilterExpression 是所有过滤器的统一模型，确保过滤器可组合、可序列化、可测试。是 v1.2 其他所有 REQ 的基础设施。

**模型设计**：

```swift
/// 单个过滤器条件
enum Filter: Sendable, Equatable {
    case size(SizeFilter)
    case dateModified(DateFilter)
    case dateCreated(DateFilter)
    case extension(ExtensionFilter)
    case fileType(FileTypeFilter)
    case depth(DepthFilter)
    case typeMacro(TypeMacroFilter)
}

/// 所有过滤器的集合（AND 语义）
struct FilterExpression: Sendable, Equatable {
    let filters: [Filter]

    /// 应用所有过滤器到 FileRecord，全部通过返回 true
    func matches(_ record: FileRecord) -> Bool
}
```

**各过滤器子模型**：

```swift
/// 比较运算符
enum ComparisonOperator: String, Sendable {
    case lessThan = "<"
    case lessThanOrEqual = "<="
    case greaterThan = ">"
    case greaterThanOrEqual = ">="
    case equal = "="
    case range = ".."
}

/// Size filter
struct SizeFilter: Sendable, Equatable {
    enum Unit: String, Sendable {
        case bytes, kb, mb, gb
    }
    enum Constant: String, Sendable, CaseIterable {
        case empty, tiny, small, medium, large, huge, gigantic
    }
    indirect enum Value: Sendable, Equatable {
        case literal(bytes: Int64)
        case constant(Constant)
    }
    let op: ComparisonOperator
    let value: Value           // single value
    let rangeEnd: Value?       // only for .range
}

/// Date filter
struct DateFilter: Sendable, Equatable {
    enum Constant: String, Sendable, CaseIterable {
        case today, yesterday, thisweek, lastweek
        case thismonth, lastmonth, thisyear, lastyear
    }
    indirect enum Value: Sendable, Equatable {
        case absolute(Date)
        case constant(Constant)
    }
    let op: ComparisonOperator
    let value: Value
    let rangeEnd: Value?       // only for .range
}

/// Extension filter
struct ExtensionFilter: Sendable, Equatable {
    let extensions: Set<String>  // lowercased, NFC
}

/// File type filter
enum FileTypeFilter: String, Sendable {
    case file
    case folder
}

/// Depth filter
struct DepthFilter: Sendable, Equatable {
    let op: ComparisonOperator
    let value: Int
    let rangeEnd: Int?          // only for .range
}

/// Type macro filter
struct TypeMacroFilter: Sendable, Equatable {
    let macro: String             // "audio", "video", "pic", "doc"
    let extensions: Set<String>   // resolved extensions
}
```

**组合语义**：
- FilterExpression.filters 中的所有过滤器之间是 AND 关系
- 同类过滤器冲突检测：`file:` + `folder:` 报错；`ext:` 重复出现报错
- 不同类过滤器自由组合：`size:>1mb ext:pdf dm:today` = 大于 1MB AND PDF AND 今天修改

**验收标准**：
- [ ] FilterExpression 模型：Filter enum + 各子模型，Codable + Sendable + Equatable
- [ ] `FilterExpression.matches(_:) -> Bool` 方法：对 FileRecord 应用所有过滤器
- [ ] 每个过滤器子模型有独立的 `matches(_:) -> Bool` 方法
- [ ] 冲突检测：`file:` + `folder:` 冲突；`ext:` 重复冲突
- [ ] 模型可序列化（Codable），支持 IPC 传输和持久化
- [ ] 空的 FilterExpression（无过滤器）匹配所有记录
- [ ] 单元测试覆盖：每个过滤器的 matches 方法、组合 AND 语义、冲突检测、空过滤器

### REQ-1.2-08 Filter integration with SearchCoordinator ✅ 🖥️ P0

**用户场景**：用户在 CLI 中输入 `report size:>1mb ext:pdf dm:today` → SearchCoordinator 先通过索引找到名称含 "report" 的文件，然后用 FilterExpression 过滤出大于 1MB、PDF 格式、今天修改的结果。整个过程无文件系统 I/O，过滤在内存中完成。

**操作流程**：
1. 用户输入查询文本 → 解析管线提取文本搜索词和过滤器
2. 文本搜索词 → SearchProvider → 索引匹配结果
3. FilterExpression → 对匹配结果逐条过滤
4. 排序 → 返回最终结果

**集成架构**：

```
查询文本 "report size:>1mb ext:pdf dm:today"
    │
    ▼
QueryParser (v1.1 解析管线扩展)
    │
    ├── SearchQuery.text = "report"
    └── FilterExpression.filters = [size(>1mb), ext(pdf), dm(today)]
    │
    ▼
SearchCoordinator.search(query:filter:)
    │
    ├── 1. SearchProvider.search(query:) → 候选结果（基于文本匹配）
    ├── 2. FilterExpression.filter(results) → 过滤后结果
    └── 3. 排序 → 返回
```

**SearchQuery 扩展**：

```swift
struct SearchQuery: Sendable {
    let id: String
    let text: String           // NFC + lowercased（纯文本搜索部分）
    let limit: Int             // 默认 100
    let options: SearchOptions
    let filter: FilterExpression  // v1.2 新增
}
```

**性能要求**：
- 过滤操作在内存中完成，不涉及文件系统 I/O
- 所有过滤器字段来自 FileRecord 已有字段（size, createdAt, modifiedAt, extension, isDirectory, path）
- 单条记录的过滤评估（所有过滤器）< 1μs
- 过滤不改变结果排序（排序逻辑不受过滤器影响）
- 过滤器在 SearchCoordinator 内执行，SearchProvider 层无感知

**错误处理**：
- 过滤器语法错误：解析阶段报错，返回错误信息给 CLI，不执行搜索
- 过滤器冲突（file: + folder:）：解析阶段报错
- 无效值（size:abc, depth:-1）：解析阶段报错
- 错误信息格式：`"过滤器错误: <具体原因>。用法: <正确语法示例>"`

**IPC 协议扩展**：

```swift
struct IPCQueryRequest: Codable {
    let text: String
    let limit: Int
    let offset: Int
    let sortBy: IPSearchSort?
    let fileType: IPCFileType?         // v0.5 保留，兼容 CLI flag

}
```

`filterExpression` 为 nil 时，过滤器从 `text` 中解析。CLI 同时使用 `--type file` flag 和查询中的 `file:` 时，以查询文本为准。

**CLI flag 与查询文本过滤器的关系**：

| 场景 | 行为 |
|------|------|
| `deepfinder "report size:>1mb"` | 从查询文本解析过滤器 |
| `deepfinder "report" --type file` | 从 CLI flag 构造 FileTypeFilter |
| `deepfinder "report size:>1mb" --type folder` | 查询文本中的过滤器优先（此处 `size:` 生效），`--type folder` 被查询文本过滤器覆盖（无 `file:`/`folder:` 时 flag 生效） |
| `deepfinder "file: report" --type folder` | 冲突：查询文本 `file:` 和 flag `--type folder`，报错 |

**REPL 集成**：
- REPL 搜索命令支持完整的过滤器语法：`> report size:>1mb ext:pdf`
- `:help` 命令列出所有可用过滤器及语法
- Tab 补全支持过滤器前缀：输入 `si` → 补全 `size:`，输入 `dm` → 补全 `dm:`

### REQ-1.1-01 QueryParser — 布尔表达式解析 ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望用空格（AND）、`|`（OR）、`!`（NOT）以及短语、分组组合查询词，以便像 Everything 一样灵活地表达组合搜索需求。

**验收标准（Given / When / Then）**

- **AC1 — AND 运算（空格）**
  - **Given** 用户输入 `a b`
  - **When** QueryParser 解析
  - **Then** 生成 `AND(term("a"), term("b"))`

- **AC2 — OR 运算（`|`）**
  - **Given** 用户输入 `a|b`
  - **When** QueryParser 解析
  - **Then** 生成 `OR(term("a"), term("b"))`

- **AC3 — NOT 运算（`!`）**
  - **Given** 用户输入 `!a`（`!` 紧跟词或带空格 `! draft` 均合法）
  - **When** QueryParser 解析
  - **Then** 生成 `NOT(term("a"))`

- **AC4 — 精确短语（`"..."`）**
  - **Given** 用户输入 `"annual report"`
  - **When** QueryParser 解析
  - **Then** 生成 `phrase("annual report")`，引号内空格/`|`/`!` 为字面量

- **AC5 — 分组（`<...>`）**
  - **Given** 用户输入 `<a|b> c`
  - **When** QueryParser 解析
  - **Then** 生成 `AND(OR(term("a"), term("b")), term("c"))`

- **AC6 — 运算符优先级**
  - **Given** 优先级从高到低为：短语 > 分组 > NOT > AND > OR；连续空格视为单个 AND；`|` 两侧无需空格
  - **When** 解析 `a|b c`
  - **Then** 生成 `OR(term("a"), AND(term("b"), term("c")))`

- **AC7 — 嵌套分组**
  - **Given** 用户输入 `<a <b|c>>`
  - **When** QueryParser 解析
  - **Then** 生成 `AND(term("a"), OR(term("b"), term("c")))`

- **AC8 — 空查询**
  - **Given** 空查询或仅含空格的查询（`` 或 `   `）
  - **When** QueryParser 解析
  - **Then** 返回空 AST，不报错

- **AC9 — 未闭合引号**
  - **Given** 用户输入 `"hello world`（未闭合）
  - **When** QueryParser 解析
  - **Then** 视为从该引号到行尾的短语 `phrase("hello world")`

- **AC10 — 空分组错误**
  - **Given** 用户输入 `<>`（空分组）
  - **When** QueryParser 解析
  - **Then** 返回解析错误，错误消息包含位置信息

- **AC11 — NFC + lowercased**
  - **Given** 已解析的 term
  - **When** QueryParser 完成解析
  - **Then** term 经过 NFC 统一化 + lowercased（除非被 `case:` 修饰符覆盖）；Unicode 词（不含空格、`|`、`<`、`>`、`"` 的连续字符序列）被正确切分

- **AC12 — 性能**
  - **Given** 1000 字符复杂查询
  - **When** 解析
  - **Then** 耗时 < 0.1ms

**实现**：`Sources/Search/QueryTerm.swift` · **测试**：`Tests/SearchTests/QueryParserTests.swift`

### REQ-1.1-02 通配符匹配（* 和 ?） ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望用 `*`（零个或多个任意字符）和 `?`（恰好一个任意字符）做模式匹配，以便用最直观的语法定位 PDF、扩展名填充位等文件。

**验收标准（Given / When / Then）**

- **AC1 — `*` 匹配**
  - **Given** 用户输入 `*.pdf`
  - **When** 通配符匹配
  - **Then** 匹配零个或多个任意字符（含 "report.pdf"、".pdf"）

- **AC2 — `?` 匹配**
  - **Given** 用户输入 `report_??.xlsx`
  - **When** 通配符匹配
  - **Then** 匹配恰好两个字符（"report_01.xlsx"、"report_Q2.xlsx"），不匹配 "report.xlsx"、"report_1.xlsx"

- **AC3 — 匹配范围**
  - **Given** 默认匹配文件名（隐式锚定整个文件名，`*.pdf` 与 `*pdf` 等效，与 Everything wfn/wholefilename 行为一致）
  - **When** 查询
  - **Then** 匹配文件名（不含路径）；`path:` 修饰符切换为匹配完整路径

- **AC4 — `*.ext` 优化**
  - **Given** 通配符 pattern 仅尾部 `*`（`*.ext` 或 `prefix*`）
  - **When** 查询
  - **Then** `*.ext` 退化为扩展名直接过滤；`prefix*` 退化为 Trie 前缀查询

- **AC5 — 路径分隔符**
  - **Given** 默认模式
  - **When** 匹配
  - **Then** `*` 和 `?` 不跨路径分隔符 `/`（`path:` 模式除外）

- **AC6 — 连续通配符**
  - **Given** pattern 含 `**` 或 `??`
  - **When** 匹配
  - **Then** `**` 等效于 `*`，`??` 保持两个字符语义

- **AC7 — 大小写与 Unicode**
  - **Given** 默认大小写不敏感（除非 `case:` 修饰）+ NFC 统一化后匹配
  - **When** 输入 `*报告*`
  - **Then** 匹配所有含 "报告" 的文件名

- **AC8 — 纯通配符**
  - **Given** 用户输入 `*`
  - **When** 查询
  - **Then** 返回全部文件，触发全量扫描警告，截断前 1000 条

- **AC9 — 组合**
  - **Given** 通配符与布尔运算组合
  - **When** 查询 `*.pdf report`、`*.pdf|*.doc`、`!*.tmp`
  - **Then** 分别正确生成 AND(wildcard, term)、OR(wildcard, wildcard)、NOT(wildcard)

- **AC10 — 性能**
  - **Given** 1M 记录
  - **When** 执行 `*.pdf` 查询
  - **Then** 耗时 < 50ms（利用优化路径）

**实现**：`Sources/Search/PatternMatcher.swift` · **测试**：`Tests/SearchTests/PatternMatcherTests.swift`

### REQ-1.1-03 正则表达式支持（regex: 前缀） ✅ 🖥️ P0

**用户故事**

作为「高级用户」，我希望用 `regex:pattern` 前缀输入正则表达式精确匹配文件名模式，以便实现复杂的模式匹配（如 `^report_\d{4}`）。

**验收标准（Given / When / Then）**

- **AC1 — `regex:` 前缀激活**
  - **Given** 用户输入 `regex:^report_\d{4}`
  - **When** QueryParser 检测 `regex:` 前缀
  - **Then** 提取其后到行尾或下一修饰符前的全部文本作为正则 pattern，激活正则匹配模式

- **AC2 — ICU/Swift Regex 语法**
  - **Given** Swift `Regex`（基于 ICU）方言
  - **When** 编译正则
  - **Then** 支持 `^` `$`、`.`、`[abc]`/`[a-z]`/`[^abc]`、`*` `+` `?`、`{n}` `{n,}` `{n,m}`、`()`、`|`、`\d` `\w` `\s`、Unicode 属性 `\p{Han}`

- **AC3 — 匹配范围与大小写**
  - **Given** 默认匹配文件名 + 大小写不敏感
  - **When** 切换修饰符
  - **Then** `path:regex:` 匹配完整路径；`case:regex:` 切换为大小写敏感；修饰符可组合（`case:regex:^Report`）

- **AC4 — 隐式锚定**
  - **Given** 正则默认隐式匹配整个文件名
  - **When** 查询 `regex:report`
  - **Then** 等效于 `regex:^.*report.*$`（子串匹配）；`^` `$` 锚定到文件名/路径开头结尾

- **AC5 — `regex:` 内空格非 AND**
  - **Given** 用户输入 `regex:hello world`
  - **When** QueryParser 解析
  - **Then** 视为单个正则 `regex("hello world")`，不拆分为 AND

- **AC6 — 无效正则**
  - **Given** 无效正则语法（如 `regex:[`）
  - **When** 编译
  - **Then** 返回 ParseError，包含编译错误描述（如 "unterminated character class"）

- **AC7 — 与布尔运算组合**
  - **Given** 正则与布尔运算组合
  - **When** 查询 `report regex:\d{4}`、`!regex:\.tmp$`
  - **Then** 分别生成 `AND(term("report"), regex("\\d{4}"))`、`NOT(regex("\\.tmp$"))`

- **AC8 — Unicode 与 NFC**
  - **Given** 含中文字符的文件名 + NFC 统一化后匹配
  - **When** 查询 `regex:\p{Han}+` 或 `regex:报告_\d{4}`
  - **Then** 匹配含中文字符的文件名（如 "报告_2025.pdf"）

- **AC9 — 实现策略**
  - **Given** 正则 pattern 预编译缓存（相同 pattern 只编译一次）
  - **When** 查询
  - **Then** 默认全量扫描；可结合 `ext:`/`parent:` 先缩小候选集再应用正则

- **AC10 — 性能**
  - **Given** 1M 记录
  - **When** 简单正则查询
  - **Then** 耗时 < 200ms

**实现**：`Sources/Search/PatternMatcher.swift` · **测试**：`Tests/SearchTests/PatternMatcherTests.swift`

### REQ-1.1-04 路径限定符 ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望用 `\` 路径限定符、`parent:` 限定符、`path:` 修饰符缩小搜索范围到特定路径，以便减少无关结果、限定目录范围。

**验收标准（Given / When / Then）**

- **AC1 — `\` 路径限定**
  - **Given** 用户输入 `Documents\ report`
  - **When** QueryParser 解析
  - **Then** 生成 `AND(path_filter("Documents"), term("report"))`，`\` 前为路径过滤器、`\` 后为搜索词

- **AC2 — `parent:` 限定**
  - **Given** 用户输入 `parent:~/Documents report`
  - **When** QueryParser 解析
  - **Then** 生成 `AND(parent_filter("~/Documents"), term("report"))`，限定为 `~/Documents` 及其子目录（`parentPath.hasPrefix(expandedPath)` 前缀匹配）

- **AC3 — `path:` 修饰符**
  - **Given** 用户输入 `path:report`
  - **When** 匹配
  - **Then** 搜索完整路径（`path + "/" + name`）而非仅文件名

- **AC4 — `~` 展开**
  - **Given** 用户输入 `parent:~/Documents`
  - **When** 路径展开
  - **Then** `~` 展开为 `/Users/<user>/Documents`

- **AC5 — NFC 与大小写**
  - **Given** 路径部分
  - **When** 匹配
  - **Then** NFC 统一化 + 大小写不敏感（macOS 文件系统默认）；`documents\ report` 匹配 "Documents/" 路径

- **AC6 — `parent:` 仅 parentPath**
  - **Given** `parent:` 限定符
  - **When** 匹配
  - **Then** 仅匹配 parentPath 前缀，不匹配文件名

- **AC7 — 嵌套路径**
  - **Given** 用户输入 `Documents\Projects\ report`（连续 `\` 表示路径层级）
  - **When** 匹配
  - **Then** 路径含 "Documents/.../Projects/" 且文件名含 "report"

- **AC8 — 组合与引号**
  - **Given** 路径限定与布尔运算组合 + 路径含空格
  - **When** 查询 `<Documents\|Desktop>\ report`、`parent:"My Documents" report`、`parent:"/Library/Frameworks"`
  - **Then** 分别生成 `(路径含 Documents OR 路径含 Desktop) AND 文件名含 report`、引号包裹空格路径

- **AC9 — 不存在路径**
  - **Given** 用户输入 `parent:/nonexistent foo`
  - **When** 查询
  - **Then** 返回 0 结果，不报错

**实现**：`Sources/Search/QueryTerm.swift` · **测试**：`Tests/SearchTests/QueryParserTests.swift`

### REQ-1.1-05 搜索修饰符 ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望用 `case:`/`file:`/`folder:`/`ext:`/`path:` 修饰符精细控制搜索行为（大小写敏感、仅文件、仅目录、按扩展名过滤、匹配完整路径），以便精确表达搜索意图。

**验收标准（Given / When / Then）**

- **AC1 — `case:` 局部大小写敏感**
  - **Given** 用户输入 `case:Report`
  - **When** 匹配
  - **Then** 仅匹配含大写 "Report" 的文件，不匹配 "report"、"REPORT"

- **AC2 — `file:` / `folder:` 全局**
  - **Given** 用户输入 `file: report` 或 `folder: projects`
  - **When** 查询
  - **Then** `file:` 仅返回文件（isDirectory == false）；`folder:` 仅返回目录（isDirectory == true）；二者为全局过滤器，出现在查询任意位置均生效

- **AC3 — `ext:` 全局扩展名**
  - **Given** 用户输入 `ext:pdf` 或 `ext:pdf;doc;xlsx`
  - **When** 查询
  - **Then** 单个扩展名或多扩展名（分号分隔，OR 语义）；`ext:` 全局生效；值不含前导 `.`；比较大小写不敏感（`ext:PDF` 匹配 ".pdf"）；`ext:` 空值匹配无扩展名文件

- **AC4 — `path:` 局部**
  - **Given** 用户输入 `path:report`
  - **When** 匹配
  - **Then** 匹配完整路径中的 "report"，局部修饰紧跟 term

- **AC5 — 修饰符作用域**
  - **Given** 作用域规则：`case:`/`path:` 局部（仅修饰紧跟 term）；`file:`/`folder:`/`ext:` 全局
  - **When** 查询 `case:Report draft`
  - **Then** "Report" 大小写敏感 AND "draft" 大小写不敏感

- **AC6 — 修饰符叠加**
  - **Given** 用户输入 `case:file:Report`
  - **When** 解析
  - **Then** 修饰符叠加正确：大小写敏感 + 仅文件 + 匹配 "Report"

- **AC7 — 修饰符对短语有效**
  - **Given** 用户输入 `case:"Annual Report"`
  - **When** 匹配
  - **Then** 大小写敏感匹配短语 "Annual Report"

- **AC8 — 否定修饰符**
  - **Given** 否定修饰符语义
  - **When** 查询 `!ext:tmp`、`!file:`、`!folder:`、`!case:xxx`
  - **Then** `!ext:tmp` 排除 .tmp 文件（等价 `!*.tmp`）；`!file:` 仅文件夹（等价 `folder:`）；`!folder:` 仅文件（等价 `file:`）；`!case:xxx` 等效 `!xxx`（大小写不敏感否定）

- **AC9 — 无效修饰符**
  - **Given** 无效修饰符前缀（如 `foo:bar`）
  - **When** 解析
  - **Then** 当作普通 term 处理，不报错（向后兼容未来修饰符）

- **AC10 — 与通配符/正则组合**
  - **Given** 用户输入 `case:*.PDF` 或 `case:regex:^Report`
  - **When** 匹配
  - **Then** 分别为大小写敏感通配符、大小写敏感正则

**实现**：`Sources/Search/QueryTerm.swift`, `Sources/Search/SearchFilter.swift` · **测试**：`Tests/SearchTests/QueryParserTests.swift`, `Tests/SearchTests/SearchFilterTests.swift`

### REQ-1.1-06 SearchQuery 增强模型 ✅ 🖥️ P0

**用户故事**

作为「系统」，我希望有一个结构化的查询模型（QueryAST + TermOptions + QueryOptions）承载 QueryParser 的输出，以便 SearchCoordinator 能高效求值布尔运算、通配符、正则、修饰符。

**验收标准（Given / When / Then）**

- **AC1 — QueryAST 节点**
  - **Given** `indirect enum QueryAST: Sendable, Codable`
  - **When** 定义
  - **Then** 支持 empty / term / wildcard / regex / and / or / not / phrase 八种节点

- **AC2 — TermOptions**
  - **Given** `struct TermOptions: Sendable, Codable, OptionSet`
  - **When** 定义
  - **Then** 支持 caseSensitive / matchPath / filesOnly / foldersOnly 四个选项（UInt8 位运算）

- **AC3 — QueryOptions**
  - **Given** 全局查询选项（从 AST 提取）
  - **When** 提取
  - **Then** 提取 fileFilter / folderFilter / extensions（Set<String>?，nil = 不过滤）/ parentPath

- **AC4 — SearchQuery 字段**
  - **Given** 增强的 `SearchQuery: Sendable`
  - **When** 定义
  - **Then** 包含 id + rawText（原始文本，用于显示/历史）+ ast + options + limit（默认 100）

- **AC5 — Codable 传输**
  - **Given** `QueryAST` 和 `TermOptions` 实现 Codable
  - **When** 序列化
  - **Then** 支持通过 IPC 传输，Codable 往返零丢失

- **AC6 — AST 求值**
  - **Given** 递归求值：AND = 左 ∩ 右，OR = 左 ∪ 右，NOT = 全集 - 子树
  - **When** 求值
  - **Then** 嵌套组合（如 `<a|b> !c d`）正确求值；优化先求值限制性最强子树

- **AC7 — 全局过滤器优先**
  - **Given** 全局过滤器（file/folder/ext）
  - **When** 求值
  - **Then** 在 AST 求值前先应用，减少候选集（如 `file: ext:pdf a b` 先过滤文件+PDF 再匹配 a AND b）

- **AC8 — 节点求值策略**
  - **Given** 各节点类型
  - **When** 求值
  - **Then** term 复用 FullSubstringMap/Trie/TrigramIndex；wildcard 逐一匹配通配符 pattern；regex 逐一匹配正则 pattern；phrase 精确子串匹配（连续位置）

- **AC9 — MatchType 扩展**
  - **Given** `enum MatchType`
  - **When** 扩展
  - **Then** 新增 wildcard / regex / phrase 三种类型（v1.1）

- **AC10 — IPC 后向兼容**
  - **Given** IPC 协议：`IPCQueryRequest` 新增 `ast` 和 `options` 可选字段
  - **When** 跨版本通信
  - **Then** v1.0 daemon 收到 v1.1 查询忽略 AST 不崩溃（回退纯文本子串搜索）；v1.1 daemon 收到纯文本查询用 QueryParser 在 daemon 端解析；`SearchQuery` 可从纯文本构造（内部调用 QueryParser）

- **AC11 — 性能**
  - **Given** AST 构造 + 全局选项提取
  - **When** 执行
  - **Then** 耗时 < 0.1ms

**实现**：`Sources/Search/SearchTypes.swift`, `Sources/Search/QueryTerm.swift` · **测试**：`Tests/SearchTests/SearchTypesTests.swift`

### REQ-1.1-07 REPL 搜索历史持久化 ✅ 🖥️ P1

**用户故事**

作为「用户」，我希望在 REPL 中用上下箭头浏览、Ctrl+R 增量搜索历史查询（含高级语法查询），退出后重新进入历史仍可用，以便复用复杂查询。

**验收标准（Given / When / Then）**

- **AC1 — 历史持久化与上限**
  - **Given** v1.0 基线（REQ-0.6-03）：readline 历史持久化到 `~/.deep-finder/history`
  - **When** 写入历史
  - **Then** 最多 1000 条，超过时删除最旧条目；历史文件权限 600

- **AC2 — 历史条目元数据**
  - **Given** v1.1 增强：历史条目包含时间戳和结果数
  - **When** 写入
  - **Then** 文件格式为 `<unix_timestamp> <result_count> <query_text>`，`#` 开头为注释（readline 忽略）；readline 读取时跳过时间戳前缀仅取查询文本；历史写入使用自定义函数（非 `write_history`）保留元数据

- **AC3 — readline 行为不变**
  - **Given** readline 内置行为
  - **When** 用户上下箭头浏览、Ctrl+R 增量搜索
  - **Then** 行为与 v1.0 一致

- **AC4 — 历史去重**
  - **Given** 连续重复查询
  - **When** 写入
  - **Then** 不记录；空查询不写入历史

- **AC5 — `:history` 命令**
  - **Given** v1.1 新增 `:history` 命令
  - **When** 执行 `:history` 或 `:history N`
  - **Then** 分别显示最近 20 条或最近 N 条（带序号、时间戳、结果数）

- **AC6 — `:history clear`**
  - **Given** v1.1 新增 `:history clear`
  - **When** 执行

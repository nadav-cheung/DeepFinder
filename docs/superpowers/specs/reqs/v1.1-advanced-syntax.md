### REQ-1.1-01 QueryParser — 布尔表达式解析 ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望用空格（AND）、`|`（OR）、`!`（NOT）以及短语、分组组合查询词，以便像 Everything 一样灵活地表达组合搜索需求。

**验收标准（Given / When / Then）**

- **AC1 — AND 运算（空格）**
  - **Given** 用户输入 `a b`
  - **When** QueryParser 解析
  - **Then** 生成 `AND(term("a"), term("b"))`

- **AC2 — OR 运算（`|`）**
  - **Given** 用户输入 `a|b`
  - **When** QueryParser 解析
  - **Then** 生成 `OR(term("a"), term("b"))`

- **AC3 — NOT 运算（`!`）**
  - **Given** 用户输入 `!a`
  - **When** QueryParser 解析
  - **Then** 生成 `NOT(term("a"))`；`!` 作为独立 token 前缀作用于紧随其后的 primary

- **AC4 — 精确短语（`"..."`）**
  - **Given** 用户输入 `"annual report"`
  - **When** QueryParser 解析
  - **Then** 引号未被 tokenizer 视为特殊字符，按空格切分为 AND；实现无独立 phrase 节点，`"annual report"` 等效 `AND(term("annual"), term("report"))`（引号字符作为 term 内容的一部分保留）

- **AC5 — 分组（`(...)`）**
  - **Given** 用户输入 `(a|b) c`
  - **When** QueryParser 解析
  - **Then** 生成 `AND(OR(term("a"), term("b")), term("c"))`；分组使用圆括号 `(` `)`

- **AC6 — 运算符优先级**
  - **Given** 优先级从高到低为：分组 > NOT > AND > OR；连续空格视为单个 AND；`|` 两侧无需空格
  - **When** 解析 `a|b c`
  - **Then** 顶层 `parseQuery` 收集多个 or_expr 后隐式 AND，生成 `AND(OR(term("a"), term("b")), term("c"))`（AND 在顶层作用域优先于 OR 绑定）

- **AC7 — 嵌套分组**
  - **Given** 用户输入 `(a (b|c))`
  - **When** QueryParser 解析
  - **Then** 生成 `AND(term("a"), OR(term("b"), term("c")))`

- **AC8 — 空查询**
  - **Given** 空查询或仅含空格的查询（`` 或 `   `）
  - **When** QueryParser 解析
  - **Then** 返回空 terms 数组，不报错

- **AC9 — 引号字符**
  - **Given** 用户输入 `"hello world`（未闭合）
  - **When** QueryParser 解析
  - **Then** 引号非特殊字符，按空格切分生成 `AND(term("\"hello"), term("world"))`（`"` 作为字面量保留在 term 中）

- **AC10 — 空分组**
  - **Given** 用户输入 `()`（空分组）
  - **When** QueryParser 解析
  - **Then** 内部 inner 为空数组，返回 `.text("")`；不抛出解析错误（容错策略）

- **AC11 — NFC + lowercased**
  - **Given** 已解析的 term
  - **When** QueryParser 完成解析
  - **Then** tokenizer 不对 term 做 NFC/lowercased（term 保留原始大小写与编码）；大小写不敏感由匹配层在 `PatternMatcher`/`SearchQuery` 中统一处理；Unicode 词（不含空格、`|`、`(`、`)`、`\\` 的连续字符序列）被正确切分

- **AC12 — 性能**
  - **Given** 1000 字符复杂查询
  - **When** 解析
  - **Then** 递归下降解析 O(n)；无独立基准断言（QueryParserTests 无 `measure` block 或耗时阈值，目标 < 0.1ms 为设计目标，未 CI 断言）

**实现**：`Sources/Search/QueryTerm.swift` · **测试**：`Tests/SearchTests/QueryParserTests.swift`

### REQ-1.1-02 通配符匹配（* 和 ?） ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望用 `*`（零个或多个任意字符）和 `?`（恰好一个任意字符）做模式匹配，以便用最直观的语法定位 PDF、扩展名填充位等文件。

**验收标准（Given / When / Then）**

- **AC1 — `*` 匹配**
  - **Given** 用户输入 `*.pdf`
  - **When** 通配符匹配
  - **Then** 匹配零个或多个任意字符（含 "report.pdf"、".pdf"）

- **AC2 — `?` 匹配**
  - **Given** 用户输入 `report_??.xlsx`
  - **When** 通配符匹配
  - **Then** 匹配恰好两个字符（"report_01.xlsx"、"report_Q2.xlsx"），不匹配 "report.xlsx"、"report_1.xlsx"

- **AC3 — 匹配范围**
  - **Given** `PatternMatcher.matchWildcard` 针对 input 字符串匹配
  - **When** 查询
  - **Then** 匹配整个 input 字符串（pattern 必须完整匹配 input，无子串语义）；实际搜索管线未将 wildcard term 路由到 `matchWildcard`，wildcard 字符串降级为子串搜索（详见 AC9）

- **AC4 — `*.ext` 优化**
  - **Given** 通配符 pattern 仅尾部 `*`（`*.ext` 或 `prefix*`）
  - **When** 查询
  - **Then** 当前实现未提供 `*.ext`/`prefix*` 专用优化路径；`matchWildcard` 走统一的两指针回溯算法

- **AC5 — 路径分隔符**
  - **Given** 默认模式
  - **When** 匹配
  - **Then** `*` 和 `?` 按 `matchWildcard` 算法匹配任意字符（含 `/`），无路径分隔符特殊处理

- **AC6 — 连续通配符**
  - **Given** pattern 含 `**` 或 `??`
  - **When** 匹配
  - **Then** `**` 等效于 `*`，`??` 保持两个字符语义

- **AC7 — 大小写与 Unicode**
  - **Given** `matchWildcard` 内部对 pattern 与 input 做 `precomposedStringWithCanonicalMapping.lowercased()` 后比较
  - **When** 输入 `*报告*`
  - **Then** 匹配所有含 "报告" 的 input

- **AC8 — 纯通配符**
  - **Given** 用户输入 `*`
  - **When** 查询
  - **Then** wildcard term 经 `textOnlyQuery` 降级为子串 `*`，对文件名做子串匹配；无独立的"全量扫描警告"机制，结果受 SearchCoordinator `resultLimit`（默认 1000）截断

- **AC9 — 组合**
  - **Given** 通配符与布尔运算组合
  - **When** 查询 `*.pdf report`、`*.pdf|*.doc`、`!*.tmp`
  - **Then** AST 正确生成 `AND(wildcard, term)`、`OR(wildcard, wildcard)`、`NOT(wildcard)`；但搜索执行时 wildcard 不经 `matchWildcard`，而是 `textOnlyQuery` 扁平化为子串文本（如 `*.pdf`）送入 InMemoryIndex 做子串查询

- **AC10 — 性能**
  - **Given** `matchWildcard` 单次匹配
  - **When** 执行
  - **Then** 两指针回溯算法 O(m*n) 最坏复杂度；1M 记录级查询无基准（仅 `SearchBenchmarks` 有 reporting 基准，无 CI 阈值断言）

**实现**：`Sources/Search/PatternMatcher.swift` · **测试**：`Tests/SearchTests/PatternMatcherTests.swift`

### REQ-1.1-03 正则表达式支持（regex: 前缀） ✅ 🖥️ P0

**用户故事**

作为「高级用户」，我希望用 `regex:pattern` 前缀输入正则表达式精确匹配文件名模式，以便实现复杂的模式匹配（如 `^report_\d{4}`）。

**验收标准（Given / When / Then）**

- **AC1 — `regex:` 前缀激活**
  - **Given** 用户输入 `regex:^report_\d{4}`
  - **When** QueryParser 检测 `regex:` 前缀
  - **Then** `classifyWord` 检测 word 前缀 `regex:`，提取其后到 word 结束（下一个空白）的文本作为 pattern，生成 `.regex` token；不支持跨空格的多词 pattern（见 AC5）

- **AC2 — ICU/NSRegularExpression 语法**
  - **Given** `NSRegularExpression`（基于 ICU）方言
  - **When** 编译正则
  - **Then** 支持 `^` `$`、`.`、`[abc]`/`[a-z]`/`[^abc]`、`*` `+` `?`、`{n}` `{n,}` `{n,m}`、`()`、`|`、`\d` `\w` `\s`、Unicode 属性 `\p{Han}`

- **AC3 — 匹配范围与大小写**
  - **Given** `matchRegex` 固定 `.caseInsensitive` + `.anchorsMatchLines`
  - **When** 匹配
  - **Then** 大小写不敏感硬编码，无运行时切换；`case:regex:` 不被支持（`case:regex:^Report` 经 tokenizer 以第一个 `:` 切分为 `.modifier(key: "case", value: "regex:^Report")`，整个 word 当作 case 修饰符值）

- **AC4 — 隐式锚定**
  - **Given** 正则使用 `firstMatch`（子串匹配）
  - **When** 查询 `regex:report`
  - **Then** `matchRegex` 使用 `firstMatch(in:options:range:)`，匹配 input 任意位置的子串；`^`/`$` 锚定由 ICU 语义决定（启用 `.anchorsMatchLines`）

- **AC5 — `regex:` 内空格非 AND**
  - **Given** 用户输入 `regex:hello world`
  - **When** QueryParser 解析
  - **Then** `regex:` 前缀使整个 word（直到下一个空白）成为一个 `.regex` token；`regex:hello world` 因 tokenizer 在空格处切分，实际生成 `AND(regex("hello"), term("world"))`（regex 仅覆盖空格前的部分）

- **AC6 — 无效正则**
  - **Given** 无效正则语法（如 `regex:[`）
  - **When** 编译
  - **Then** `matchRegex` 用 `try?` 捕获 `NSRegularExpression` 异常，返回 `false`（不抛出 ParseError，不崩溃）

- **AC7 — 与布尔运算组合**
  - **Given** 正则与布尔运算组合
  - **When** 查询 `report regex:\d{4}`、`!regex:\.tmp$`
  - **Then** 分别生成 `AND(term("report"), regex("\\d{4}"))`、`NOT(regex("\\.tmp$"))`；执行层 regex term 经 `textOnlyQuery` 重建为 `regex:...` 前缀字符串，最终作为子串文本查询

- **AC8 — Unicode 与 NFC**
  - **Given** 含中文字符的文件名
  - **When** 查询 `regex:\p{Han}+` 或 `regex:报告_\d{4}`
  - **Then** `NSRegularExpression` 基于 ICU 支持 Unicode 属性类，可匹配含中文字符的 input（如 "报告_2025.pdf"）；input 未单独 NFC 统一化

- **AC9 — 实现策略**
  - **Given** 正则 pattern 无预编译缓存
  - **When** 查询
  - **Then** 每次 `matchRegex` 调用重新编译 `NSRegularExpression`；默认全量扫描

- **AC10 — 性能**
  - **Given** `matchRegex` 单次匹配
  - **When** 执行
  - **Then** 1M 记录级正则查询无基准（`SearchBenchmarks` 仅 reporting 基准，无 CI 阈值断言）

**实现**：`Sources/Search/PatternMatcher.swift` · **测试**：`Tests/SearchTests/PatternMatcherTests.swift`

### REQ-1.1-04 路径限定符 ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望用 `\` 路径限定符、`parent:` 限定符、`path:` 修饰符缩小搜索范围到特定路径，以便减少无关结果、限定目录范围。

**验收标准（Given / When / Then）**

- **AC1 — `\` 路径限定**
  - **Given** 用户输入 `Documents\ report`
  - **When** QueryParser 解析
  - **Then** 生成 `AND(path_filter("Documents"), term("report"))`，`\` 前为路径过滤器、`\` 后为搜索词

- **AC2 — `parent:` 限定**
  - **Given** 无 `parent:` 修饰符键（`modifierKeys` 不含 `parent`）
  - **When** 用户输入 `parent:~/Documents report`
  - **Then** tokenizer 不识别 `parent:` 为修饰符，按字面量解析为 `text("parent:~/Documents")` + `text("report")`；不支持独立 `parent:` 限定符

- **AC3 — `path:` 修饰符**
  - **Given** `path` 在 `modifierKeys` 中
  - **When** 用户输入 `path:report`
  - **Then** 生成 `.modifier(key: "path", value: "report")`；`FilterPipeline.parse` 未为 `path` key 实现分支，该 modifier 被忽略，不产生匹配完整路径的过滤器

- **AC4 — `~` 展开**
  - **Given** 用户输入带 `~` 的路径限定
  - **When** 路径展开
  - **Then** search 层不对 pathQualifier 内容做 `~` 展开（`expandingTildeInPath` 仅在 CLI 层用于文件系统路径，不作用于查询 AST）

- **AC5 — NFC 与大小写**
  - **Given** 路径部分
  - **When** 匹配
  - **Then** pathQualifier 作为字面量字符串保留，无内置 NFC/大小写处理；匹配行为取决于下游子串查询（默认大小写不敏感）

- **AC6 — pathQualifier 与 parentPath**
  - **Given** pathQualifier term
  - **When** 匹配
  - **Then** 当前实现未将 pathQualifier 路由到 parentPath 前缀过滤；`textOnlyQuery` 将 pathQualifier 重建为 `<word>\ ` 字符串，作为子串文本参与查询

- **AC7 — 嵌套路径**
  - **Given** 用户输入 `Documents\Projects\ report`（多个连续 `\` 路径层级）
  - **When** tokenizer 解析
  - **Then** 每个 `\ ` 将其前缀作为独立 pathQualifier token；`Documents\` 后接 `Projects\` 生成两个 pathQualifier token `Documents`、`Projects`，再 AND `report`

- **AC8 — 组合与转义**
  - **Given** 路径限定与布尔运算组合
  - **When** 查询 `(Documents\|Desktop)\ report`、`Projects\ report`
  - **Then** tokenizer 对 `\|` 视为转义（字面量 `|`），对 `\<space>` 视为 pathQualifier 分隔；`\ ` 仅识别紧随其前的 word 为 pathQualifier

- **AC9 — 不存在路径**
  - **Given** 用户输入带不存在路径的查询
  - **When** 查询
  - **Then** 无独立路径校验机制；pathQualifier 作为子串参与查询，结果集为空（0 结果，不报错）

**实现**：`Sources/Search/QueryTerm.swift` · **测试**：`Tests/SearchTests/QueryParserTests.swift`

### REQ-1.1-05 搜索修饰符 ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望用 `case:`/`file:`/`folder:`/`ext:`/`path:` 修饰符精细控制搜索行为（大小写敏感、仅文件、仅目录、按扩展名过滤、匹配完整路径），以便精确表达搜索意图。

**验收标准（Given / When / Then）**

- **AC1 — `case:` 局部大小写敏感**
  - **Given** 用户输入 `case:Report`
  - **When** 匹配
  - **Then** 仅匹配含大写 "Report" 的文件，不匹配 "report"、"REPORT"

- **AC2 — `file:` / `folder:` 全局**
  - **Given** 用户输入 `file:foo` 或 `folder:bar`（需带非空值）
  - **When** 查询
  - **Then** `file:foo` 生成 `.modifier(key: "file", value: "foo")`，`FilterPipeline.parse` 将 `file` key 映射为 `.isFile` 过滤器（value 被忽略）；`folder:` 映射为 `.isDirectory`；二者为全局过滤器；空值 `file:` 被当作字面量文本（tokenizer 要求 `!value.isEmpty`）

- **AC3 — `ext:` 全局扩展名**
  - **Given** 用户输入 `ext:pdf` 或 `ext:pdf;doc;xlsx`
  - **When** 查询
  - **Then** 单个扩展名或多扩展名（分号分隔，`Set` OR 语义）；`ext:` 全局生效；值不含前导 `.`；比较大小写不敏感（`ext:PDF` 匹配 ".pdf"）；`ext:` 空值在 tokenizer 被当作字面量文本（不识别为修饰符），无"匹配无扩展名文件"语义

- **AC4 — `path:` 局部**
  - **Given** 用户输入 `path:report`
  - **When** 匹配
  - **Then** 生成 `.modifier(key: "path", value: "report")`；`FilterPipeline.parse` 未实现 `path` 分支，该 modifier 被忽略，不产生匹配完整路径的过滤器

- **AC5 — 修饰符作用域**
  - **Given** 作用域规则：所有修饰符经 `modifierPairs` 递归提取后由 `FilterPipeline` 全局应用；无"局部修饰符仅修饰紧跟 term"机制
  - **When** 查询 `case:Report draft`
  - **Then** `case:Report` 生成 `.modifier(key: "case", value: "Report")`，由 FilterPipeline 转为 `.caseSensitiveText("Report")` 过滤器；`draft` 作为子串查询；二者通过过滤器管线组合

- **AC6 — 修饰符叠加**
  - **Given** 用户输入 `case:file:Report`
  - **When** 解析
  - **Then** tokenizer 以第一个 `:` 切分，key=`case`、value=`file:Report`，生成单个 `.modifier(key: "case", value: "file:Report")`；不支持 `case:file:` 链式叠加

- **AC7 — 修饰符对引号文本**
  - **Given** 用户输入 `case:"Annual Report"`
  - **When** 解析
  - **Then** tokenizer 不识别引号为特殊字符，按 `:` 切分得到 `.modifier(key: "case", value: "\"Annual")`，后续 `Report"` 作为独立 term；无独立短语节点

- **AC8 — 否定修饰符**
  - **Given** 否定修饰符语义
  - **When** 查询 `!ext:tmp`、`!file:foo`、`!case:xxx`
  - **Then** `!ext:tmp` 生成 `.not(.modifier(key: "ext", value: "tmp"))`，但 `modifierPairs` 对 `.not` 子树递归提取 modifier 后转为全局 `.extensionFilter(["tmp"])` 过滤器（非否定，过滤为"仅 tmp"而非"排除 tmp"）；`!file:` 同理生成 `.isFile` 过滤器（非否定）

- **AC9 — 无效修饰符**
  - **Given** 无效修饰符前缀（如 `foo:bar`，key 不在 `modifierKeys` 集合）
  - **When** 解析
  - **Then** 当作普通 term 处理（`.text("foo:bar")`），不报错

- **AC10 — 与通配符/正则组合**
  - **Given** 用户输入 `case:*.PDF` 或 `case:regex:^Report`
  - **When** 解析
  - **Then** tokenizer 以第一个 `:` 切分为 `.modifier(key: "case", value: "*.PDF")`，整个 word 被当作 modifier（不识别 `case:` 后接的 `*`/`regex:` 前缀）；不支持修饰符前缀叠加 wildcard/regex 节点

**实现**：`Sources/Search/QueryTerm.swift`, `Sources/Search/SearchFilter.swift` · **测试**：`Tests/SearchTests/QueryParserTests.swift`, `Tests/SearchTests/SearchFilterTests.swift`

### REQ-1.1-06 SearchQuery 增强模型 ✅ 🖥️ P0

**用户故事**

作为「系统」，我希望有一个结构化的查询模型（QueryAST + TermOptions + QueryOptions）承载 QueryParser 的输出，以便 SearchCoordinator 能高效求值布尔运算、通配符、正则、修饰符。

**验收标准（Given / When / Then）**

- **AC1 — QueryTerm 节点**
  - **Given** `indirect enum QueryTerm: Equatable, Sendable`
  - **When** 定义
  - **Then** 支持 text / and / or / not / wildcard / regex / modifier / pathQualifier 八种节点（无独立 phrase/empty 节点）

- **AC2 — ParsedQuery**
  - **Given** `struct ParsedQuery: Equatable, Sendable`
  - **When** 定义
  - **Then** 包含 `terms: [QueryTerm]` 和 `rawQuery: String`；无独立 TermOptions OptionSet 类型

- **AC3 — Modifier 提取**
  - **Given** 全局查询选项（从 AST 递归提取）
  - **When** 通过 `modifierPairs` 提取
  - **Then** 提取所有 `.modifier(key:value:)` 节点（含嵌套在 and/or/not 子树中的），返回 `[(key, value)]`；由 `FilterPipeline.parse(from:)` 转换为 `SearchFilter[]`

- **AC4 — SearchQuery 字段**
  - **Given** `SearchQuery: Sendable`
  - **When** 定义
  - **Then** 包含 `rawQuery: String`（原始文本）+ `normalizedQuery: String`（NFC + lowercased，用于匹配）；无 ast/options/limit 字段；limit 由 IPC 层 `IPCRequest.query(query:limit:)` 单独传递

- **AC5 — Codable 传输**
  - **Given** `QueryTerm` 实现 `Sendable`（未声明 Codable）
  - **When** 序列化
  - **Then** IPC 协议传输纯文本 query 字符串（`IPCRequest.query(String, limit:)`），daemon 端用 `QueryParser` 解析；无 AST 跨进程序列化

- **AC6 — AST 扁平化**
  - **Given** `textOnlyQuery` 将 AST 重建为扁平字符串
  - **When** 执行
  - **Then** 递归 walk terms，移除 modifier 节点，保留 text/wildcard/regex/pathQualifier，按空格连接（or 子树用 `|` 连接，not 子树加 `!` 前缀）；非完整布尔求值（AND/∪/∩），用于喂给 InMemoryIndex 做子串查询

- **AC7 — 全局过滤器优先**
  - **Given** 全局过滤器（file/folder/ext/case/size/dm/...）
  - **When** 求值
  - **Then** `IPCServer.dispatchRequest` 先 `QueryParser.parse` 提取 modifierPairs → `FilterPipeline.parse` 构造过滤器，再用 `textOnlyQuery` 做子串查询，最后对结果应用 FilterPipeline（AND 过滤）

- **AC8 — 节点求值策略**
  - **Given** 各节点类型
  - **When** 求值
  - **Then** `textOnlyQuery` 扁平化后由 `InMemoryIndex.search` 做子串查询（FullSubstringMap/Trie/TrigramIndex）；wildcard/regex 节点不被路由到 `PatternMatcher.matchWildcard`/`matchRegex`（这些工具函数当前无生产调用方）；无独立 phrase 求值

- **AC9 — MatchType**
  - **Given** `enum MatchType: Int, Codable, Comparable, Sendable`
  - **When** 定义
  - **Then** 仅 exact(0) / prefix(1) / pinyin(2) / substring(3) 四种（v1.0 基线）；未新增 wildcard/regex/phrase 类型

- **AC10 — IPC 后向兼容**
  - **Given** IPC 协议：`IPCRequest.query(_ query: String, limit: Int?)` 传输纯文本字符串
  - **When** 跨版本通信
  - **Then** daemon 端解析 query 字符串（`QueryParser.parse` → modifierPairs + textOnlyQuery）；无 `ast`/`options` 字段；`SearchQuery` 从纯文本构造（`SearchQuery(_ query: String)`），不做 QueryParser 解析（仅 NFC + lowercased）

- **AC11 — 性能**
  - **Given** QueryParser.parse 构造 + modifierPairs 提取
  - **When** 执行
  - **Then** 无独立基准断言（`SearchBenchmarks` 仅 reporting 基准，覆盖 index 构建/搜索/排序，无 QueryParser 计时）

### REQ-1.1-07 REPL 搜索历史持久化 ✅ 🖥️ P1

**用户故事**

作为「用户」，我希望在 REPL 中用上下箭头浏览、Ctrl+R 增量搜索历史查询（含高级语法查询），退出后重新进入历史仍可用，以便复用复杂查询。

**验收标准（Given / When / Then）**

- **AC1 — 历史持久化与上限**
  - **Given** REPL 使用 libedit（Darwin.readline）原生 `_write_history` + `_history_truncate_file(path, 1000)` 持久化
  - **When** 写入历史
  - **Then** libedit 截断到最多 1000 条；文件权限由 libedit 默认决定（`REPLHistory` actor 单独提供 0600 权限，但 REPL 未接入该 actor）

- **AC2 — 历史条目格式**
  - **Given** libedit 原生格式
  - **When** 写入
  - **Then** 文件格式为 libedit 标准格式（每行一条查询文本，无时间戳/结果数前缀元数据）；`REPLHistory` actor 提供独立的字符串列表 API（`add`/`recent`/`search`/`clear`），但未被 REPL 调用

- **AC3 — readline 行为不变**
  - **Given** libedit 内置行为
  - **When** 用户上下箭头浏览、Ctrl+R 增量搜索
  - **Then** 行为由 libedit 提供，与 v1.0 一致

- **AC4 — 历史去重**
  - **Given** 连续重复查询
  - **When** 写入
  - **Then** libedit 原生 `_add_history` 不做去重（连续重复会被记录）；`REPLHistory` actor 提供 `add` 去重（`entries.last == entry` 跳过），但 REPL 未使用该 actor；空查询在 REPL 入口被跳过（trimmed 空串不调用 `_add_history`）

- **AC5 — `:history` 命令**
  - **Given** `REPLCommand` 枚举不包含 `history`（仅 help/quit/stats/config/open/reveal/daemon/explain/dataPreview/undo）
  - **When** 执行 `:history`
  - **Then** 返回 `(nil, [], false)`（未识别命令，无操作）；无 `:history` 或 `:history N` 命令

- **AC6 — `:history clear`**
  - **Given** 无 `:history` 命令
  - **When** 执行 `:history clear`
  - **Then** 同 AC5，未识别命令，无操作；`REPLHistory` actor 提供 `clear()` 方法（清空内存条目，需后续 `save()` 持久化），但 REPL 未接入

- **AC7 — 历史文件损坏恢复**
  - **Given** 历史文件损坏（非法格式）
  - **When** libedit `_read_history` 加载
  - **Then** libedit 内部处理损坏行（行为未自定义）；`REPLHistory.load` 跳过空行（`filter { !$0.isEmpty }`），无 warning 日志机制

- **AC8 — 高级语法查询保存**
  - **Given** 高级语法查询（`*.pdf report`、`regex:^report_\d{4}`、`(a|b) !c`）
  - **When** 保存与恢复
  - **Then** libedit 按行保存与恢复原始查询字符串，高级语法字符不破坏回显

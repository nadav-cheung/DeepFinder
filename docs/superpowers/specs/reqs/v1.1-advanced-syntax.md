# v1.1 — 高级搜索语法

对标 Windows Everything 搜索语法。在 v1.0 基础子串搜索之上增加布尔运算、通配符、正则、路径限定、修饰符和搜索历史。

7 项 REQ (P0: 6, P1: 1)

### REQ-1.1-01 QueryParser — 布尔表达式解析 📋 🖥️ P0

**用户场景**：用户输入 `report 2026` → 只看到同时包含 "report" 和 "2026" 的文件。输入 `report|memo` → 看到包含 "report" 或 "memo" 的文件。输入 `report !draft` → 看到包含 "report" 但不含 "draft" 的文件。三种布尔运算符覆盖绝大多数组合搜索需求。

**操作流程**：
1. 用户输入查询文本 → CLI 发送原始文本到 daemon
2. daemon 端 QueryParser 将原始文本解析为结构化 AST
3. AST 传递给 SearchCoordinator → 执行搜索 → 返回结果

**语法规则**（对标 Everything）：

| 运算符 | 语法 | 含义 | 示例 |
|--------|------|------|------|
| AND | 空格 | 两个词都出现 | `report 2026` = 包含 "report" 且包含 "2026" |
| OR | `\|` | 任一词出现 | `report\|memo` = 包含 "report" 或 "memo" |
| NOT | `!` | 排除包含该词的文件 | `report !draft` = 包含 "report" 但不含 "draft" |
| 分组 | `< >` | 显式优先级分组 | `<a\|b> c` = (a OR b) AND c |
| 短语 | `" "` | 精确短语匹配（含空格） | `"annual report"` = 精确包含 "annual report" |

**优先级**（从高到低）：
1. `"exact phrase"` — 精确短语
2. `<grouping>` — 显式分组
3. `!NOT` — 否定（一元前缀）
4. `space AND` — 交集
5. `|OR` — 并集

**解析规则**：
- 连续空格视为单个 AND
- `|` 两侧无需空格：`a|b` 等效于 `a | b`
- `!` 紧跟词：`!draft`，`! draft`（有空格）也合法
- 双引号内空格是短语的一部分，不视为 AND
- 双引号内 `|` 和 `!` 是字面量，不是运算符
- 未闭合的双引号视为从该引号到行尾的短语
- 空查询或仅含空格的查询返回空结果（不报错）
- Unicode 词：一个 "词" 是不含空格、`|`、`<`、`>`、`"` 的连续字符序列

**验收标准**：
- [ ] QueryParser 将原始查询文本解析为 `QueryAST`
- [ ] AND 运算（空格）：`a b` → AND(term("a"), term("b"))
- [ ] OR 运算（`|`）：`a|b` → OR(term("a"), term("b"))
- [ ] NOT 运算（`!`）：`!a` → NOT(term("a"))
- [ ] 精确短语（`"..."`）：`"annual report"` → phrase("annual report")
- [ ] 分组（`<...>`）：`<a|b> c` → AND(OR(term("a"), term("b")), term("c"))
- [ ] 运算符优先级：NOT > AND > OR（`a|b c` → OR(term("a"), AND(term("b"), term("c")))）
- [ ] 嵌套分组：`<a <b|c>>` → AND(term("a"), OR(term("b"), term("c")))
- [ ] 空查询返回空 AST（不报错）
- [ ] 未闭合引号视为到行尾的短语
- [ ] 错误处理：`<>` 空分组返回解析错误，位置信息包含在错误消息中
- [ ] 解析后的 term 经过 NFC 统一化 + lowercased（除非被 `case:` 修饰符覆盖）
- [ ] 性能：解析 1000 字符复杂查询 < 0.1ms

**测试覆盖**：
- [ ] 简单 AND：`a b c` → AND(AND(term("a"), term("b")), term("c"))
- [ ] 简单 OR：`a|b|c` → OR(OR(term("a"), term("b")), term("c"))
- [ ] 简单 NOT：`!a` → NOT(term("a"))
- [ ] 混合：`a b|c` → OR(AND(term("a"), term("b")), term("c"))（AND 优先级高于 OR）
- [ ] NOT + AND：`a !b c` → AND(AND(term("a"), NOT(term("b"))), term("c"))
- [ ] 分组覆盖优先级：`<a|b> <c|d>` → AND(OR(term("a"), term("b")), OR(term("c"), term("d")))
- [ ] 短语：`"hello world"` → phrase("hello world")
- [ ] 短语 + AND：`"hello world" foo` → AND(phrase("hello world"), term("foo"))
- [ ] 短语内运算符字面量：`"a|b !c"` → phrase("a|b !c")
- [ ] 未闭合引号：`"hello world` → phrase("hello world")
- [ ] 空查询：`` → empty AST
- [ ] 仅空格：`   ` → empty AST
- [ ] Unicode：`报告 季度` → AND(term("报告"), term("季度"))
- [ ] 边界：`!` → 解析错误（NOT 后无 term）

### REQ-1.1-02 通配符匹配（* 和 ?） 📋 🖥️ P0

**用户场景**：用户输入 `*.pdf` → 看到所有 PDF 文件。输入 `report_??.xlsx` → 看到如 `report_01.xlsx`、`report_Q2.xlsx` 等恰好两个字符填充的文件名。通配符是最直观的模式匹配语法。

**操作流程**：
1. QueryParser 检测 term 中是否含 `*` 或 `?`
2. 含通配符的 term 标记为 `wildcard` 类型
3. 搜索时对候选项应用通配符匹配

**语法规则**：

| 通配符 | 含义 | 示例 |
|--------|------|------|
| `*` | 零个或多个任意字符 | `*.pdf` 匹配 "report.pdf", ".pdf" |
| `?` | 恰好一个任意字符 | `report_?.xlsx` 匹配 "report_1.xlsx" 但不匹配 "report.xlsx" |

**匹配范围**：
- 默认匹配文件名（不含路径）
- `path:` 修饰符切换为匹配完整路径
- 通配符匹配整个文件名（隐式锚定）：`*.pdf` 匹配 "report.pdf"，`*pdf` 也匹配 "report.pdf"
- 与 Everything 行为一致：通配符隐式匹配整个文件名（等价于 wfn/wholefilename 模式）

**实现策略**：
- 通配符 pattern 编译为 NFA 或逐字符扫描匹配
- `*` 和 `?` 默认不跨路径分隔符 `/`（`path:` 模式除外）
- 优化：`*.ext`（无其他通配符）可退化为 `extension == ext` 的直接过滤，避免全量扫描
- 优化：`prefix*`（仅尾部 `*`）退化为前缀查询，复用 Trie

**与其他语法组合**：
- `*.pdf report` → AND(wildcard("*.pdf"), term("report")) = PDF 文件中包含 "report"
- `*.pdf|*.doc` → OR(wildcard("*.pdf"), wildcard("*.doc")) = PDF 或 DOC 文件
- `!*.tmp` → NOT(wildcard("*.tmp")) = 排除所有 .tmp 文件

**验收标准**：
- [ ] `*` 匹配零个或多个任意字符
- [ ] `?` 匹配恰好一个任意字符
- [ ] 默认匹配文件名（不含路径）
- [ ] `path:*.pdf` 匹配完整路径中的 pattern
- [ ] `*.ext` 优化路径：退化为扩展名直接过滤
- [ ] `prefix*` 优化路径：退化为 Trie 前缀查询
- [ ] 连续通配符合并：`**` 等效于 `*`，`??` 保持两个字符语义
- [ ] 大小写不敏感（除非 `case:` 修饰）
- [ ] Unicode 通配符：`*报告*` 匹配所有含 "报告" 的文件名
- [ ] 纯通配符查询 `*` 返回全部文件（触发全量扫描警告，截断前 1000 条）
- [ ] 通配符与布尔运算正确组合（AND/OR/NOT）
- [ ] NFC 统一化后匹配

**测试覆盖**：
- [ ] `*.pdf` → ["report.pdf", "summary.pdf"], NOT ["report.doc"]
- [ ] `report_??.xlsx` → ["report_01.xlsx", "report_Q2.xlsx"], NOT ["report.xlsx", "report_1.xlsx"]
- [ ] `*report*` → 包含 "report" 的任何文件
- [ ] `???.txt` → 恰好 3 字符文件名的 .txt 文件
- [ ] `*.pdf report` → PDF 中含 "report" 的文件
- [ ] `!*.tmp` → 排除 .tmp 文件
- [ ] 空匹配：`*.zzzzz` → 0 结果（不报错）
- [ ] 性能：1M 记录中 `*.pdf` 查询 < 50ms（利用优化路径）

### REQ-1.1-03 正则表达式支持（regex: 前缀） 📋 🖥️ P0

**用户场景**：用户输入 `regex:^report_\d{4}` → 精确匹配以 "report_" 开头后跟四位数字的文件。高级用户用正则实现复杂的文件名模式匹配。

**操作流程**：
1. 用户输入 `regex:pattern` → QueryParser 检测 `regex:` 前缀
2. 提取 `regex:` 后面的全部文本作为正则 pattern
3. 编译为 Swift `Regex` 或 `NSRegularExpression`
4. 对候选文件名逐一匹配

**语法规则**：

| 语法 | 说明 |
|------|------|
| `regex:pattern` | 对文件名应用正则匹配 |
| `path:regex:pattern` | 对完整路径应用正则匹配 |
| `case:regex:pattern` | 大小写敏感正则 |

**正则方言**：Swift `Regex`（基于 ICU），支持：
- `^` `$` 锚定（文件名开头/结尾）
- `.` 任意字符
- `[abc]` `[a-z]` 字符类
- `[^abc]` 取反字符类
- `*` `+` `?` 量词
- `{n}` `{n,}` `{n,m}` 重复
- `()` 分组
- `|` 交替
- `\d` `\w` `\s` 等转义类
- Unicode 属性：`\p{Han}` 匹配中文字符

**与 Everything 行为一致**：
- `regex:` 模式下，空格和 `|` 是正则语法的一部分，不是布尔运算符
- `regex:` 后到行尾或到下一个修饰符前的全部文本是正则 pattern
- 修饰符可组合：`case:regex:^Report` = 大小写敏感正则
- 正则默认匹配整个文件名（隐式 `^...$` 锚定），除非 pattern 自行指定锚定位置

**实现策略**：
- 使用 Swift `Regex`（编译时类型安全）或 `NSRegularExpression`（运行时动态编译）
- 正则 pattern 预编译缓存：相同 pattern 只编译一次
- 正则查询默认全量扫描文件名，无法利用索引结构
- 性能优化：结合 `ext:` 或 `parent:` 等限定符先缩小候选集，再应用正则
- 无效正则语法 → 友好错误提示（包含正则编译错误信息）

**验收标准**：
- [ ] `regex:` 前缀激活正则匹配模式
- [ ] 支持 Swift Regex / ICU 正则完整语法
- [ ] 默认匹配文件名，`path:regex:` 匹配完整路径
- [ ] 默认大小写不敏感，`case:regex:` 切换为大小写敏感
- [ ] 正则隐式匹配整个文件名（`regex:report` 等效于 `regex:^.*report.*$`，即子串匹配）
- [ ] `^` 和 `$` 锚定到文件名（或路径）开头和结尾
- [ ] 无效正则语法 → ParseError 包含编译错误描述
- [ ] 正则与布尔运算可组合：`report regex:\d{4}` → AND(term("report"), regex("\\d{4}"))
- [ ] `regex:` 模式内空格不视为 AND：`regex:hello world` → regex("hello world")，不是 AND
- [ ] Unicode 正则：`regex:\p{Han}+` 匹配含中文字符的文件名
- [ ] NFC 统一化后匹配
- [ ] 性能：1M 记录简单正则查询 < 200ms

**测试覆盖**：
- [ ] `regex:^report_\d{4}` → ["report_2025.txt", "report_2026.txt"], NOT ["my_report_123.txt"]
- [ ] `regex:\.(pdf|doc)$` → PDF 或 DOC 文件
- [ ] `regex:^...$` 精确匹配：`regex:^report\.pdf$` → 仅 "report.pdf"
- [ ] 无效正则：`regex:[` → ParseError 含 "unterminated character class"
- [ ] 正则 + AND：`2026 regex:^report` → 文件名以 "report" 开头且含 "2026"
- [ ] 正则 + NOT：`!regex:\.tmp$` → 排除 .tmp 文件
- [ ] `path:regex:Documents/.*\.pdf` → 匹配 Documents 目录下 PDF 文件路径
- [ ] Unicode：`regex:报告_\d{4}` → ["报告_2025.pdf"]

### REQ-1.1-04 路径限定符 📋 🖥️ P0

**用户场景**：用户输入 `Documents\ report` → 只在路径含 "Documents" 的文件中搜索 "report"。输入 `parent:~/Documents report` → 只在 ~/Documents 及其子目录中搜索 "report"。路径限定缩小搜索范围，减少无关结果。

**操作流程**：
1. QueryParser 检测 `\`（路径限定符）或 `parent:` 前缀
2. 提取路径部分和搜索词部分
3. 搜索时先过滤路径，再匹配搜索词

**语法规则**：

| 语法 | 含义 | 示例 |
|------|------|------|
| `path\ term` | 路径中含 `path` 的文件中搜 `term` | `Documents\ report` = 路径含 "Documents" 且文件名含 "report" |
| `parent:path` | 限定在指定目录下搜索 | `parent:~/Documents report` = ~/Documents 下含 "report" 的文件 |
| `path:term` | 搜索完整路径而非仅文件名 | `path:report` = 路径或文件名含 "report" |

**`\` 路径限定符详解**：
- `\` 前的 token 是路径过滤器，`\` 后的部分是搜索词
- 路径部分支持 NFC 统一化 + 大小写不敏感匹配
- `~` 展开为用户 home 目录
- 路径部分可包含 `\` 自身：`Documents\Projects\ report` = 路径含 "Documents/.../Projects/" 且文件名含 "report"
- 注意：macOS 路径使用 `/` 作为分隔符，但 `\` 在搜索语法中是路径限定操作符。连续 `\` 表示路径层级（`Docs\Projects\` → 匹配路径含 "Docs/.../Projects/" 的文件）

**`parent:` 限定符详解**：
- `parent:~/Documents` → 限定搜索范围为 `~/Documents` 及其子目录
- `parent:"/Library/Frameworks"` → 空格路径需引号
- 支持路径展开：`~` → home 目录
- 匹配方式：`parentPath` 前缀匹配（`parentPath.hasPrefix(expandedPath)`）

**`path:` 修饰符详解**：
- `path:report` → 搜索完整路径（`path + "/" + name`）而非仅文件名
- `path:` 改变紧跟 term 的匹配范围为完整路径
- 与其他修饰符组合：`case:path:Report` = 大小写敏感 + 匹配完整路径

**验收标准**：
- [ ] `\` 路径限定：`Documents\ report` → AND(path_filter("Documents"), term("report"))
- [ ] `parent:` 限定：`parent:~/Documents report` → AND(parent_filter("~/Documents"), term("report"))
- [ ] `path:` 修饰符：`path:report` → 匹配完整路径中的 "report"
- [ ] `~` 展开：`parent:~/Documents` → `/Users/<user>/Documents`
- [ ] 路径部分 NFC 统一化
- [ ] 路径匹配大小写不敏感（macOS 文件系统默认不敏感）
- [ ] `parent:` 仅匹配 parentPath 前缀，不匹配文件名
- [ ] 路径限定与布尔运算组合：`<Documents\|Desktop>\ report` → (路径含 "Documents" OR 路径含 "Desktop") AND 文件名含 "report"
- [ ] 路径含空格时支持引号：`parent:"My Documents" report`
- [ ] 不存在的路径 → 0 结果（不报错，友好提示可选）

**测试覆盖**：
- [ ] `Documents\ report` → 路径含 "Documents" 且文件名含 "report" 的文件
- [ ] `parent:~/Documents report` → ~/Documents 下含 "report" 的文件
- [ ] `path:report` → 完整路径含 "report" 的文件（包含路径中含 "report" 的目录名）
- [ ] `parent:/nonexistent foo` → 0 结果
- [ ] `~` 展开正确
- [ ] 路径大小写不敏感：`documents\ report` 匹配 "Documents/" 路径
- [ ] 嵌套路径：`Documents\Projects\ report` → 路径含 "Documents" 和 "Projects"

### REQ-1.1-05 搜索修饰符 📋 🖥️ P0

**用户场景**：用户输入 `case:Report` → 仅看到文件名含大写 "Report" 的文件。输入 `file: report` → 仅看到文件（排除文件夹）。输入 `ext:pdf;doc report` → 仅在 PDF 和 DOC 文件中搜索 "report"。修饰符精细控制搜索行为。

**操作流程**：
1. QueryParser 检测修饰符前缀（`case:`, `file:`, `folder:`, `ext:`, `path:`）
2. 修饰符附加到紧随其后的 term 或改变全局搜索选项
3. 搜索时按修饰符调整匹配行为

**修饰符清单**：

| 修饰符 | 作用 | 示例 |
|--------|------|------|
| `case:` | 大小写敏感匹配 | `case:Report` 仅匹配含大写 "Report" 的文件 |
| `file:` | 仅匹配文件（排除目录） | `file: report` 仅文件 |
| `folder:` | 仅匹配目录（排除文件） | `folder: projects` 仅目录 |
| `ext:list` | 按扩展名过滤（分号分隔） | `ext:pdf;doc;xlsx` PDF/DOC/XLSX 文件 |
| `path:` | 匹配完整路径 | `path:report` 匹配路径中的 "report" |

**修饰符作用域规则**：
- 修饰符作用于紧随其后的单个 term：`case:Report draft` = 大小写敏感匹配 "Report" AND（默认大小写不敏感）匹配 "draft"
- 修饰符对短语有效：`case:"Annual Report"` = 大小写敏感匹配短语 "Annual Report"
- 修饰符可叠加：`case:file:Report` = 大小写敏感 + 仅文件 + 匹配 "Report"
- `file:` 和 `folder:` 是全局过滤器（二选一），出现在查询任意位置均生效
- `ext:` 是全局过滤器，出现在查询任意位置均生效
- `case:` 是局部修饰符，只修饰紧跟的 term
- `path:` 是局部修饰符，只修饰紧跟的 term

**`ext:` 语法**：
- `ext:pdf` → 单个扩展名
- `ext:pdf;doc;xlsx` → 多个扩展名（OR 语义）
- `ext:` 值不含前导 `.`：`ext:pdf` 而非 `ext:.pdf`
- 比较大小写不敏感：`ext:PDF` 匹配 ".pdf"
- 无扩展名的文件：`ext:`（空值）匹配无扩展名文件（文件名不含 `.`）
- 与布尔组合：`report ext:pdf;doc` → 含 "report" 且扩展名为 pdf 或 doc

**否定修饰符**：
- `!case:xxx` 无特殊语义（NOT + 大小写敏感），正常解析但效果等同 `!xxx`（大小写不敏感否定）
- `!ext:tmp` → 排除 .tmp 文件（等价于 `!*.tmp` 的语义）
- `!file:` → 仅文件夹（等价于 `folder:`）
- `!folder:` → 仅文件（等价于 `file:`）

**验收标准**：
- [ ] `case:Report` → 大小写敏感匹配 "Report"（不匹配 "report", "REPORT"）
- [ ] `file: report` → 仅返回文件（isDirectory == false）
- [ ] `folder: projects` → 仅返回目录（isDirectory == true）
- [ ] `ext:pdf` → 仅 PDF 文件
- [ ] `ext:pdf;doc;xlsx` → PDF 或 DOC 或 XLSX 文件
- [ ] `path:report` → 匹配完整路径中的 "report"
- [ ] `case:file:Report` → 修饰符叠加正确
- [ ] `case:` 局部作用域：`case:Report draft` → "Report" 大小写敏感 AND "draft" 大小写不敏感
- [ ] `file:` 全局作用域：出现在查询任意位置均生效
- [ ] `ext:` 全局作用域：出现在查询任意位置均生效
- [ ] `ext:` 大小写不敏感：`ext:PDF` 匹配 ".pdf"
- [ ] 无效修饰符（如 `foo:bar`）→ 当作普通 term 处理（不报错，向后兼容未来修饰符）
- [ ] 修饰符与通配符组合：`case:*.PDF` → 大小写敏感通配符
- [ ] 修饰符与正则组合：`case:regex:^Report` → 大小写敏感正则

**测试覆盖**：
- [ ] `case:Report` → 仅 ["Annual Report.docx"], NOT ["annual report.pdf", "REPORT.txt"]
- [ ] `file: report` → 仅文件结果
- [ ] `folder: projects` → 仅目录结果

### REQ-1.1-06 SearchQuery 增强模型 📋 🖥️ P0

**用户场景**：用户无直接感知。但结构化的查询模型确保 QueryParser 的输出能被 SearchCoordinator 正确执行，布尔运算、通配符、正则、修饰符都能高效求值。

**操作流程**：
1. QueryParser 输出 `QueryAST`（抽象语法树）
2. `QueryAST` 序列化为 `SearchQuery` 的新字段
3. IPC 传输到 daemon → SearchCoordinator 按 AST 执行搜索

**数据模型**：

```swift
/// 查询抽象语法树
indirect enum QueryAST: Sendable, Codable {
    case empty
    case term(String, TermOptions)           // 普通词
    case wildcard(String, TermOptions)       // 通配符 pattern
    case regex(String, TermOptions)          // 正则 pattern
    case and(QueryAST, QueryAST)             // AND
    case or(QueryAST, QueryAST)              // OR
    case not(QueryAST)                       // NOT
    case phrase(String, TermOptions)         // 精确短语
}

/// 词级选项
struct TermOptions: Sendable, Codable, OptionSet {
    let rawValue: UInt8
    static let caseSensitive = TermOptions(rawValue: 1 << 0)
    static let matchPath      = TermOptions(rawValue: 1 << 1)
    static let filesOnly      = TermOptions(rawValue: 1 << 2)  // 全局过滤器
    static let foldersOnly    = TermOptions(rawValue: 1 << 3)  // 全局过滤器
}

/// 全局查询选项（从 AST 提取）
struct QueryOptions: Sendable, Codable {
    var fileFilter: Bool = false             // file: 修饰
    var folderFilter: Bool = false           // folder: 修饰
    var extensions: Set<String>? = nil       // ext: 限定（nil = 不过滤）
    var parentPath: String? = nil            // parent: 限定
}

/// 增强的 SearchQuery（v1.1 扩展 v0.3 的 SearchQuery）
struct SearchQuery: Sendable {
    let id: String
    let rawText: String       // 原始查询文本（用于显示和历史记录）
    let ast: QueryAST         // 解析后的 AST
    let options: QueryOptions // 全局选项
    let limit: Int            // 默认 100
}
```

**AST 求值策略**：
- 递归求值：AND = 左结果 ∩ 右结果，OR = 左结果 ∪ 右结果，NOT = 全集 - 子树结果
- 优化：先求值限制性最强的子树（如 `ext:`, `parent:`），缩小后续求值范围
- 全局过滤器（`file:`, `folder:`, `ext:`）在 AST 求值前先应用，减少候选集
- `term` 类型：复用现有 FullSubstringMap / Trie / TrigramIndex 索引
- `wildcard` 类型：对候选集逐一匹配通配符 pattern
- `regex` 类型：对候选集逐一匹配正则 pattern
- `phrase` 类型：精确子串匹配（连续位置匹配）

**SearchQuery 兼容性**：
- IPC 协议扩展：`IPCQueryRequest` 新增 `ast` 和 `options` 字段（可选，v1.0 CLI 发送 `text` 字段仍兼容）
- v1.0 daemon（不支持 AST）收到 AST 查询 → 忽略 AST，回退到纯文本子串搜索
- v1.1 daemon 收到纯文本查询（无 AST）→ 用 QueryParser 在 daemon 端解析

**MatchType 扩展**：
```swift
enum MatchType: Sendable {
    case exact
    case prefix
    case substring
    case pinyin
    case wildcard    // v1.1: 通配符匹配
    case regex       // v1.1: 正则匹配
    case phrase      // v1.1: 精确短语匹配
    case semantic    // future
}
```

**验收标准**：
- [ ] `QueryAST` 支持 empty / term / wildcard / regex / and / or / not / phrase 八种节点
- [ ] `TermOptions` 支持 caseSensitive / matchPath / filesOnly / foldersOnly 四个选项
- [ ] `QueryOptions` 提取全局过滤器：file / folder / extensions / parentPath
- [ ] `SearchQuery` 包含 rawText + ast + options + limit
- [ ] `QueryAST` 和 `TermOptions` 实现 Codable（IPC 传输）
- [ ] AST 求值：AND = 交集，OR = 并集，NOT = 差集
- [ ] 全局过滤器在 AST 求值前应用
- [ ] `MatchType` 新增 wildcard / regex / phrase 三种类型
- [ ] IPC 后向兼容：v1.0 daemon 收到 v1.1 查询不崩溃（忽略 AST 字段）
- [ ] `SearchQuery` 可从纯文本构造（内部调用 QueryParser）
- [ ] 性能：AST 构造 + 全局选项提取 < 0.1ms

**测试覆盖**：
- [ ] QueryAST Codable 往返零丢失
- [ ] TermOptions OptionSet 位运算正确
- [ ] QueryOptions 从 AST 正确提取全局过滤器
- [ ] AST 求值：AND 交集
- [ ] AST 求值：OR 并集
- [ ] AST 求值：NOT 差集
- [ ] AST 求值：嵌套组合 `<a|b> !c d`
- [ ] 全局过滤器优先应用：`file: ext:pdf a b` → 先过滤文件+PDF，再匹配 a AND b
- [ ] 空 AST 求值返回空结果

### REQ-1.1-07 REPL 搜索历史持久化 📋 🖥️ P1

**用户场景**：用户在 REPL 中按上箭头 → 浏览之前的搜索查询（包括高级语法查询）。Ctrl+R 搜索历史中包含 `*.pdf report` 等复杂查询。退出 REPL 后重新进入，历史记录仍然可用。搜索历史是 v1.0 REPL 历史功能的增强。

**操作流程**：
1. 用户在 REPL 中输入查询 → 查询写入历史
2. 用户按上箭头 → 浏览历史查询
3. Ctrl+R → 增量搜索历史
4. 退出 REPL → 历史持久化到磁盘
5. 下次进入 REPL → 加载历史，可继续浏览

**v1.0 基线**（REQ-0.6-03）：
- readline 历史持久化到 `~/.deep-finder/history`
- 上下箭头浏览（readline 内置）
- Ctrl+R 增量搜索（readline 内置）
- 历史去重：连续重复查询不记录
- 最近 1000 条

**v1.1 增强**：
- 历史条目包含时间戳（方便未来按日期过滤）
- 历史条目记录结果数（方便用户回忆查询效果）
- 历史文件格式兼容 readline（纯文本，每行一条查询）
- 新增 `:history` 命令：显示最近 N 条历史（带序号和时间戳）
- 新增 `:history clear` 命令：清空历史

**历史文件格式**：
```
# DeepFinder search history
# Format: <unix_timestamp> <result_count> <query_text>
1748577600 15 report ext:pdf
1748577700 3 *.xlsx 2026
1748577800 0 regex:^report_\d{4}
```

- `#` 开头的行为注释（readline 忽略）
- 非注释行：时间戳 + 结果数 + 查询文本
- readline 读取时跳过时间戳前缀（仅取查询文本部分）
- 历史写入使用自定义函数（非 `write_history`），保留元数据

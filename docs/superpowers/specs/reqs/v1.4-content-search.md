# v1.4 — 内容搜索

4 项 REQ (P0: 3, P1: 1)

实现文件内容搜索（`content:` 函数）。在文件名搜索基础上，搜索文件内部的文本内容，支持多编码、文件类型限定和行号定位。对标 Everything 的 `content:` 函数。

**前置依赖**：v1.1 高级搜索语法（QueryParser）、v1.2 元数据过滤（FilterExpression）、v0.3 SearchProvider 协议。

**性能约束**：内容搜索涉及文件 I/O，比文件名搜索慢几个数量级。设计上必须：流式读取（不一次性加载大文件）、二进制文件跳过、文件类型限定缩小范围。

---

### REQ-1.4-01 Content Search Provider — content: 函数 📋 🖥️ P0

**用户场景**：用户在项目中寻找含 "TODO" 的源代码文件——输入 `ext:swift;py;md content:TODO`。在邮件归档中找含 "banana" 的邮件——输入 `*.eml dm:thisweek content:banana`。内容搜索打开文件逐行扫描，找到匹配的行并返回。

**操作流程**：
1. 用户输入 `content:banana` 或 `report content:banana` → QueryParser 检测 `content:` 前缀
2. 提取 content 搜索词和可选的文件名搜索词
3. 如有文件名搜索词，先通过索引获取候选文件列表
4. 对候选文件逐一打开，流式读取，搜索匹配行
5. 返回含匹配行号和上下文的结果

**语法**（对标 Everything `content:` 函数）：

```
content:<text>                搜索文件内容包含 <text> 的文件
<keyword> content:<text>      文件名含 <keyword> 且内容含 <text>
ext:<ext> content:<text>      指定扩展名文件中搜索内容
case:content:<text>           大小写敏感内容搜索
```

**与现有语法组合**：
- `content:` 是全局限定符：缩小搜索范围到内容匹配的文件
- 与文件名搜索 AND 组合：`report content:banana` = 文件名含 "report" AND 内容含 "banana"
- 与过滤器组合：`ext:swift;py content:TODO` = Swift/Python 文件中含 "TODO"
- 与日期过滤组合：`dm:thisweek content:meeting` = 本周修改的文件中含 "meeting"
- `case:content:Banana` = 大小写敏感内容搜索（不匹配 "banana"）
- 同一查询中 `content:` 只能出现一次，重复出现报错

**ContentSearchProvider**：
- 实现 `SearchProvider` 协议（v0.3 定义的 AsyncSequence 接口）
- 输入：候选文件列表（来自索引的文件名匹配结果，或全量文件列表）
- 输出：`ContentSearchResult`（扩展 `SearchResult`，增加匹配行信息）

**流式读取策略**：
- 文件按 64KB 块读取（`FileHandle.read(upToCount:)`）
- 逐行扫描：维护行缓冲区，检测到换行符后对完整行执行子串匹配
- 跨块行处理：行缓冲区拼接未完成的行直到遇到换行符
- 文件大小上限：单文件读取不超过 64MB（超过跳过，标记为 "skipped_large"）
- 总 I/O 上限：单次查询总读取量不超过 512MB（超过停止扫描，返回已找到的结果 + 截断警告）

**二进制文件检测**：
- 读取前 8KB 检测是否为二进制文件
- 检测规则：如果前 8KB 中 NUL 字节（`\0`）出现 > 1 个，视为二进制文件跳过
- 二进制文件不报错，静默跳过

**并发**：
- 使用 `TaskGroup` 并行扫描多个文件
- 并发度限制：最多 8 个并发文件读取（避免磁盘 I/O 饱和）
- 搜索取消（`cancel(queryID:)`）时停止所有正在进行的文件读取

**验收标准**：
- [ ] `content:<text>` 搜索文件内容
- [ ] 与文件名搜索 AND 组合
- [ ] 与过滤器（ext:, dm:, size:）组合
- [ ] `case:content:` 大小写敏感内容搜索
- [ ] `content:` 重复出现报错
- [ ] 流式读取：按 64KB 块读取，不一次性加载整个文件
- [ ] 跨块行正确拼接
- [ ] 文件大小上限 64MB（超过跳过）
- [ ] 总 I/O 上限 512MB（超过截断 + 警告）
- [ ] 二进制文件检测（NUL 字节检测）并跳过
- [ ] TaskGroup 并行扫描，最多 8 并发
- [ ] 搜索取消时停止文件读取
- [ ] 性能：1000 个小文件（<100KB）内容搜索 < 5s

**测试覆盖**：
- [ ] 简单内容匹配：文件含 "banana" 的被返回
- [ ] 文件名 + 内容 AND：`report content:TODO` 正确组合
- [ ] ext 限定：`ext:swift content:TODO` 仅扫描 Swift 文件
- [ ] 大小写敏感：`case:content:Banana` 不匹配 "banana"
- [ ] 二进制文件跳过
- [ ] 大文件跳过（>64MB）
- [ ] 跨块行拼接：长行（>64KB）正确匹配
- [ ] 搜索取消停止读取
- [ ] 空 content 查询：`content:` 报错

### REQ-1.4-02 Encoding Support — 多编码支持 📋 🖥️ P0

**用户场景**：用户在国际化环境中工作，文件可能是 UTF-8、UTF-16 LE（Windows 导出）或 UTF-16 BE 编码。输入 `content:报告` 应能在各种编码的文件中找到包含 "报告" 的文件。

**操作流程**：
1. ContentSearchProvider 打开文件 → 读取前几字节检测编码
2. 按检测到的编码解码文件内容
3. 在解码后的 Unicode 文本中搜索目标字符串
4. 返回匹配结果

**支持的编码**：

| 编码 | BOM | 检测方式 |
|------|-----|----------|
| UTF-8 | `EF BB BF`（可选） | 默认：无 BOM 且无其他编码特征时假定为 UTF-8 |
| UTF-16 LE | `FF FE` | BOM 检测 |
| UTF-16 BE | `FE FF` | BOM 检测 |
| UTF-16 LE (无 BOM) | 无 | 统计检测：前 8KB 中 `\0` 交替出现模式 |
| UTF-16 BE (无 BOM) | 无 | 统计检测：前 8KB 中 `\0` 交替出现模式 |

**编码检测逻辑**：
1. 读取前 4 字节
2. 检查 BOM：`FF FE` → UTF-16 LE，`FE FF` → UTF-16 BE，`EF BB BF` → UTF-8（有 BOM）
3. 无 BOM 时：检查前 8KB 是否符合 UTF-16 LE/BE 的 `\0` 交替模式
4. 均不匹配时：假定为 UTF-8（最常见情况）

**解码处理**：
- UTF-8：直接使用 `String(data:, encoding: .utf8)`
- UTF-16：使用 `String(data:, encoding: .utf16LittleEndian)` 或 `.utf16BigEndian`
- BOM 字节不包含在解码内容中（跳过前 2-3 字节）
- 解码失败（非法字节序列）的文件跳过，不报错
- 解码后的文本统一为 Swift `String`（Unicode scalar），后续搜索在统一文本上执行

**验收标准**：
- [ ] 支持 UTF-8 编码（有 BOM 和无 BOM）
- [ ] 支持 UTF-16 LE 编码（有 BOM 和无 BOM）
- [ ] 支持 UTF-16 BE 编码（有 BOM 和无 BOM）
- [ ] BOM 检测优先
- [ ] 无 BOM 时 UTF-16 统计检测
- [ ] 均不匹配时默认 UTF-8
- [ ] BOM 字节不包含在搜索内容中
- [ ] 解码失败的文件跳过（不报错）
- [ ] 搜索前 NFC 统一化（与其他搜索行为一致）
- [ ] 查询词 NFC 统一化后与内容比较

**测试覆盖**：
- [ ] UTF-8 文件（无 BOM）内容搜索
- [ ] UTF-8 文件（有 BOM）内容搜索
- [ ] UTF-16 LE 文件（有 BOM）内容搜索
- [ ] UTF-16 BE 文件（有 BOM）内容搜索
- [ ] UTF-16 LE 文件（无 BOM）内容搜索
- [ ] UTF-16 BE 文件（无 BOM）内容搜索
- [ ] 非 Unicode 编码文件（如 Latin-1）→ 假定为 UTF-8，解码失败跳过
- [ ] 中文字符搜索：UTF-8 和 UTF-16 文件中搜 "报告" 均匹配
- [ ] BOM 不干扰搜索

### REQ-1.4-03 Line Number Display — 匹配行号与上下文 📋 🖥️ P0

**用户场景**：用户搜索 `content:TODO` → 结果中不仅显示匹配的文件名，还显示匹配所在行号和该行内容。类似 `grep -n` 的输出：`report.swift:42: // TODO: fix this later`。用户可以快速定位到文件中的具体位置。

**操作流程**：
1. ContentSearchProvider 扫描文件 → 记录每个匹配的行号和行内容
2. 结果包含 `ContentMatch` 数组（行号 + 行内容 + 列偏移）
3. TerminalFormatter 输出时显示行号和匹配行
4. `:open N` 命令打开文件时可选跳转到第一个匹配行

**数据模型**：

```swift
/// 单个内容匹配
struct ContentMatch: Sendable, Codable {
    let lineNumber: Int       // 1-based 行号
    let columnOffset: Int     // 0-based 列偏移（匹配开始位置）
    let lineContent: String   // 匹配行的完整内容（去除首尾空白）
    let matchedRange: Range<String.Index>  // lineContent 中匹配的范围
}

/// 内容搜索结果（扩展 SearchResult）
struct ContentSearchResult: Sendable {
    let record: FileRecord
    let matches: [ContentMatch]       // 该文件中所有匹配（最多显示前 10 条）
    let totalMatchCount: Int          // 该文件中总匹配数
    let truncated: Bool               // totalMatchCount > 10 时为 true
}
```

**终端输出格式**：

```
report.swift (3 matches)
  42 │ // TODO: fix this later
  87 │ // TODO: optimize this
 156 │ // TODO: add error handling

notes.md (1 match)
  15 │ - TODO: review quarterly report
```

- 行号右对齐，`│` 分隔符
- 匹配部分 ANSI 高亮（与文件名匹配高亮一致的颜色）
- 每个文件最多显示前 10 行匹配，超出显示 `... (N more matches)`
- `--json` 模式：完整输出 ContentMatch 数组
- `--0` 模式：不输出内容匹配（仅路径，与 v0.5 行为一致）

**行内容截断**：
- 行内容超过 200 字符时，截断到 200 字符并追加 `...`
- 截断时保留匹配区域（确保匹配部分可见）
- Tab 字符替换为 4 个空格

**REPL 集成**：
- `:open N` 打开文件时使用 `--line <lineNumber>` 参数（如果应用支持）
- `:reveal N` 行为不变（在 Finder 中显示文件）

**验收标准**：
- [ ] ContentMatch 包含 lineNumber (1-based)、columnOffset (0-based)、lineContent、matchedRange
- [ ] ContentSearchResult 包含 matches 数组、totalMatchCount、truncated 标志
- [ ] 终端输出显示行号 + `│` + 匹配行内容
- [ ] 匹配部分 ANSI 高亮
- [ ] 每文件最多显示 10 行匹配，超出显示 `... (N more matches)`
- [ ] 长行截断到 200 字符，保留匹配区域可见
- [ ] Tab 替换为 4 空格
- [ ] `--json` 模式输出完整 ContentMatch 数组
- [ ] `--0` 模式不输出内容匹配（仅路径）
- [ ] 行号从 1 开始计数
- [ ] 空行（仅空白字符）显示为空行，不跳过

**测试覆盖**：
- [ ] 单行匹配：行号和内容正确
- [ ] 多行匹配：所有匹配行号和内容正确
- [ ] 超过 10 行匹配：truncated = true，显示前 10 条
- [ ] 长行截断：匹配区域保留在截断后内容中
- [ ] Tab 替换
- [ ] `--json` 输出格式正确
- [ ] `--0` 不输出内容
- [ ] 空文件：0 匹配，不报错
- [ ] 行号从 1 开始

### REQ-1.4-04 Content Search Performance & Limits 📋 🖥️ P1

**用户场景**：用户无直接感知。内容搜索的性能约束和资源限制确保搜索不会卡死系统或消耗过多资源。

**性能基准**：

| 场景 | 目标 | 说明 |
|------|------|------|
| 100 个小文件（<10KB）| < 1s | 典型代码搜索 |
| 1000 个小文件（<100KB）| < 5s | 大范围代码搜索 |
| 10 个中等文件（1-10MB）| < 3s | 日志文件搜索 |
| 1 个大文件（60MB）| < 10s | 接近上限的大文件 |

**资源限制**：

| 限制 | 值 | 行为 |
|------|-----|------|
| 单文件读取上限 | 64MB | 超过跳过，结果标记 skipped_large |
| 单次查询总 I/O 上限 | 512MB | 超过截断，返回已找到结果 + 警告 |
| 每文件最大匹配数 | 1000 | 超过停止该文件扫描 |
| 并发文件读取 | 8 | TaskGroup maxConcurrentTasks |
| 单文件读取超时 | 5s | 超时跳过该文件 |
| 候选文件数上限 | 10,000 | 超过拒绝执行，提示缩小范围 |

**结果截断处理**：
- 文件被跳过（过大、二进制、解码失败）→ 计入 `skippedCount`，最终输出摘要
- 查询被截断（总 I/O 超限）→ 输出警告：`"内容搜索已截断：已扫描 X/Y 文件。建议使用 ext: 或文件名搜索缩小范围。"`
- 单文件匹配过多 → 返回前 10 条匹配（ContentSearchResult.truncated = true）

**进度反馈**（REPL 模式）：
- 内容搜索耗时较长时显示进度指示器
- 格式：`扫描中... X/Y 文件 (Z 匹配)`（每 100ms 更新）
- 进度使用 ANSI `\r` 回车覆盖同一行（不产生多行输出）

**验收标准**：
- [ ] 单文件 >64MB 跳过，结果标记 skipped_large
- [ ] 总 I/O >512MB 截断，返回已找到结果 + 警告
- [ ] 每文件匹配 >1000 条停止扫描
- [ ] 并发限制 8 个文件
- [ ] 候选文件 >10,000 拒绝执行，提示缩小范围
- [ ] REPL 进度指示器
- [ ] 搜索取消立即停止所有文件读取
- [ ] 资源限制可通过 ConfigStore 配置（未来扩展，v1.4 硬编码默认值）

**测试覆盖**：
- [ ] 大文件（>64MB）跳过
- [ ] 总 I/O 截断
- [ ] 候选文件数上限拒绝
- [ ] 搜索取消停止读取
- [ ] 性能基准：100 个小文件 < 1s
- [ ] 进度指示器输出格式

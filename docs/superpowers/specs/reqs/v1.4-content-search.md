# v1.4 — 内容搜索

4 项 REQ（P0: 3，P1: 1）

> 卡片格式（OSS · BDD）：用户故事 + 验收标准（Given/When/Then）+ 实现/测试溯源。金标准见 v0.1-index-core.md。

实现文件内容搜索（`content:` 函数）。在文件名搜索基础上，搜索文件内部的文本内容，支持多编码、文件类型限定和行号定位。对标 Everything 的 `content:` 函数。

**前置依赖**：v1.1 高级搜索语法（QueryParser）、v1.2 元数据过滤（FilterExpression）、v0.3 SearchProvider 协议。

**性能约束**：内容搜索涉及文件 I/O，比文件名搜索慢几个数量级。设计上必须：流式读取（不一次性加载大文件）、二进制文件跳过、文件类型限定缩小范围。

### REQ-1.4-01 Content Search Provider — content: 函数 ✅ 🖥️ P0

**用户故事**

作为「开发者/邮件用户」，我希望用 `content:` 前缀搜索文件内部的文本内容（如 `ext:swift;py;md content:TODO`、`*.eml dm:thisweek content:banana`），以便定位含特定关键词的源代码、邮件或文档文件。

**验收标准（Given / When / Then）**

- **AC1 — 基础内容搜索**
  - **Given** 用户输入 `content:<text>` 或 `report content:banana`
  - **When** QueryParser 检测到 `content:` 前缀，提取内容搜索词和可选的文件名搜索词
  - **Then** 对候选文件逐一打开、流式读取、扫描匹配行，返回含匹配行号与上下文的结果

- **AC2 — 与文件名搜索 AND 组合**
  - **Given** 查询 `report content:banana`
  - **When** 执行搜索
  - **Then** 结果为「文件名含 report」AND「内容含 banana」的文件

- **AC3 — 与过滤器组合**
  - **Given** 查询 `ext:swift;py content:TODO` 或 `dm:thisweek content:meeting`
  - **When** 执行搜索
  - **Then** 内容搜索范围被 ext / dm / size 过滤器缩小

- **AC4 — 大小写敏感**
  - **Given** 查询 `case:content:Banana`
  - **When** 执行大小写敏感内容搜索
  - **Then** 不匹配小写「banana」

- **AC5 — content: 重复报错**
  - **Given** 同一查询中 `content:` 出现多次
  - **When** 解析查询
  - **Then** 报错（`content:` 只能出现一次）

- **AC6 — 流式读取**
  - **Given** 任意待扫描文件
  - **When** 读取文件内容
  - **Then** 按 64KB 块读取（`FileHandle.read(upToCount:)`），不一次性加载整个文件；跨块的行通过行缓冲区正确拼接

- **AC7 — 文件大小上限**
  - **Given** 单文件 > 64MB
  - **When** 扫描
  - **Then** 跳过该文件，标记为 `skipped_large`

- **AC8 — 总 I/O 上限**
  - **Given** 单次查询总读取量 > 512MB
  - **When** 超限
  - **Then** 停止扫描，返回已找到结果并附截断警告

- **AC9 — 二进制文件检测**
  - **Given** 文件前 8KB 中 NUL 字节（`\0`）出现 > 1 个
  - **When** 判定文件类型
  - **Then** 视为二进制文件静默跳过，不报错

- **AC10 — 并发扫描**
  - **Given** 多个候选文件
  - **When** 使用 `TaskGroup` 并行扫描
  - **Then** 最多 8 个并发文件读取

- **AC11 — 搜索取消**
  - **Given** 正在进行的文件读取
  - **When** 调用 `cancel(queryID:)`
  - **Then** 立即停止所有文件读取

- **AC12 — 性能**
  - **Given** 1000 个小文件（< 100KB）
  - **When** 执行内容搜索
  - **Then** 耗时 < 5s

**实现**：`Sources/Search/ContentSearchProvider.swift`、`Sources/Search/ContentScanner.swift` · **测试**：`Tests/SearchTests/ContentSearchTests.swift`

### REQ-1.4-02 Encoding Support — 多编码支持 ✅ 🖥️ P0

**用户故事**

作为「国际化环境用户」，我希望在不同编码（UTF-8、UTF-16 LE/BE，含或不含 BOM）的文件中搜索同一个关键词（如 `content:报告`），以便无需关心文件来源编码即可定位内容。

**验收标准（Given / When / Then）**

- **AC1 — UTF-8 支持**
  - **Given** UTF-8 编码的文件（有 BOM 或无 BOM）
  - **When** 执行内容搜索
  - **Then** 正确解码并返回匹配

- **AC2 — UTF-16 LE/BE 支持**
  - **Given** UTF-16 LE 或 UTF-16 BE 编码的文件（有 BOM 或无 BOM）
  - **When** 执行内容搜索
  - **Then** 正确解码并返回匹配

- **AC3 — BOM 检测优先**
  - **Given** 文件前 4 字节
  - **When** 检测编码
  - **Then** BOM 优先：`FF FE` → UTF-16 LE，`FE FF` → UTF-16 BE，`EF BB BF` → UTF-8（有 BOM）

- **AC4 — 无 BOM 统计检测**
  - **Given** 无 BOM 的文件
  - **When** 检查前 8KB
  - **Then** 符合 UTF-16 LE/BE 的 `\0` 交替模式时识别为对应 UTF-16；否则假定为 UTF-8

- **AC5 — BOM 字节不入搜索内容**
  - **Given** 含 BOM 的文件
  - **When** 解码
  - **Then** BOM 字节（前 2-3 字节）不包含在解码后的搜索内容中

- **AC6 — 解码失败跳过**
  - **Given** 非 Unicode 编码（如 Latin-1）文件，假定为 UTF-8 后解码失败
  - **When** 解码
  - **Then** 跳过该文件，不报错

- **AC7 — NFC 统一化**
  - **Given** 解码后的文本和查询词
  - **When** 执行搜索
  - **Then** 搜索前对内容 NFC 统一化，查询词同样 NFC 统一化后比较（与其他搜索行为一致）

**实现**：`Sources/Search/ContentScanner.swift` · **测试**：`Tests/SearchTests/ContentSearchTests.swift`

### REQ-1.4-03 Line Number Display — 匹配行号与上下文 ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望内容搜索结果像 `grep -n` 一样显示匹配文件名 + 行号 + 该行内容（如 `report.swift:42: // TODO: fix this later`），以便快速定位到文件中的具体位置。

**验收标准（Given / When / Then）**

- **AC1 — 数据模型**
  - **Given** `ContentMatch`
  - **When** 读取字段
  - **Then** 包含 `lineNumber`（1-based）、`columnOffset`（0-based）、`lineContent`（去首尾空白）、`matchedRange`

- **AC2 — ContentSearchResult**
  - **Given** `ContentSearchResult`
  - **When** 读取字段
  - **Then** 包含 `matches` 数组、`totalMatchCount`、`truncated` 标志（totalMatchCount > 10 时为 true）

- **AC3 — 终端输出格式**
  - **Given** 内容搜索结果
  - **When** TerminalFormatter 输出
  - **Then** 显示「文件名 (N matches)」+ 行号（右对齐）+ `│` 分隔符 + 匹配行内容

- **AC4 — 匹配高亮**
  - **Given** 输出中的匹配部分
  - **When** 渲染
  - **Then** 匹配部分 ANSI 高亮（与文件名匹配高亮一致的颜色）

- **AC5 — 行数限制**
  - **Given** 单文件匹配行数
  - **When** 输出
  - **Then** 每文件最多显示前 10 行匹配，超出显示 `... (N more matches)`

- **AC6 — 长行截断**
  - **Given** 行内容超过 200 字符
  - **When** 输出
  - **Then** 截断到 200 字符并追加 `...`，保留匹配区域可见；Tab 字符替换为 4 个空格

- **AC7 — --json 模式**
  - **Given** `--json` 模式
  - **When** 输出结果
  - **Then** 完整输出 `ContentMatch` 数组

- **AC8 — --0 模式**
  - **Given** `--0` 模式
  - **When** 输出结果
  - **Then** 不输出内容匹配，仅输出路径（与 v0.5 行为一致）

- **AC9 — 行号起始**
  - **Given** 文件内容
  - **When** 计数行号
  - **Then** 行号从 1 开始；空行（仅空白字符）显示为空行，不跳过

**实现**：`Sources/Search/ContentSearchProvider.swift`、`Sources/CLI/TerminalFormatter.swift` · **测试**：`Tests/SearchTests/ContentSearchTests.swift`

### REQ-1.4-04 Content Search Performance & Limits ✅ 🖥️ P1

**用户故事**

作为「系统」，我希望内容搜索具备明确的性能基准与资源限制，以便搜索不会卡死系统或消耗过多资源（用户无直接感知）。

**验收标准（Given / When / Then）**

- **AC1 — 性能基准**
  - **Given** 不同规模的内容搜索任务
  - **When** 执行
  - **Then** 100 个小文件（< 10KB）< 1s；1000 个小文件（< 100KB）< 5s；10 个中等文件（1-10MB）< 3s；1 个大文件（60MB）< 10s

- **AC2 — 单文件大小上限**
  - **Given** 单文件 > 64MB
  - **When** 扫描
  - **Then** 跳过，结果标记 `skipped_large`

- **AC3 — 总 I/O 截断**
  - **Given** 单次查询总 I/O > 512MB
  - **When** 超限
  - **Then** 截断扫描，返回已找到结果 + 警告「内容搜索已截断：已扫描 X/Y 文件。建议使用 ext: 或文件名搜索缩小范围。」

- **AC4 — 每文件匹配上限**
  - **Given** 单文件匹配数 > 1000
  - **When** 超限
  - **Then** 停止该文件扫描

- **AC5 — 并发限制**
  - **Given** TaskGroup 并发文件读取
  - **When** 设置并发度
  - **Then** 最多 8 个并发（maxConcurrentTasks = 8）

- **AC6 — 候选文件数上限**
  - **Given** 候选文件数 > 10,000
  - **When** 准备扫描
  - **Then** 拒绝执行，提示用户缩小范围

- **AC7 — REPL 进度指示器**
  - **Given** 内容搜索耗时较长（REPL 模式）
  - **When** 扫描中
  - **Then** 显示进度指示器 `扫描中... X/Y 文件 (Z 匹配)`（每 100ms 更新，ANSI `\r` 回车覆盖同一行，不产生多行输出）

- **AC8 — 搜索取消**
  - **Given** 正在进行的文件读取
  - **When** 搜索被取消
  - **Then** 立即停止所有文件读取

- **AC9 — 资源限制配置**
  - **Given** v1.4 版本
  - **When** 查看资源限制来源
  - **Then** 使用硬编码默认值（未来可通过 ConfigStore 配置）

**实现**：`Sources/Search/ContentScanner.swift` · **测试**：`Tests/SearchTests/ContentSearchTests.swift`

# v1.4 — 内容搜索

4 项 REQ（P0: 3，P1: 1）

> 卡片格式（OSS · BDD）：用户故事 + 验收标准（Given/When/Then）+ 实现/测试溯源。金标准见 v0.1-index-core.md。

实现文件内容搜索（`content:` 函数）。在文件名搜索基础上，搜索文件内部的文本内容，支持多编码、文件类型限定和行号定位。对标 Everything 的 `content:` 函数。

**前置依赖**：v1.1 高级搜索语法（QueryParser）、v1.2 元数据过滤（FilterExpression）、v0.3 SearchProvider 协议。

**性能约束**：内容搜索涉及文件 I/O，比文件名搜索慢几个数量级。设计上必须：流式读取（不一次性加载大文件）、二进制文件跳过、文件类型限定缩小范围。

### REQ-1.4-01 Content Search Provider — content: 函数 ✅ 🖥️ P0

**用户故事**

作为「开发者/邮件用户」，我希望用 `content:` 前缀搜索文件内部的文本内容（如 `ext:swift;py;md content:TODO`、`*.eml dm:thisweek content:banana`），以便定位含特定关键词的源代码、邮件或文档文件。

**验收标准（Given / When / Then）**

- **AC1 — 基础内容搜索**
  - **Given** `ContentSearchProvider` 被注册到 SearchCoordinator（当前实现中 provider 存在但 daemon 仅注册 FileIndexProvider，`content:` 语法解析尚未接入 QueryParser）
  - **When** provider 收到查询词（经 SearchQuery NFC 统一化）
  - **Then** 对候选文件逐一打开、读取、扫描匹配行，返回含匹配行号与行内容的结果（`ContentMatch`）

- **AC2 — 与文件名搜索 AND 组合**
  - **Given** 查询 `report content:banana`
  - **When** 执行搜索
  - **Then** 结果为「文件名含 report」AND「内容含 banana」的文件

- **AC3 — 与过滤器组合**
  - **Given** 查询 `ext:swift;py content:TODO` 或 `dm:thisweek content:meeting`
  - **When** 执行搜索
  - **Then** 内容搜索范围被 ext / dm / size 过滤器缩小

- **AC4 — 大小写敏感**
  - **Given** 查询 `case:content:Banana`
  - **When** 执行大小写敏感内容搜索
  - **Then** 不匹配小写「banana」

- **AC5 — content: 重复报错**
  - **Given** 同一查询中 `content:` 出现多次
  - **When** 解析查询
  - **Then** 报错（`content:` 只能出现一次）

- **AC6 — 文件读取**
  - **Given** 任意待扫描文件
  - **When** 读取文件内容
  - **Then** 通过 `Data(contentsOf:)` 一次性读入内存（受单文件大小上限保护），解码后逐行扫描匹配

- **AC7 — 文件大小上限**
  - **Given** 单文件 > 50MB（`Constants.ContentScanner.maxFileSize = 50_000_000`）
  - **When** 扫描
  - **Then** 静默跳过该文件，返回空匹配

- **AC8 — 总 I/O 上限**
  - **Given** 单次查询累计读取量达到 512MB（`Constants.ContentScanner.maxTotalIO`）
  - **When** 超限
  - **Then** 停止扫描，返回已找到结果（当前实现未附显式截断警告）

- **AC9 — 二进制文件检测**
  - **Given** 文件前 8KB（`Constants.Scan.binaryProbeSize`）中存在任意 NUL 字节
  - **When** 判定文件类型
  - **Then** 视为二进制文件静默跳过（UTF-16 BOM 文件除外，正常解码）；不报错

- **AC10 — 扫描执行**
  - **Given** 多个候选文件
  - **When** 执行扫描
  - **Then** 当前实现按候选顺序串行扫描（`Constants.ContentScanner.maxConcurrentScans = 8` 已定义但未在 provider 中应用 TaskGroup）

- **AC11 — 搜索取消**
  - **Given** 正在进行的扫描
  - **When** 调用 `cancel(queryID:)`
  - **Then** 当前实现为 no-op（每文件扫描同步完成，无可取消的 in-flight 读取）

- **AC12 — 性能**
  - **Given** 1000 个小文件（< 100KB）
  - **When** 执行内容搜索
  - **Then** 耗时 < 5s

**实现**：`Sources/Search/ContentSearchProvider.swift`、`Sources/Search/ContentScanner.swift` · **测试**：`Tests/SearchTests/ContentSearchTests.swift`

### REQ-1.4-02 Encoding Support — 多编码支持 ✅ 🖥️ P0

**用户故事**

作为「国际化环境用户」，我希望在不同编码（UTF-8、UTF-16 LE/BE，含或不含 BOM）的文件中搜索同一个关键词（如 `content:报告`），以便无需关心文件来源编码即可定位内容。

**验收标准（Given / When / Then）**

- **AC1 — UTF-8 支持**
  - **Given** UTF-8 编码的文件（有 BOM 或无 BOM）
  - **When** 执行内容搜索
  - **Then** 正确解码并返回匹配

- **AC2 — UTF-16 LE/BE 支持**
  - **Given** UTF-16 LE 或 UTF-16 BE 编码的文件（有 BOM 或无 BOM）
  - **When** 执行内容搜索
  - **Then** 正确解码并返回匹配

- **AC3 — BOM 检测优先**
  - **Given** 文件前 4 字节
  - **When** 检测编码
  - **Then** BOM 优先：`FF FE` → UTF-16 LE，`FE FF` → UTF-16 BE，`EF BB BF` → UTF-8（有 BOM）

- **AC4 — 无 BOM 回退**
  - **Given** 无 BOM 的文件
  - **When** 检测编码
  - **Then** 直接尝试 UTF-8 解码（当前实现不做无 BOM 的 UTF-16 统计检测）；UTF-8 解码失败则跳过文件

- **AC5 — BOM 字节不入搜索内容**
  - **Given** 含 BOM 的文件
  - **When** 解码
  - **Then** BOM 字节（前 2-3 字节）不包含在解码后的搜索内容中

- **AC6 — 解码失败跳过**
  - **Given** 非 Unicode 编码（如 Latin-1）文件，假定为 UTF-8 后解码失败
  - **When** 解码
  - **Then** 跳过该文件，不报错

- **AC7 — 查询 NFC 统一化**
  - **Given** 查询词和文件内容
  - **When** 执行搜索
  - **Then** 查询词经 `SearchQuery` NFC 统一化（`precomposedStringWithCanonicalMapping`）后传入；文件内容当前不做额外 NFC 统一化（依赖 UTF 解码结果直接比较）

**实现**：`Sources/Search/ContentScanner.swift` · **测试**：`Tests/SearchTests/ContentSearchTests.swift`

### REQ-1.4-03 Line Number Display — 匹配行号与上下文 ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望内容搜索结果像 `grep -n` 一样显示匹配文件名 + 行号 + 该行内容（如 `report.swift:42: // TODO: fix this later`），以便快速定位到文件中的具体位置。

**验收标准（Given / When / Then）**

- **AC1 — 数据模型**
  - **Given** `ContentMatch`
  - **When** 读取字段
  - **Then** 包含 `filePath`（绝对路径）、`lineNumber`（1-based）、`lineContent`（整行原文，不去空白）、`matchRange`（`Range<String.Index>`，行内字符范围）

- **AC2 — 匹配存储**
  - **Given** 一次搜索完成
  - **When** 读取某文件的匹配详情
  - **Then** 通过 `ContentSearchProvider.contentMatches(for: recordID)` 返回该文件的 `[ContentMatch]`（无独立的 `ContentSearchResult` 类型、无 `totalMatchCount`/`truncated` 字段）

- **AC3 — 终端输出格式**
  - **Given** 内容搜索结果（与其他 provider 共用同一 `SearchResult`）
  - **When** TerminalFormatter 输出
  - **Then** 当前实现按通用 SearchResult 格式输出（文件名 + 路径 + 大小 + 日期），行级匹配详情需通过 `contentMatches(for:)` API 单独获取，TerminalFormatter 未内置内容专用的 `文件名 (N matches)` + 行号 + 行内容渲染

- **AC4 — 匹配高亮**
  - **Given** 输出中的文件名匹配部分
  - **When** 渲染
  - **Then** TerminalFormatter 对文件名做查询高亮（ANSI bold + 黄色）；行内内容匹配的高亮当前未在 formatter 中实现

- **AC5 — 行数限制**
  - **Given** 单文件匹配行数
  - **When** 输出
  - **Then** 当前实现未对显示行数做截断（无 `... (N more matches)` 占位）

- **AC6 — 长行处理**
  - **Given** 行内容长度
  - **When** 扫描
  - **Then** 超过 `ScanOptions.maxLineLength`（默认 10 000 字符）的行直接跳过扫描（避免压缩/单行大文件）；当前实现不做 200 字符截断、不做 Tab→4 空格替换

- **AC7 — --json 模式**
  - **Given** `--json` 模式
  - **When** 输出结果
  - **Then** 输出 `[SearchResult]` JSON 数组（`SearchResult` 不含行级 `ContentMatch`，匹配详情需经 IPC/provider API 单独查询）

- **AC8 — --0 模式**
  - **Given** `--0` 模式
  - **When** 输出结果
  - **Then** 不输出内容匹配，仅输出路径（与 v0.5 行为一致）

- **AC9 — 行号起始**
  - **Given** 文件内容
  - **When** 计数行号
  - **Then** 行号从 1 开始；空行（仅空白字符）显示为空行，不跳过

**实现**：`Sources/Search/ContentSearchProvider.swift`、`Sources/CLI/TerminalFormatter.swift` · **测试**：`Tests/SearchTests/ContentSearchTests.swift`

### REQ-1.4-04 Content Search Performance & Limits ✅ 🖥️ P1

**用户故事**

作为「系统」，我希望内容搜索具备明确的性能基准与资源限制，以便搜索不会卡死系统或消耗过多资源（用户无直接感知）。

**验收标准（Given / When / Then）**

- **AC1 — 性能基准（目标，未自动化）**
  - **Given** 不同规模的内容搜索任务
  - **When** 执行
  - **Then** 设计目标：100 个小文件（< 10KB）< 1s；1000 个小文件（< 100KB）< 5s；10 个中等文件（1-10MB）< 3s；1 个大文件（接近 50MB 上限）< 10s。当前测试套件未对这些阈值做 `measure` 基准断言，仅验证功能正确性

- **AC2 — 单文件大小上限**
  - **Given** 单文件 > 50MB（`Constants.ContentScanner.maxFileSize = 50_000_000`）
  - **When** 扫描
  - **Then** 静默跳过，返回空匹配（无 `skipped_large` 标记）

- **AC3 — 总 I/O 截断**
  - **Given** 单次查询累计读取量达到 512MB（`Constants.ContentScanner.maxTotalIO`）
  - **When** 超限
  - **Then** 停止后续文件扫描，返回已找到结果（当前实现未附截断警告文案）

- **AC4 — 每文件匹配上限**
  - **Given** 单文件匹配数
  - **When** 扫描
  - **Then** 当前实现未设置单文件匹配上限（收集该文件全部匹配行，仅受行数与 `maxLineLength` 自然限制）

- **AC5 — 并发限制**
  - **Given** 候选文件
  - **When** 执行扫描
  - **Then** 当前实现按候选顺序串行扫描；`Constants.ContentScanner.maxConcurrentScans = 8` 常量已定义（测试断言其值），但 provider 未应用 TaskGroup

- **AC6 — 候选文件数上限**
  - **Given** 候选文件数 > 10,000
  - **When** 准备扫描
  - **Then** 拒绝执行，提示用户缩小范围

- **AC7 — REPL 进度指示器**
  - **Given** 内容搜索耗时较长（REPL 模式）
  - **When** 扫描中
  - **Then** 当前实现未提供 `扫描中... X/Y 文件 (Z 匹配)` 进度指示器（无 ANSI `\r` 覆盖输出）

- **AC8 — 搜索取消**
  - **Given** 正在进行的扫描
  - **When** 搜索被取消（`cancel(queryID:)`）
  - **Then** 当前实现为 no-op，不主动中断 in-flight 读取

- **AC9 — 资源限制配置**
  - **Given** v1.4 版本
  - **When** 查看资源限制来源
  - **Then** 使用硬编码默认值（未来可通过 ConfigStore 配置）

**实现**：`Sources/Search/ContentScanner.swift` · **测试**：`Tests/SearchTests/ContentSearchTests.swift`

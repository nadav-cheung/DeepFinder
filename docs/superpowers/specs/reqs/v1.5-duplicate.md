# v1.5 — 重复文件

6 项 REQ (P0: 5, P1: 1)

提供重复文件查找能力：按文件名、大小、内容哈希查找重复文件，以及查找空文件/空目录和超长文件名。对标 Everything 的 `dupe:`、`dupesize:`、`dupe:content:`、`empty:` 函数。

**前置依赖**：v1.1 高级搜索语法（QueryParser）、v1.2 元数据过滤（FilterExpression、SizeFilter）、v0.2 文件系统（FileRecord）。

**设计原则**：重复查找是特殊的搜索操作——不是文件名匹配，而是对文件集合的分组/聚合操作。结果以"重复组"形式呈现，每组包含 2 个或更多相同属性的文件。

---

### REQ-1.5-01 Name Duplicate — dupe: 按名称查重 ✅ 🖥️ P0

**用户场景**：用户整理下载文件夹，发现可能有多个同名文件散落在不同目录中。输入 `dupe:` → 立即看到所有同名文件分组显示。输入 `report dupe:` → 只看名称含 "report" 的同名文件。

**操作流程**：
1. 用户输入 `dupe:` → 对所有文件按文件名（不含路径）分组
2. 筛选出组大小 >= 2 的组（至少 2 个同名文件）
3. 按组显示结果
4. 用户输入 `report dupe:` → 先筛选名称含 "report" 的文件，再按名称分组

**语法**（对标 Everything `dupe:`）：

```
dupe:                  查找所有同名文件
<keyword> dupe:        名称含 <keyword> 的同名文件
folder: dupe:          仅目录中的同名项
ext:<ext> dupe:        指定扩展名的同名文件
```

**分组规则**：
- 比较基于 `FileRecord.name`（文件名，不含路径），NFC 统一化 + lowercased
- 大小写不敏感：`Report.pdf` 和 `report.PDF` 视为同名
- 扩展名参与比较：`report.pdf` 和 `report.docx` 不是同名
- 目录参与比较：同名目录也视为重复

**结果格式**：

```
--- report.pdf (3 copies) ---
  /Users/nadav/Documents/report.pdf        2.4 MB  2026-05-28
  /Users/nadav/Downloads/report.pdf        2.4 MB  2026-05-15
  /Users/nadav/Desktop/report.pdf          2.1 MB  2026-04-30

--- notes.md (2 copies) ---
  /Users/nadav/Documents/notes.md          1.2 KB  2026-05-29
  /Users/nadav/projects/deep/notes.md      3.8 KB  2026-05-20
```

- 每组标题行：`--- <filename> (<N> copies) ---`
- 组内文件按路径排序，显示完整路径、大小、修改日期
- `--json` 模式：输出 `DuplicateGroup` 数组
- `--0` 模式：输出所有重复文件的路径（NUL 分隔，不分组）

**数据模型**：

```swift
struct DuplicateGroup: Sendable, Codable {
    let key: String                 // 分组键（文件名 / 大小 / 哈希值，取决于查重类型）
    let records: [FileRecord]       // 组内所有文件记录
    let groupSize: Int              // records.count（>= 2）
}

enum DuplicateType: String, Sendable, Codable {
    case name       // dupe: — 按文件名
    case size       // sizedupe: — 按文件大小
    case hash       // hashdupe: — 按内容哈希
}
```

**验收标准**：
- [ ] `dupe:` 返回所有同名文件分组
- [ ] 与文件名搜索组合：`report dupe:` 先筛选再分组
- [ ] 与过滤器组合：`ext:pdf dupe:` 仅 PDF 同名文件
- [ ] `folder: dupe:` 仅同名目录
- [ ] 分组基于 NFC + lowercased 文件名比较
- [ ] 大小写不敏感
- [ ] 每组 >= 2 个文件
- [ ] 结果以 `DuplicateGroup` 格式输出
- [ ] 终端输出分组格式（`--- <name> (N copies) ---`）
- [ ] `--json` 输出 `DuplicateGroup` 数组
- [ ] `--0` 输出所有重复文件路径（不分组）
- [ ] 无重复时输出 "未找到重复文件"（exit code 1）

**测试覆盖**：
- [ ] 3 个同名文件 → 1 组，3 个文件
- [ ] 不同大小写同名文件 → 同一组
- [ ] 同名不同扩展名 → 不同组
- [ ] 与文件名搜索组合过滤
- [ ] 与 ext: 组合
- [ ] 无重复文件 → 0 组
- [ ] 大量文件性能：10 万文件中查重 < 500ms

### REQ-1.5-02 Size Duplicate — sizedupe: 按大小查重 ✅ 🖥️ P0

**用户场景**：用户想找出内容可能相同的文件（相同大小的文件有可能是重复的）。输入 `sizedupe:` → 看到所有大小相同的文件分组。结合文件大小过滤缩小范围：`size:>1mb sizedupe:` → 只看大于 1MB 的等大文件。

**操作流程**：
1. 用户输入 `sizedupe:` → 对所有文件按 `FileRecord.size` 分组
2. 筛选出组大小 >= 2 的组
3. 按组显示结果
4. 用户输入 `size:>1mb sizedupe:` → 先筛选大文件，再按大小分组

**语法**（对标 Everything `dupesize:`）：

```
sizedupe:              查找所有同大小文件
size:><value> sizedupe:   大于指定大小的同大小文件
<keyword> sizedupe:    名称含 <keyword> 的同大小文件
ext:<ext> sizedupe:    指定扩展名的同大小文件
```

**分组规则**：
- 比较基于 `FileRecord.size`（Int64 字节值），精确匹配
- 目录的 size 为 0：`folder: sizedupe:` 会把所有空目录归为一组
- 同大小组按大小降序排列（大文件优先，更有可能是重复内容）

**结果格式**：

```
--- 2,431,744 bytes (3 files) ---
  /Users/nadav/Documents/report.pdf        report.pdf
  /Users/nadav/Downloads/report.pdf        report.pdf
  /Users/nadav/Backup/report-2026.pdf      report-2026.pdf

--- 1,536 bytes (2 files) ---
  /Users/nadav/config.json                 config.json
  /Users/nadav/projects/app/config.json    config.json
```

- 组标题行：`--- <size> bytes (<N> files) ---`
- 大小格式化：带千位分隔符的字节数
- 组内按文件名排序

**验收标准**：
- [ ] `sizedupe:` 返回所有同大小文件分组
- [ ] 与 size: 过滤组合：`size:>1mb sizedupe:`
- [ ] 与文件名搜索组合
- [ ] 与 ext: 组合
- [ ] 分组基于 FileRecord.size 精确匹配
- [ ] 同大小组按大小降序排列
- [ ] 组内按文件名排序
- [ ] 终端输出分组格式
- [ ] `--json` 和 `--0` 模式
- [ ] 性能：10 万文件中按大小分组 < 200ms（Hash 分组 O(n)）

**测试覆盖**：
- [ ] 相同大小文件 → 同组
- [ ] 不同大小文件 → 不同组
- [ ] 与 size: 过滤组合
- [ ] 空文件归为一组
- [ ] 按大小降序排列
- [ ] 无同大小文件 → 0 组

### REQ-1.5-03 Hash Duplicate — hashdupe: 按内容哈希查重 ✅ 🖥️ P0

**用户场景**：用户想找出真正内容相同的文件（即使文件名不同）。输入 `hashdupe:` → 对文件计算 SHA-256 哈希，找到内容完全一致的文件。这是最精确的重复文件检测，用于清理磁盘空间。

**操作流程**：
1. 用户输入 `hashdupe:` → 先按文件大小分组（优化：只对同大小文件计算哈希）
2. 对组大小 >= 2 的大小组中的文件计算 SHA-256 哈希
3. 按哈希值二次分组
4. 筛选出哈希相同的组（>= 2 个文件）
5. 显示结果

**语法**（对标 Everything `dupe:content:`）：

```
hashdupe:              查找所有内容相同的文件
size:><value> hashdupe:   大于指定大小的内容重复文件
ext:<ext> hashdupe:    指定扩展名的内容重复文件
<keyword> hashdupe:    名称含 <keyword> 的内容重复文件
```

**两阶段优化**：
1. **第一阶段（快速）**：按 FileRecord.size 分组 → 仅组大小 >= 2 的组进入第二阶段
2. **第二阶段（慢速）**：对候选文件计算 SHA-256 哈希 → 按哈希值二次分组

**哈希计算策略**：
- 使用 CryptoKit `SHA256`（Apple 框架，非外部依赖）
- 流式哈希：按 64KB 块读取文件，增量更新哈希（不一次性加载整个文件）
- 文件打开失败（权限、不存在）→ 跳过该文件，不计入结果
- 哈希计算超时：单文件 10s（大文件），超时跳过
- 总哈希计算超时：60s（超出截断，返回已找到的结果 + 警告）

**并发**：
- 使用 `TaskGroup` 并行计算多个文件的哈希
- 并发度限制：最多 4 个并发哈希计算（哈希计算是 CPU + I/O 密集型，不宜过高）

**提前终止优化**：
- 如果两个文件的前 64KB 哈希不同，可以提前判定不同（无需读取整个文件）
- 实现：先计算前 64KB 的 "前缀哈希"，不同则跳过完整哈希
- 前缀哈希相同的文件才计算完整 SHA-256

**结果格式**：

```
--- SHA256:a1b2c3...f0e1 (3 files, 2.4 MB each, 4.8 MB wasted) ---
  /Users/nadav/Documents/report.pdf        2.4 MB  2026-05-28
  /Users/nadav/Downloads/report.pdf        2.4 MB  2026-05-15
  /Users/nadav/Desktop/copy.pdf            2.4 MB  2026-04-30
```

- 组标题行包含哈希前缀、文件数、单文件大小、浪费空间（`(N-1) * size`）
- "wasted space" = 重复文件占用的额外空间

**验收标准**：
- [ ] `hashdupe:` 返回所有内容相同的文件分组
- [ ] 两阶段优化：先按大小分组，再计算哈希
- [ ] 使用 CryptoKit SHA-256 流式哈希
- [ ] 与过滤器组合：`size:>1mb hashdupe:`、`ext:pdf hashdupe:`
- [ ] 前缀哈希提前终止优化
- [ ] 并发哈希计算，最多 4 并发
- [ ] 单文件哈希超时 10s，总超时 60s
- [ ] 文件打开失败跳过（不报错）
- [ ] 结果显示浪费空间
- [ ] `--json` 和 `--0` 模式
- [ ] 性能：1000 个小文件（<100KB）哈希查重 < 30s

**测试覆盖**：
- [ ] 相同内容不同文件名 → 同组
- [ ] 不同内容同大小 → 不同组
- [ ] 两阶段优化验证：不同大小文件不计算哈希
- [ ] 前缀哈希提前终止
- [ ] 文件打开失败跳过
- [ ] 与 size: 过滤组合
- [ ] 与 ext: 过滤组合
- [ ] 超时截断
- [ ] 无重复 → 0 组

### REQ-1.5-04 Empty Files & Directories — empty: ✅ 🖥️ P0

**用户场景**：用户清理项目目录，想找到所有空文件夹和零字节文件。输入 `empty:` → 立即看到所有空目录和空文件列表。方便删除无用的空目录结构。

**操作流程**：
1. 用户输入 `empty:` → 扫描索引中所有文件和目录
2. 零字节文件（`FileRecord.size == 0` 且 `isDirectory == false`）
3. 空目录（`FileRecord.isDirectory == true` 且该目录下无子项）
4. 结果分两类显示

**语法**（对标 Everything `empty:`）：

```
empty:                 查找空文件和空目录
file: empty:           仅零字节文件
folder: empty:         仅空目录
ext:<ext> empty:       指定扩展名的空文件
parent:<path> empty:   指定目录下的空项
```

**空目录判定**：
- 空目录判定基于索引数据（非实时文件系统扫描）
- 判定方式：在 InMemoryIndex 中，如果某目录路径不是任何其他文件的 parentPath 前缀，则为空目录
- 具体实现：遍历所有 `isDirectory == true` 的记录，检查 InMemoryIndex 中是否有 `parentPath.hasPrefix(directoryPath + "/")` 的其他记录
- 新建但尚未被 FSEvents 索引的空目录可能不在结果中（已知限制）

**结果格式**：

```
Empty directories (12):
  /Users/nadav/projects/app/old-module/
  /Users/nadav/projects/app/unused-plugin/
  ...

Empty files (3):
  /Users/nadav/projects/app/.gitkeep          0 B
  /Users/nadav/projects/app/placeholder.txt   0 B
  /Users/nadav/tmp/scratch                    0 B
```

- 空目录和空文件分区显示
- 空目录不显示大小（目录大小无意义）
- 空文件显示 `0 B`

**验收标准**：
- [ ] `empty:` 返回空文件和空目录
- [ ] `file: empty:` 仅返回零字节文件
- [ ] `folder: empty:` 仅返回空目录
- [ ] 与 ext: 组合：`ext:txt empty:` 仅空 .txt 文件
- [ ] 与 parent: 组合：`parent:~/Documents empty:` 仅指定目录下
- [ ] 空目录判定基于索引数据（InMemoryIndex 中无子项）
- [ ] 零字节文件判定：`FileRecord.size == 0 && !isDirectory`
- [ ] 结果分区显示：空目录在前，空文件在后
- [ ] `--json` 和 `--0` 模式
- [ ] 无空项时输出 "未找到空文件或空目录"（exit code 1）

**测试覆盖**：
- [ ] 零字节文件出现在结果中
- [ ] 空目录出现在结果中
- [ ] 非空目录不出现在结果中
- [ ] 非零文件不出现在结果中
- [ ] `file:` 和 `folder:` 分别过滤
- [ ] 与 ext: 组合
- [ ] 与 parent: 组合
- [ ] 无空项 → 0 结果

### REQ-1.5-05 Filename Length Filter — len: ✅ 🖥️ P0

**用户场景**：用户在迁移文件到云存储时发现某些文件名超过 255 字符（部分文件系统限制）。输入 `len:>200` → 找到所有文件名超过 200 字符的文件。输入 `len:>100` → 找到超长文件名，逐一检查是否需要缩短。

**操作流程**：
1. 用户输入 `len:>100` → 过滤文件名长度 > 100 字符的文件
2. 用户输入 `len:255` → 精确匹配文件名长度为 255 的文件
3. 用户输入 `report len:>50` → 名称含 "report" 且文件名超过 50 字符

**语法**：

```
len:<N>                文件名长度 == N
len:<=<N>              文件名长度 <= N
len:<<N>               文件名长度 < N
len:>=<N>              文件名长度 >= N
len:><N>               文件名长度 > N
len:<start>..<end>     文件名长度范围
```

**长度计算**：
- 基于 `FileRecord.name` 的 `count`（Swift String `count` = Unicode scalar 视角的字符数）
- NFC 统一化后计算长度（与其他搜索行为一致）
- 不含路径，仅文件名部分

**规则**：
- N 必须是非负整数
- 无比较运算符时默认精确匹配（`==`）
- 与 v1.2 `size:` 语法风格一致（复用 `ComparisonOperator` 模型）

**结果格式**：
- 终端输出时额外显示文件名长度：`very_long_filename_that_exceeds_... [152 chars]`
- `--json` 模式：SearchResult 中包含 `nameLength` 字段

**验收标准**：
- [ ] 支持 5 种比较运算符 + 范围语法
- [ ] 无运算符时默认 `==`
- [ ] 长度基于 FileRecord.name.count（Unicode scalar）
- [ ] NFC 统一化后计算
- [ ] 与文件名搜索组合：`report len:>50`
- [ ] 与过滤器组合：`ext:swift len:>80`
- [ ] 无效输入（非整数、负数）报错
- [ ] 终端输出显示文件名长度
- [ ] `--json` 输出 nameLength 字段

**测试覆盖**：
- [ ] `len:>100` 返回文件名长度 > 100 的文件
- [ ] `len:10` 精确匹配长度为 10 的文件名
- [ ] `len:5..10` 范围匹配
- [ ] Unicode 文件名长度正确（中文、emoji）
- [ ] 与文件名搜索组合
- [ ] 无效输入报错
- [ ] 无匹配 → 0 结果

### REQ-1.5-06 DuplicateResult Model & IPC ✅ 🖥️ P1

**用户场景**：用户无直接感知。DuplicateResult 模型和 IPC 协议确保重复查找结果在 CLI 和 daemon 之间正确传输，并为未来 GUI（v2.0）提供统一的重复文件展示接口。

**数据模型**：

```swift
/// 重复查找结果
struct DuplicateResult: Sendable, Codable {
    let type: DuplicateType        // name / size / hash
    let groups: [DuplicateGroup]
    let totalGroups: Int           // 总组数
    let totalFiles: Int            // 重复文件总数
    let wastedSpace: Int64?        // 浪费空间（仅 hashdupe 可计算）
    let scanDuration: TimeInterval // 查找耗时
}

/// 重复文件组
struct DuplicateGroup: Sendable, Codable {
    let key: String                 // 分组键（文件名 / 大小字节数 / SHA-256 哈希前缀）
    let records: [FileRecord]       // 组内所有文件记录
    let groupSize: Int { records.count }
}

/// 空文件/目录结果
struct EmptyResult: Sendable, Codable {
    let emptyDirectories: [FileRecord]
    let emptyFiles: [FileRecord]
    var totalCount: Int { emptyDirectories.count + emptyFiles.count }
}
```

**IPC 协议**：

```swift
/// 重复查找请求
struct IPCDuplicateRequest: Codable {
    let type: DuplicateType        // name / size / hash
    let text: String?              // 可选的文件名搜索词
    let filterExpression: Data?    // 可选的过滤器
    let limit: Int                 // 最大组数（默认 1000）
}

/// 空文件查找请求
struct IPCEmptyRequest: Codable {
    let fileType: IPCFileType?     // file / folder / nil(both)
    let filterExpression: Data?
}

/// 重复查找响应
struct IPCDuplicateResponse: Codable {
    let result: DuplicateResult
    let error: String?
}

/// 空文件查找响应
struct IPCEmptyResponse: Codable {
    let result: EmptyResult
    let error: String?
}
```

**QueryParser 集成**：
- `dupe:`, `sizedupe:`, `hashdupe:`, `empty:`, `len:` 作为特殊查询类型解析
- 这些不是过滤器（FilterExpression），而是独立的查询操作
- SearchCoordinator 根据 QueryAST 中的特殊标记路由到对应处理器
- `DuplicateSearchProvider` 和 `EmptySearchProvider` 实现 `SearchProvider` 协议

**验收标准**：
- [ ] DuplicateResult / DuplicateGroup / EmptyResult Codable 往返零丢失
- [ ] IPCDuplicateRequest/Response 序列化正确
- [ ] IPCEmptyRequest/Response 序列化正确
- [ ] QueryParser 正确识别 dupe:/sizedupe:/hashdupe:/empty:/len: 特殊查询
- [ ] SearchCoordinator 路由到对应 Provider
- [ ] DuplicateResult 包含 totalGroups、totalFiles、wastedSpace、scanDuration
- [ ] 与文件名搜索和过滤器正确组合

**测试覆盖**：
- [ ] DuplicateResult Codable 往返
- [ ] DuplicateGroup Codable 往返
- [ ] EmptyResult Codable 往返
- [ ] IPC 请求/响应序列化
- [ ] QueryParser 解析特殊查询类型
- [ ] 与过滤器组合的端到端流程

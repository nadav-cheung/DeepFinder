# v1.5 — 重复文件

6 项 REQ（P0: 5，P1: 1）

> 卡片格式（OSS · BDD）：用户故事 + 验收标准（Given/When/Then）+ 实现/测试溯源。金标准见 v0.1-index-core.md。

提供重复文件查找能力：按文件名、大小、内容哈希查找重复文件，以及查找空文件/空目录和超长文件名。对标 Everything 的 `dupe:`、`dupesize:`、`dupe:content:`、`empty:` 函数。

**前置依赖**：v1.1 高级搜索语法（QueryParser）、v1.2 元数据过滤（FilterExpression、SizeFilter）、v0.2 文件系统（FileRecord）。

**设计原则**：重复查找是特殊的搜索操作——不是文件名匹配，而是对文件集合的分组/聚合操作。结果以"重复组"形式呈现，每组包含 2 个或更多相同属性的文件。

### REQ-1.5-01 Name Duplicate — dupe: 按名称查重 ✅ 🖥️ P0

**用户故事**

作为「整理下载文件夹的用户」，我希望按文件名查找散落在不同目录中的同名文件并分组显示，以便快速清理冗余副本。

**验收标准（Given / When / Then）**

- **AC1 — 全量按名分组**
  - **Given** 索引中存在若干同名文件（不同路径）
  - **When** 用户输入 `dupe:`
  - **Then** 对所有文件按 `FileRecord.name`（NFC 统一化 + lowercased）分组，仅返回组大小 ≥ 2 的组

- **AC2 — 关键词预筛选**
  - **Given** 索引中存在多个文件
  - **When** 用户输入 `report dupe:`
  - **Then** 先筛选名称含 "report" 的文件，再按名称分组

- **AC3 — 与过滤器组合**
  - **Given** 索引中存在多种扩展名的同名文件
  - **When** 用户输入 `ext:pdf dupe:` 或 `folder: dupe:`
  - **Then** `ext:pdf dupe:` 仅返回 PDF 同名文件；`folder: dupe:` 仅返回同名目录

- **AC4 — 比较规则**
  - **Given** 大小写混合、不同扩展名、同名目录
  - **When** 分组比较
  - **Then** 基于文件名（不含路径）NFC + lowercased 比较；`Report.pdf` 与 `report.PDF` 视为同名；`report.pdf` 与 `report.docx` 不同组；同名目录视为重复

- **AC5 — 终端分组格式**
  - **Given** 一组同名文件
  - **When** 终端输出
  - **Then** 组标题行 `--- <filename> (<N> copies) ---`，组内文件按路径排序，显示完整路径、大小、修改日期

- **AC6 — JSON / NUL 模式**
  - **Given** 重复查找结果
  - **When** 使用 `--json` 或 `--0`
  - **Then** `--json` 输出 `DuplicateGroup` 数组；`--0` 输出所有重复文件路径（NUL 分隔，不分组）

- **AC7 — 无重复**
  - **Given** 索引中无同名文件
  - **When** 用户输入 `dupe:`
  - **Then** 输出 "未找到重复文件"，exit code 1

- **AC8 — 大规模性能**
  - **Given** 10 万文件的索引
  - **When** 执行 `dupe:` 查重
  - **Then** 耗时 < 500 ms

**实现**：`Sources/Search/DuplicateFinder.swift` · **测试**：`Tests/SearchTests/DuplicateFinderTests.swift`

### REQ-1.5-02 Size Duplicate — sizedupe: 按大小查重 ✅ 🖥️ P0

**用户故事**

作为「排查重复内容的用户」，我希望按文件大小查找等大文件并分组（大小相同可能是内容重复），以便结合大小过滤进一步定位可能的重复。

**验收标准（Given / When / Then）**

- **AC1 — 全量按大小分组**
  - **Given** 索引中存在多个文件
  - **When** 用户输入 `sizedupe:`
  - **Then** 对所有文件按 `FileRecord.size`（Int64 字节值）精确匹配分组，仅返回组大小 ≥ 2 的组

- **AC2 — 与过滤器组合**
  - **Given** 索引中存在不同大小的文件
  - **When** 用户输入 `size:>1mb sizedupe:`、`<keyword> sizedupe:` 或 `ext:<ext> sizedupe:`
  - **Then** 先筛选（大文件 / 关键词 / 扩展名），再按大小分组

- **AC3 — 目录与空文件**
  - **Given** 目录 size 为 0
  - **When** 用户输入 `folder: sizedupe:`
  - **Then** 所有空目录归为同一组

- **AC4 — 排序**
  - **Given** 多个同大小组
  - **When** 输出
  - **Then** 同大小组按大小降序排列（大文件优先），组内按文件名排序

- **AC5 — 终端分组格式**
  - **Given** 一组同大小文件
  - **When** 终端输出
  - **Then** 组标题行 `--- <size> bytes (<N> files) ---`，大小带千位分隔符

- **AC6 — JSON / NUL 模式**
  - **Given** 大小查重结果
  - **When** 使用 `--json` 或 `--0`
  - **Then** `--json` 输出 `DuplicateGroup` 数组；`--0` 输出所有重复文件路径（NUL 分隔）

- **AC7 — 大规模性能**
  - **Given** 10 万文件的索引
  - **When** 执行 `sizedupe:` 按大小分组
  - **Then** 耗时 < 200 ms（Hash 分组 O(n)）

**实现**：`Sources/Search/DuplicateFinder.swift` · **测试**：`Tests/SearchTests/DuplicateFinderTests.swift`

### REQ-1.5-03 Hash Duplicate — hashdupe: 按内容哈希查重 ✅ 🖥️ P0

**用户故事**

作为「清理磁盘空间的用户」，我希望按文件内容哈希查找真正内容相同的文件（即使文件名不同），以便精确清理冗余副本。

**验收标准（Given / When / Then）**

- **AC1 — 全量按哈希分组**
  - **Given** 索引中存在多个文件
  - **When** 用户输入 `hashdupe:`
  - **Then** 返回所有内容完全相同的文件分组（组大小 ≥ 2）

- **AC2 — 两阶段优化**
  - **Given** 大量文件
  - **When** 执行哈希查重
  - **Then** 第一阶段（快速）按 `FileRecord.size` 分组，仅组大小 ≥ 2 的组进入第二阶段；第二阶段（慢速）对候选文件计算 SHA-256 并二次分组

- **AC3 — 与过滤器组合**
  - **Given** 索引中存在多种文件
  - **When** 用户输入 `size:>1mb hashdupe:` 或 `ext:pdf hashdupe:`
  - **Then** 先筛选大文件 / PDF 文件，再执行内容查重

- **AC4 — 流式哈希与失败处理**
  - **Given** 使用 CryptoKit `SHA256`
  - **When** 计算哈希
  - **Then** 按 64KB 块流式读取、增量更新；文件打开失败（权限 / 不存在）跳过该文件不计入结果

- **AC5 — 前缀哈希提前终止**
  - **Given** 两个同大小文件
  - **When** 前 64KB 的前缀哈希不同
  - **Then** 提前判定不同，不读取整个文件；前缀哈希相同者才计算完整 SHA-256

- **AC6 — 并发与超时**
  - **Given** 使用 `TaskGroup` 并行哈希
  - **When** 计算哈希
  - **Then** 并发度最多 4；单文件哈希超时 10s 跳过；总超时 60s 截断并返回已找到的结果 + 警告

- **AC7 — 结果格式与浪费空间**
  - **Given** 一组内容相同的文件
  - **When** 输出
  - **Then** 组标题行包含哈希前缀、文件数、单文件大小、浪费空间（`(N-1) * size`）；支持 `--json` 和 `--0` 模式

- **AC8 — 性能**
  - **Given** 1000 个小文件（< 100KB）
  - **When** 哈希查重
  - **Then** 耗时 < 30 s

**实现**：`Sources/Search/FileHasher.swift`, `Sources/Search/DuplicateFinder.swift` · **测试**：`Tests/SearchTests/DuplicateFinderTests.swift`

### REQ-1.5-04 Empty Files & Directories — empty: ✅ 🖥️ P0

**用户故事**

作为「清理项目目录的用户」，我希望查找所有空文件夹和零字节文件，以便删除无用的空目录结构和占位空文件。

**验收标准（Given / When / Then）**

- **AC1 — 全量查找空项**
  - **Given** 索引中存在空文件与空目录
  - **When** 用户输入 `empty:`
  - **Then** 返回零字节文件与空目录，分区显示

- **AC2 — 类型过滤**
  - **Given** 索引中同时存在空文件和空目录
  - **When** 用户输入 `file: empty:` 或 `folder: empty:`
  - **Then** `file: empty:` 仅返回零字节文件；`folder: empty:` 仅返回空目录

- **AC3 — 与过滤器组合**
  - **Given** 索引中存在多种文件
  - **When** 用户输入 `ext:txt empty:` 或 `parent:~/Documents empty:`
  - **Then** `ext:txt empty:` 仅返回空 .txt 文件；`parent:~/Documents empty:` 仅返回指定目录下的空项

- **AC4 — 零字节文件判定**
  - **Given** 文件记录
  - **When** 判定是否为空文件
  - **Then** 条件为 `FileRecord.size == 0 && isDirectory == false`

- **AC5 — 空目录判定（基于索引）**
  - **Given** `isDirectory == true` 的记录
  - **When** 判定是否为空目录
  - **Then** 基于 InMemoryIndex 数据：若没有任何其他记录的 `parentPath.hasPrefix(directoryPath + "/")`，则为空目录（非实时文件系统扫描）

- **AC6 — 结果格式**
  - **Given** 空目录与空文件结果
  - **When** 输出
  - **Then** 分区显示，空目录在前、空文件在后；空目录不显示大小；空文件显示 `0 B`；支持 `--json` 和 `--0` 模式

- **AC7 — 无空项**
  - **Given** 索引中无空文件或空目录
  - **When** 用户输入 `empty:`
  - **Then** 输出 "未找到空文件或空目录"，exit code 1

**实现**：`Sources/Search/DuplicateFinder.swift` · **测试**：`Tests/SearchTests/DuplicateFinderTests.swift`

> 备注：新建但尚未被 FSEvents 索引的空目录可能不在结果中（已知限制）。

### REQ-1.5-05 Filename Length Filter — len: ✅ 🖥️ P0

**用户故事**

作为「迁移文件到云存储的用户」，我希望按文件名长度过滤文件（如 `len:>200` 找超长文件名），以便逐一检查并缩短可能超出文件系统限制的文件名。

**验收标准（Given / When / Then）**

- **AC1 — 比较运算符**
  - **Given** 索引中的文件
  - **When** 用户输入 `len:<N>` / `len:<=<N>` / `len:<<N>` / `len:>=<N>` / `len:><N>` / `len:<start>..<end>`
  - **Then** 对应返回文件名长度 == / <= / < / >= / > N，或落在 [start, end] 范围内的文件

- **AC2 — 默认精确匹配**
  - **Given** 输入无比较运算符
  - **When** 用户输入 `len:255`
  - **Then** 默认精确匹配（`==`），返回文件名长度为 255 的文件

- **AC3 — 长度计算口径**
  - **Given** 任意文件名
  - **When** 计算长度
  - **Then** 基于 `FileRecord.name` 的 `count`（Swift String `count` = Unicode scalar 视角的字符数），NFC 统一化后计算，不含路径

- **AC4 — 与搜索 / 过滤器组合**
  - **Given** 索引中的文件
  - **When** 用户输入 `report len:>50` 或 `ext:swift len:>80`
  - **Then** 名称含 "report" 且长度 > 50 的文件；或扩展名为 swift 且长度 > 80 的文件

- **AC5 — 无效输入**
  - **Given** 非整数或负数 N
  - **When** 用户输入
  - **Then** 报错

- **AC6 — 结果格式**
  - **Given** 长度过滤结果
  - **When** 输出
  - **Then** 终端额外显示文件名长度（`<name> [152 chars]`）；`--json` 模式 `SearchResult` 含 `nameLength` 字段

- **AC7 — 无匹配**
  - **Given** 无文件满足长度条件
  - **When** 用户输入
  - **Then** 0 结果

**实现**：`Sources/Search/SearchFilter.swift` · **测试**：`Tests/SearchTests/SearchFilterTests.swift`

### REQ-1.5-06 DuplicateResult Model & IPC ✅ 🖥️ P1

**用户故事**

作为「系统」，我希望有统一的 `DuplicateResult` 数据模型和 IPC 协议，以便重复查找结果在 CLI 与 daemon 之间正确传输，并为未来 GUI（v2.0）提供统一的重复文件展示接口。

**验收标准（Given / When / Then）**

- **AC1 — 数据模型往返零丢失**
  - **Given** `DuplicateResult` / `DuplicateGroup` / `EmptyResult`
  - **When** Codable 编码后解码
  - **Then** 所有字段相等（零丢失）

- **AC2 — DuplicateResult 字段**
  - **Given** 一次重复查找
  - **When** 构造 DuplicateResult
  - **Then** 含 `type`（name/size/hash）、`groups`、`totalGroups`、`totalFiles`、`wastedSpace`（仅 hashdupe 可计算）、`scanDuration`

- **AC3 — IPC 序列化**
  - **Given** `IPCDuplicateRequest/Response` 和 `IPCEmptyRequest/Response`
  - **When** IPC 传输
  - **Then** 请求 / 响应序列化正确

- **AC4 — QueryParser 特殊查询识别**
  - **Given** 用户输入含 `dupe:` / `sizedupe:` / `hashdupe:` / `empty:` / `len:`
  - **When** QueryParser 解析
  - **Then** 识别为独立查询操作（非 FilterExpression），SearchCoordinator 路由到对应 Provider（`DuplicateSearchProvider` / `EmptySearchProvider`）

- **AC5 — 与搜索 / 过滤器组合**
  - **Given** 特殊查询 + 文件名搜索词 + 过滤器
  - **When** 端到端流程
  - **Then** 文件名搜索与过滤器正确参与组合

**实现**：`Sources/Search/DuplicateFinder.swift`, `Sources/Daemon/IPCProtocol.swift` · **测试**：`Tests/SearchTests/DuplicateFinderTests.swift`

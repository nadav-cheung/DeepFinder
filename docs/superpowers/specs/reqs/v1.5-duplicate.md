# v1.5 — 重复文件

6 项 REQ（P0: 5，P1: 1）

> 卡片格式（OSS · BDD）：用户故事 + 验收标准（Given/When/Then）+ 实现/测试溯源。金标准见 v0.1-index-core.md。

提供重复文件查找能力：按文件名、大小、内容哈希查找重复文件，以及查找空文件/空目录和超长文件名。对标 Everything 的 `dupe:`、`dupesize:`、`dupe:content:`、`empty:` 函数。

**前置依赖**：v1.1 高级搜索语法（QueryParser）、v1.2 元数据过滤（FilterExpression、SizeFilter）、v0.2 文件系统（FileRecord）。

**设计原则**：重复查找是特殊的搜索操作——不是文件名匹配，而是对文件集合的分组/聚合操作。结果以"重复组"形式呈现，每组包含 2 个或更多相同属性的文件。

### REQ-1.5-01 Name Duplicate — dupe: 按名称查重 ✅ 🖥️ P0

**用户故事**

作为「整理下载文件夹的用户」，我希望按文件名查找散落在不同目录中的同名文件并分组显示，以便快速清理冗余副本。

**验收标准（Given / When / Then）**

- **AC1 — 全量按名分组**
  - **Given** 索引中存在若干同名文件（不同路径）
  - **When** 调用 `DuplicateFinder.findByName()`
  - **Then** 对所有记录按 `FileRecord.name`（lowercased）分组，仅返回组大小 ≥ 2 的组，组按键（lowercased 文件名）升序排序

- **AC2 — 不同名不分组**
  - **Given** 索引中存在两个不同名文件
  - **When** 调用 `findByName()`
  - **Then** 返回空数组

- **AC3 — 比较规则**
  - **Given** 大小写混合、不同扩展名的文件
  - **When** 分组比较
  - **Then** 基于文件名（不含路径）lowercased 比较；`Report.pdf` 与 `report.PDF` 视为同名；`report.pdf` 与 `report.docx` 不同组

- **AC4 — 大规模性能**
  - **Given** 10 万文件的索引
  - **When** 执行按名查重
  - **Then** 耗时 < 500 ms（manual target，仅作设计目标，无对应测试基准）

**实现**：`Sources/Search/DuplicateFinder.swift` · **测试**：`Tests/SearchTests/DuplicateFinderTests.swift`

### REQ-1.5-02 Size Duplicate — sizedupe: 按大小查重 ✅ 🖥️ P0

**用户故事**

作为「排查重复内容的用户」，我希望按文件大小查找等大文件并分组（大小相同可能是内容重复），以便结合大小过滤进一步定位可能的重复。

**验收标准（Given / When / Then）**

- **AC1 — 全量按大小分组**
  - **Given** 索引中存在多个文件
  - **When** 调用 `DuplicateFinder.findBySize()`
  - **Then** 对所有记录按 `FileRecord.size`（Int64 字节值）精确匹配分组，仅返回组大小 ≥ 2 的组，组按键（size 字符串）升序排序

- **AC2 — 不同大小不分组**
  - **Given** 索引中存在两个不同大小的文件
  - **When** 调用 `findBySize()`
  - **Then** 返回空数组

- **AC3 — 大规模性能**
  - **Given** 10 万文件的索引
  - **When** 执行按大小分组
  - **Then** 耗时 < 200 ms（manual target，Hash 分组 O(n)，仅作设计目标，无对应测试基准）

**实现**：`Sources/Search/DuplicateFinder.swift` · **测试**：`Tests/SearchTests/DuplicateFinderTests.swift`

### REQ-1.5-03 Hash Duplicate — hashdupe: 按内容哈希查重 ✅ 🖥️ P0

**用户故事**

作为「清理磁盘空间的用户」，我希望按文件内容哈希查找真正内容相同的文件（即使文件名不同），以便精确清理冗余副本。

**验收标准（Given / When / Then）**

- **AC1 — 按哈希分组**
  - **Given** 索引中存在多个文件，且调用方传入候选文件路径列表
  - **When** 调用 `DuplicateFinder.findByHash(paths:)`
  - **Then** 对传入路径计算 SHA-256，按哈希分组，仅返回组大小 ≥ 2 的组（组按键=哈希值升序）。调用方负责预先按大小筛选以提升效率

- **AC2 — 不同内容不分组**
  - **Given** 两个内容不同的文件
  - **When** 调用 `findByHash(paths:)`
  - **Then** 返回空数组

- **AC3 — 流式哈希与失败处理**
  - **Given** 使用 CryptoKit `SHA256`
  - **When** 计算哈希
  - **Then** 按 64KB 块流式读取、`hasher.update` 增量更新，返回小写十六进制摘要（64 字符）；文件不存在或打开失败返回 nil 并跳过该文件不计入结果

- **AC4 — 已知值正确性**
  - **Given** 空文件 / 已知内容文件
  - **When** 调用 `FileHasher.sha256(ofFileAtPath:)`
  - **Then** 空文件摘要为 `e3b0c442...b855`；"hello" 摘要为 `2cf24dba...b9824`；多次调用一致

- **AC5 — 性能**
  - **Given** 1000 个小文件（< 100KB）
  - **When** 哈希查重
  - **Then** 耗时 < 30 s（manual target，仅作设计目标，无对应测试基准）

> 备注：`FileHasher` 为单线程顺序哈希（无 TaskGroup 并发、无超时、无前缀哈希提前终止）。两阶段 size 预筛由 daemon 的 `duplicateProvider` 在分发 `.hash` 策略时完成（先按 size 分组，仅对组大小 ≥ 2 的同 size 组计算 SHA-256），避免对全索引哈希。`hashdupe:` 命令经 CLI `DuplicateCommand` 路由到 `.duplicateQuery(strategy: .hash)`，结果经 `TerminalFormatter.formatDuplicates` 分组输出（single-shot + REPL）。

**实现**：`Sources/Search/FileHasher.swift`, `Sources/Search/DuplicateFinder.swift` · **测试**：`Tests/SearchTests/DuplicateFinderTests.swift`

### REQ-1.5-04 Empty Files & Directories — empty: ✅ 🖥️ P0

**用户故事**

作为「清理项目目录的用户」，我希望查找所有空文件夹和零字节文件，以便删除无用的空目录结构和占位空文件。

**验收标准（Given / When / Then）**

- **AC1 — 全量查找空项**
  - **Given** 索引中存在空文件与空目录
  - **When** 调用 `DuplicateFinder.findEmpty()`
  - **Then** 返回零字节文件与空目录的合并列表（按 `FileRecord.id` 升序排序）

- **AC2 — 零字节文件判定**
  - **Given** 文件记录
  - **When** 判定是否为空文件
  - **Then** 条件为 `FileRecord.size == 0 && isDirectory == false`

- **AC3 — 空目录判定（基于索引）**
  - **Given** `isDirectory == true` 的记录
  - **When** 判定是否为空目录
  - **Then** 基于 InMemoryIndex 数据：先收集所有非目录记录的 `parentPath` 构成"有子项目录集合"，若该目录的 `path` 不在此集合中则为空目录（精确匹配，非实时文件系统扫描）

- **AC4 — 非空目录排除**
  - **Given** 一个目录记录，且索引中存在以该目录为父路径的文件记录
  - **When** 调用 `findEmpty()`
  - **Then** 该目录不在结果中

- **AC5 — 无空项**
  - **Given** 索引中无空文件或空目录
  - **When** 调用 `findEmpty()`
  - **Then** 返回空数组

**实现**：`Sources/Search/DuplicateFinder.swift` · **测试**：`Tests/SearchTests/DuplicateFinderTests.swift`

> 备注：新建但尚未被 FSEvents 索引的空目录可能不在结果中（已知限制）。

### REQ-1.5-05 Filename Length Filter — len: ✅ 🖥️ P0

**用户故事**

作为「迁移文件到云存储的用户」，我希望按文件名长度过滤文件（如 `len:>200` 找超长文件名），以便逐一检查并缩短可能超出文件系统限制的文件名。

**验收标准（Given / When / Then）**

- **AC1 — 比较运算符**
  - **Given** 索引中的文件
  - **When** 用户输入 `len:<N>` / `len:<=<N>` / `len:<<N>` / `len:>=<N>` / `len:><N>` / `len:<start>..<end>`
  - **Then** 对应返回文件名长度 >= / <= / < / >= / > N，或落在 [start, end] 范围内的文件（`>` 映射为 `nameLengthMin(n+1)`，`<` 映射为 `nameLengthMax(n-1)`）

- **AC2 — 默认精确匹配**
  - **Given** 输入无比较运算符
  - **When** 用户输入 `len:255`
  - **Then** 默认精确匹配，实现为 `nameLengthRange(255...255)`，返回文件名长度为 255 的文件

- **AC3 — 长度计算口径**
  - **Given** 任意文件名
  - **When** 计算长度
  - **Then** 基于 `FileRecord.name.unicodeScalars.count`（Unicode scalar 数，非 Swift `String.count` 的字形簇数，也非字节数），不含路径

- **AC4 — 与搜索 / 过滤器组合**
  - **Given** 索引中的文件
  - **When** 用户输入 `report len:>50` 或 `ext:swift len:>80`
  - **Then** 名称含 "report" 且长度 > 50 的文件；或扩展名为 swift 且长度 > 80 的文件

- **AC5 — 无效输入**
  - **Given** 非整数 N（如 `len:abc`）
  - **When** 解析 `len:` 值
  - **Then** `parseNameLengthFilter` 返回 nil，该过滤器被静默忽略（不抛错，不影响其他搜索条件）

- **AC6 — 无匹配**
  - **Given** 无文件满足长度条件
  - **When** 用户输入
  - **Then** 0 结果

**实现**：`Sources/Search/SearchFilter.swift` · **测试**：`Tests/SearchTests/SearchFilterTests.swift`

### REQ-1.5-06 DuplicateResult Model & IPC ✅ 🖥️ P1

**用户故事**

作为「系统」，我希望有统一的 `DuplicateResult` 数据模型和 IPC 协议，以便重复查找结果在 CLI 与 daemon 之间正确传输，并为未来 GUI（v2.0）提供统一的重复文件展示接口。

**验收标准（Given / When / Then）**

- **AC1 — DuplicateGroup 数据模型往返零丢失**
  - **Given** `DuplicateGroup`（key + records）
  - **When** Codable 编码后解码
  - **Then** key 与 records 相等（零丢失）

- **AC2 — DuplicateGroup 字段**
  - **Given** 一次重复查找
  - **When** 构造 DuplicateGroup
  - **Then** 含 `key`（分组键：文件名 / size 字符串 / 哈希值）与 `records: [FileRecord]`

- **AC3 — IPC 请求枚举**
  - **Given** `IPCRequest.duplicateQuery(strategy:)`
  - **When** IPC 传输
  - **Then** `strategy` 为 `DuplicateQueryStrategy`（name/size/hash/empty）枚举，请求 / 响应序列化正确

- **AC4 — IPC 响应枚举**
  - **Given** daemon 端 `duplicateProvider` 闭包返回 `[DuplicateGroup]`
  - **When** IPCServer 处理 `.duplicateQuery(strategy:)`
  - **Then** 返回 `IPCResponse.duplicates([DuplicateGroup])`

- **AC5 — 端到端 CLI 路由**
  - **Given** CLI 输入 `dupe:` / `sizedupe:` / `hashdupe:` / `empty:`（裸关键字或冒号形式，大小写不敏感）
  - **When** SingleShot 或 REPL 处理
  - **Then** `DuplicateCommand.detect` 识别策略并发送 `.duplicateQuery(strategy:)`；daemon `duplicateProvider`（`DaemonMain.makeIPCServer`，两阶段 size 预筛仅对 `.hash`）分发到对应 `DuplicateFinder` 方法；`.duplicates` 响应经 `TerminalFormatter.formatDuplicates` 输出（JSON / NUL / 分组 ANSI 三种模式）

> 备注：当前 IPC 协议不含 `DuplicateResult` / `EmptyResult` 聚合类型，也无 `totalGroups` / `totalFiles` / `wastedSpace` / `scanDuration` 等汇总字段——聚合统计由调用方在收到 `[DuplicateGroup]` 后自行计算。`dupe:` / `sizedupe:` / `hashdupe:` / `empty:` 经 `DuplicateCommand.detect` 路由（不经 `SearchCoordinator`——重复查找是聚合操作而非文件名匹配）；`len:` 经 FilterPipeline 作为 SearchFilter 处理。

**实现**：`Sources/Search/DuplicateFinder.swift`, `Sources/Daemon/IPCProtocol.swift` · **测试**：`Tests/SearchTests/DuplicateFinderTests.swift`

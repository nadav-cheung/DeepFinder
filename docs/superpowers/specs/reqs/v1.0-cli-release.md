### REQ-1.0-01 CLI 集成测试 ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望安装 DeepFinder 后 CLI 搜索、daemon 启停、REPL 交互全部可靠工作，以便端到端功能稳定可用、不因集成问题出错。

**验收标准（Given / When / Then）**

- **AC1 — Single-shot 查询**
  - **Given** 一个已索引的 daemon 与可用的 CLI 二进制
  - **When** 执行 `deepfinder "query"`
  - **Then** 正确输出结果（含 ANSI 高亮、JSON 模式、NUL 模式）

- **AC2 — Exit codes**
  - **Given** 不同搜索结果与运行状态
  - **When** 执行 single-shot 查询
  - **Then** 退出码符合约定：0=成功，1=无结果，2=daemon 错误，3=查询错误

- **AC3 — Daemon 生命周期**
  - **Given** daemon 子命令
  - **When** 执行 start → status → stop → status 序列
  - **Then** status 在 start 后显示 running，stop 后显示 not running

- **AC4 — REPL 交互**
  - **Given** 已进入 REPL 模式
  - **When** 输入查询、`:help`、`:quit`
  - **Then** 查询输出结果；`:help` 显示命令列表；`:quit` 退出 REPL

- **AC5 — JSON 输出结构**
  - **Given** `--json` 模式输出
  - **When** 对输出进行 Codable 编解码验证
  - **Then** 结构正确，可往返还原

- **AC6 — 管道模式**
  - **Given** 输出被管道传递（如 `deepfinder "query" | cat`）
  - **When** 检查 stdout
  - **Then** 不包含任何 ANSI escape codes（纯文本）

- **AC7 — Daemon 自动启动**
  - **Given** daemon 未运行
  - **When** 执行 single-shot 查询
  - **Then** CLI 自动启动 daemon 后正常返回结果

### REQ-1.0-03 模糊纠错 ✅ 🖥️ P1

**用户故事**

作为「用户」，我希望打字过快输错时（如 `deepfinder "repotr"`）能看到纠错建议（「Did you mean: report?」）并能一键接受，以便快速修正查询、不浪费往返时间。

**验收标准（Given / When / Then）**

- **AC1 — 编辑距离纠错**
  - **Given** 用户输入的查询无精确匹配
  - **When** 计算与已知索引项的 Levenshtein 编辑距离
  - **Then** 对编辑距离 ≤2 的整词匹配给出纠错建议（按整词匹配，非前缀匹配）

- **AC2 — 建议时机**
  - **Given** 当前查询
  - **When** 评估是否给出建议
  - **Then** 建议只在实际无结果或存在更优匹配时出现

- **AC3 — 不阻塞搜索**
  - **Given** 同时显示建议与原始查询结果
  - **When** 用户选择忽略建议
  - **Then** 用户仍能看到原始查询结果（建议不阻塞当前搜索）

- **AC4 — 纠错延迟**
  - **Given** 本地算法（无网络）
  - **When** 计算纠错
  - **Then** 延迟 < 10 ms（手工目标，非 CI 断言；测试仅验证正确性，未含 `measure` 基准）

- **AC5 — 中文拼音纠错**
  - **Given** 中文输入场景
  - **When** 输入拼音查询
  - **Then** 当前实现仅支持 Levenshtein 整词纠错，不含拼音纠错（拼音能力仅在 TerminalFormatter 的高亮匹配中，见 REQ-1.0-04 AC4；拼音纠错列为未来增强）

- **AC6 — REPL 模式**
  - **Given** REPL 模式下查询无结果
  - **When** 查询返回空结果
  - **Then** 当前实现仅显示「No results found.」，不展示纠错建议，Tab 键用于命令/路径补全（libedit），不接受纠错建议（纠错建议仅在 single-shot 经由 stderr 输出，见 AC7）

- **AC7 — Single-shot 模式**
  - **Given** Single-shot 模式
  - **When** 触发纠错
  - **Then** 建议输出到 stderr，不干扰 stdout 管道

**实现**：`Sources/CLI/FuzzyCorrection.swift` · **测试**：`Tests/CLITests/FuzzyCorrectionTests.swift`

### REQ-1.0-04 ANSI 高亮匹配 ✅ 🖥️ P1

**用户故事**

作为「用户」，我希望搜索结果中匹配的部分用 ANSI 颜色高亮（如搜 `port` 时 `re**port**.pdf`、`air**port**_map.png`），以便一眼看出每个文件为何出现在结果中。

**验收标准（Given / When / Then）**

- **AC1 — 匹配高亮**
  - **Given** 终端显示模式
  - **When** 输出含匹配子串的文件名
  - **Then** 匹配子串用 ANSI bold/color 高亮

- **AC2 — 保留原始大小写**
  - **Given** 含大小写混合的文件名
  - **When** 搜索「PORT」
  - **Then** 命中文件名（如 `report.pdf`）保留原始大小写显示

- **AC3 — 多处匹配全部高亮**
  - **Given** 文件名中多处出现查询子串（如 `port` 在 `airport_transport_report.pdf` 中 3 处）
  - **When** 输出该文件
  - **Then** 所有匹配位置全部高亮

- **AC4 — 拼音匹配高亮中文**
  - **Given** 中文文件名 + 拼音查询
  - **When** 搜索「bg」
  - **Then** 高亮命中的中文字符（如「**报告**.pdf」高亮「报告」）

- **AC5 — isatty 自动检测**
  - **Given** 输出被管道 / 重定向
  - **When** `isatty()` 检测到非终端
  - **Then** 自动禁用 ANSI 高亮，输出纯文本

- **AC6 — JSON / NUL 模式**
  - **Given** `--json` 或 `--0` 模式
  - **When** 输出结果
  - **Then** 不包含任何 ANSI codes

**实现**：`Sources/CLI/TerminalFormatter.swift` · **测试**：`Tests/CLITests/TerminalFormatterTests.swift`

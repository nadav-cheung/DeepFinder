### REQ-3.1-01 文件内容分块 📋 🖥️ P0

**用户故事**

作为「RAG 系统」，我希望将文件内容切成合适大小的文本块，以便后续 embedding 阶段能够基于固定粒度的文本单元计算向量。

**验收标准（Given / When / Then）**

- **AC1 — 分块策略**
  - **Given** 一份待索引的文本文件
  - **When** 执行内容分块
  - **Then** 按 256 tokens/chunk、overlap 64 tokens 切分

- **AC2 — 支持格式**
  - **Given** 待分块的文件
  - **When** 识别文件类型
  - **Then** 支持 txt、md、pdf、docx、rtf 及代码文件（swift、py、js、ts、java、go、rs）

- **AC3 — PDF 文本提取**
  - **Given** 一个 PDF 文件
  - **When** 分块
  - **Then** 通过 PDFKit 提取文本后再分块

- **AC4 — docx 文本提取**
  - **Given** 一个 docx 文件
  - **When** 分块
  - **Then** 通过系统框架提取文本后再分块

- **AC5 — 溯源元数据**
  - **Given** 切分出的任意 chunk
  - **When** 查看其属性
  - **Then** 保留源文件 ID 与偏移量，用于后续溯源

- **AC6 — 二进制文件跳过**
  - **Given** 图片 / 视频 / 音频等二进制文件
  - **When** 尝试分块
  - **Then** 跳过，不产生 chunk

### REQ-3.1-02 本地 Embedding 引擎 📋 🖥️ P0

**用户故事**

作为「用户」，我希望问「去年收入增长多少」时本地 Embedding 引擎能将查询转为向量并匹配最相关的文件块，以便全程不联网即可完成语义匹配。

**验收标准（Given / When / Then）**

- **AC1 — 模型选择**
  - **Given** 本地 Embedding 引擎
  - **When** 加载模型
  - **Then** 使用 paraphrase-multilingual-MiniLM-L12-v2 的 CoreML 量化版，模型大小约 470MB，支持 50+ 语言（含中文）

- **AC2 — GPU 加速性能**
  - **Given** M4 GPU 与待 embedding 的 chunk
  - **When** 计算向量
  - **Then** 约 1-2ms/chunk（12 层模型比 6 层略慢）

- **AC3 — 输出维度**
  - **Given** 任一文本 chunk
  - **When** 计算向量
  - **Then** 输出 384 维浮点向量

- **AC4 — 中英文支持**
  - **Given** 中文或英文文本
  - **When** 计算 embedding
  - **Then** 均原生支持（multilingual 模型）

- **AC5 — 首次加载**
  - **Given** 首次使用 Embedding 引擎
  - **When** 触发加载
  - **Then** 从 bundle 加载模型（或预装）

### REQ-3.1-03 向量索引存储 📋 🖥️ P0

**用户故事**

作为「系统」，我希望 50 万个文件 chunk 的向量索引存储在本地，以便查询时以 cosine similarity 在毫秒级返回 Top-K 结果，且用户无感知存储细节。

**验收标准（Given / When / Then）**

- **AC1 — 存储引擎**
  - **Given** 向量索引存储
  - **When** 持久化向量
  - **Then** 使用 SQLite vec 扩展或 hnswlib

- **AC2 — chunk 元数据**
  - **Given** 索引中的每个 chunk
  - **When** 查看记录
  - **Then** 包含向量（384 维）+ 源文件 ID + 偏移量 + 文本预览（50 字）

- **AC3 — 存储上限**
  - **Given** 50 万 chunk 的向量索引
  - **When** 估算占用空间
  - **Then** < 1GB（raw vectors 约 0.77GB + metadata）

- **AC4 — 查询性能**
  - **Given** 已建立的向量索引
  - **When** 发起 Top-10 查询
  - **Then** 延迟 < 50ms

- **AC5 — 增量插入**
  - **Given** 新文件加入索引
  - **When** 更新向量库
  - **Then** 只 embedding 新内容，无需重建全部向量

### REQ-3.1-04 增量 Embedding 更新 📋 🖥️ P1

**用户故事**

作为「用户」，我希望创建或修改新文件后后台自动 embedding 新内容，以便下次搜索即可命中，而不需要重建全部向量。

**验收标准（Given / When / Then）**

- **AC1 — 变更检测**
  - **Given** FSEvents 监听文件系统
  - **When** 检测到文件变更
  - **Then** 只对变更文件执行 embedding

- **AC2 — 文件修改**
  - **Given** 一个已索引文件被修改
  - **When** 处理变更

- **AC3 — 文件删除**
  - **Given** 一个已索引文件被删除
  - **When** 处理变更
  - **Then** 删除其关联的所有 chunk 向量

- **AC4 — 不阻塞搜索**
  - **Given** 正在进行的增量更新
  - **When** 用户发起正常搜索
  - **Then** 增量更新不阻塞搜索流程

### REQ-3.1-05 语义检索 📋 🖥️ P0

**用户故事**

作为「用户」，我希望搜索「团队架构调整方案」时即使文件名为「org_change_2026.pdf」也能通过内容语义匹配命中，以便跨越文件名表述差异定位文件。

**验收标准（Given / When / Then）**

- **AC1 — 全流程延迟**
  - **Given** 用户输入的查询文字
  - **When** 走完查询 → embedding → Top-K 检索全流程
  - **Then** 总延迟 < 100ms

- **AC2 — 结果内容**
  - **Given** 语义检索返回的结果
  - **When** 展示
  - **Then** 包含源文件路径与匹配 chunk 的预览文本

- **AC3 — 与文件名搜索合并**
  - **Given** 同一次查询同时命中文件名与语义匹配
  - **When** 合并结果
  - **Then** 标注匹配类型（文件名 / 语义）

- **AC4 — 排序规则**
  - **Given** 文件名精确匹配与语义匹配并存
  - **When** 排序
  - **Then** 语义匹配结果排在文件名精确匹配之后

- **AC5 — 操作流程**
  - **Given** 用户输入查询文字
  - **When** 触发语义检索
  - **Then** 依次执行：查询 embedding → 向量索引 cosine similarity Top-K → 返回最相关 chunk + 源文件信息 → 结果列表显示匹配文件并标注「语义匹配」

### REQ-3.1-06 本地小模型生成 📋 🖥️ P1

**用户故事**

作为「用户」，我希望问「去年收入增长多少」时本地小模型基于检索到的 chunk 生成附带源文件引用的回答，以便直接获得可追溯的结论性回答。

**验收标准（Given / When / Then）**

- **AC1 — 模型与框架**
  - **Given** 本地生成引擎
  - **When** 加载模型
  - **Then** 使用 Llama 3.2 1B/3B，优先采用 MLX 框架（比 CoreML 更适合 LLM 推理）

- **AC2 — 运行性能**
  - **Given** M4+ 24GB 内存设备
  - **When** 运行生成
  - **Then** 1B 约 150-250 tok/s、3B 约 80-105 tok/s（via MLX）；CoreML 较慢但仍远超 10 tok/s

- **AC3 — 源文件引用**
  - **Given** 生成的回答
  - **When** 展示
  - **Then** 附带源文件路径引用

- **AC4 — 隐私**
  - **Given** 整个生成流程
  - **When** 执行
  - **Then** 完全本地，零外传

- **AC5 — 内存不足降级**
  - **Given** 可用内存不足以运行模型
  - **When** 尝试生成
  - **Then** 降级：跳过生成，只返回检索结果

- **AC6 — 超时处理**
  - **Given** 生成耗时达到 30s
  - **When** 触发超时
  - **Then** 中断生成并显示已生成的部分

- **AC7 — 操作流程**
  - **Given** 用户输入问题
  - **When** 触发生成
  - **Then** 依次执行：语义检索 Top-K 相关 chunk → 本地 LLM 基于检索 chunk 生成回答 → CLI 在搜索结果上方输出回答文本块（v2.0 GUI 在 AI 卡片中显示）→ 回答中的文件路径可点击 / 可复制并跳转到文件

### REQ-3.1-07 RAG 问答 📋 🖥️ P1

**用户故事**

作为「用户」，我希望用自然语言提问即可直接获得基于本地文件内容的回答，以便搜索文件内容而非仅搜索文件名。

**验收标准（Given / When / Then）**

- **AC1 — 自动检测问题**
  - **Given** 用户输入的查询
  - **When** 判断语义
  - **Then** 问号结尾或疑问词开头触发 RAG 问答

- **AC2 — 关键词搜索分流**
  - **Given** 关键词搜索（非问题）
  - **When** 处理查询
  - **Then** 走正常搜索流程，不走 RAG

- **AC3 — 输出形态**
  - **Given** RAG 生成的回答
  - **When** 展示
  - **Then** CLI 输出回答文本 + 引用文件路径列表；v2.0 GUI 回答卡片显示回答文本 + 引用文件列表（可点击）

- **AC4 — 全流程本地**
  - **Given** 整个 RAG 问答流程
  - **When** 执行
  - **Then** 全流程本地执行

- **AC5 — 流式输出**
  - **Given** 回答生成中
  - **When** 推送 token
  - **Then** 显示流式输出（CLI 逐字输出，v2.0 GUI 逐字显示）

- **AC6 — 无相关文件**
  - **Given** 检索未命中任何相关文件
  - **When** 生成回答
  - **Then** 回答「未在本地文件中找到相关信息」

- **AC7 — 操作流程**
  - **Given** 用户在 CLI 输入问题
  - **When** 触发 RAG 问答
  - **Then** 依次执行：自动检测是否为问题（vs 关键词搜索）→ 问题经本地 Embedding → 向量检索 → 本地 LLM 生成回答 → CLI 在结果上方输出回答文本块、引用文件显示为终端可点击路径（v2.0 GUI 回答卡片显示在搜索框下方）→ 下方仍显示相关的文件列表

### REQ-0.3-01 SearchProvider 协议 ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望新的搜索能力（内容搜索、AI 搜索）未来接入时无需学习新界面，以便协议层的扩展对终端体验零成本。

**验收标准（Given / When / Then）**

- **AC1 — 流式查询接口**
  - **Given** 任意 SearchProvider 实现
  - **When** 调用 `search(query:)`
  - **Then** 返回 `AsyncSequence<SearchResult, Never>`，可逐步 yield 结果

- **AC2 — 取消进行中查询**
  - **Given** 一个已发起、尚未完成的查询
  - **When** 调用 `cancel(queryID:)`
  - **Then** 该查询被取消，停止后续结果产出

- **AC3 — 预热**
  - **Given** Provider 尚未首次查询
  - **When** 调用 `prepare()`
  - **Then** 完成预热，首次查询无冷启动惩罚

- **AC4 — MVP 一次性产出**
  - **Given** MVP 的 FileIndexProvider
  - **When** 执行查询
  - **Then** 一次 yield 返回全部结果（内存索引语义）

- **AC5 — 流式 Provider 兼容**
  - **Given** 未来的 AI Provider 或内容搜索 Provider
  - **When** 实现该协议
  - **Then** 接口兼容流式产出，无需改动调用方

**实现**：`Sources/Search/SearchProvider.swift` · **测试**：`Tests/SearchTests/SearchProviderTests.swift`

### REQ-0.3-02 SearchQuery / SearchResult ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望输入 "Report"（大写 R）时结果包含 "report.pdf" 和 "REPORT_FINAL.docx"，以便大小写不影响命中，同时显示时保留原始大小写。

**验收标准（Given / When / Then）**

- **AC1 — 查询规范化**
  - **Given** 任意用户输入
  - **When** 构造 SearchQuery
  - **Then** 执行 NFC 统一化 + lowercased 处理

- **AC2 — 结果结构完整**
  - **Given** 一次命中
  - **When** 构造 SearchResult
  - **Then** 包含 `record` + `provider` + `score`(0.0–1.0) + `matchType`

- **AC3 — MatchType 优先级**
  - **Given** 多种匹配类型同时存在
  - **When** 比较优先级
  - **Then** 顺序为 exact > prefix > pinyin > substring

- **AC4 — 大小写不敏感但显示保留原样**
  - **Given** 输入 "Report" 且索引中含 "report.pdf" / "REPORT_FINAL.docx"
  - **When** 查询
  - **Then** 两者都命中；显示使用 `originalName` 保留原始大小写

**实现**：`Sources/Search/SearchTypes.swift` · **测试**：`Tests/SearchTests/SearchTypesTests.swift`

### REQ-0.3-03 SearchCoordinator (actor) ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望 CLI 或 daemon 发起搜索后立即获得结果，无防抖、无延迟，以便体验如同本地数据库直查。

**验收标准（Given / When / Then）**

- **AC1 — 直接执行**
  - **Given** 内存索引
  - **When** 发起查询
  - **Then** 直接执行，无防抖、无延迟

- **AC2 — 多 Provider 合并流程**
  - **Given** 多个 ready Provider
  - **When** 执行查询
  - **Then** 遍历 ready Providers → 消费 AsyncSequence → 合并结果 → 排序 → 返回

- **AC3 — 自动取消旧查询**
  - **Given** 一个进行中的查询
  - **When** 新查询发起
  - **Then** 旧查询自动取消

- **AC4 — 跨 Provider 去重**
  - **Given** 同一文件被多个 Provider 返回
  - **When** 合并结果
  - **Then** 按 FileRecord.ID 去重，同 ID 取优先级最高的 Provider 结果（优先级：MatchType 权重 > score）

- **AC5 — 搜索不阻塞**
  - **Given** 调用方为 CLI / daemon / GUI 任意上下文
  - **When** 发起查询
  - **Then** 搜索在 actor 隔离上下文执行，不阻塞调用方

- **AC6 — 单调递增 sequence number**
  - **Given** 连续多个查询
  - **When** 分配 sequence number
  - **Then** 单调递增；调用方忽略 stale 查询（sequence number < 当前最新）的结果

- **AC7 — Provider 超时**
  - **Given** 内存索引（5s 超时）或 AI Provider（30s 超时）
  - **When** 超过对应阈值
  - **Then** 取消查询并返回已收到的部分结果

**实现**：`Sources/Search/SearchCoordinator.swift` · **测试**：`Tests/SearchTests/SearchCoordinatorTests.swift`

### REQ-0.3-04 排序策略 ✅ 🖥️ P1

**用户故事**

作为「用户」，我希望搜 "report" 时最上面是精确匹配 "report.pdf"，然后是前缀匹配 "report_v2.pdf"，然后是子串匹配 "quarterly_report.xlsx"，以便最想找的文件排在最前面。

**验收标准（Given / When / Then）**

- **AC1 — 排序权重链**
  - **Given** 多条搜索结果
  - **When** 排序
  - **Then** 权重顺序为 MatchType(最高) > 文件名长度(短优先) > 使用频率(高优先) > 修改时间(新优先) > 路径深度(浅优先)

- **AC2 — tie-break 确定性**
  - **Given** 所有权重均相同的两条结果
  - **When** 排序
  - **Then** 使用 FileRecord.ID 作为最终 tie-break，结果稳定且确定性

- **AC3 — locale-aware 文件名排序**
  - **Given** 含中文 / 日文的文件名
  - **When** 文件名排序
  - **Then** 使用 `String.localizedStandardCompare`（locale-aware，支持中文/日文自然排序）

- **AC4 — 短文件名优先**
  - **Given** 同一 matchType 的 "report.pdf" 与 "annual_report_summary.pdf"
  - **When** 排序
  - **Then** "report.pdf" 排在前面

- **AC5 — 自然排序延后**
  - **Given** v0.3 阶段
  - **When** 处理 file1/file2/file10 序列
  - **Then** 使用 localizedStandardCompare；自然排序延后至 v1.3

- **AC6 — 使用频率预留**
  - **Given** v0.3 MVP 阶段
  - **When** 排序涉及使用频率权重
  - **Then** MVP 可不实现该权重，但 FileRecord schema 预留字段

**实现**：`Sources/Search/SearchSorter.swift` · **测试**：`Tests/SearchTests/SearchSorterTests.swift`

### REQ-0.3-05 性能基准测试 ✅ 🖥️ P0

**用户故事**

作为「用户」，我虽然无感知基准测试，但希望每次发布不引入性能退化，以便速度始终是 #1 优先级。

**验收标准（Given / When / Then）**

- **AC1 — 自动化基准**
  - **Given** XCTMetric + measure blocks
  - **When** 编码为测试
  - **Then** 基准测试自动化运行，纳入 CI

- **AC2 — 100k 索引构建**
  - **Given** 100k 文件
  - **When** 构建索引
  - **Then** 耗时 < 3s

- **AC3 — 1M 索引构建**
  - **Given** 1M 文件
  - **When** 构建索引
  - **Then** 耗时 < 10s

- **AC4 — 前缀查询性能**
  - **Given** 1M 记录
  - **When** 发起前缀查询
  - **Then** p99 < 2ms

- **AC5 — 子串查询性能**
  - **Given** 1M 记录
  - **When** 发起 3+ 字符子串查询
  - **Then** p99 < 1ms；1-2 字符短查询因结果集巨大，p99 < 50ms 且结果集截断至前 1000 条

- **AC6 — 启动加载性能**
  - **Given** 1M 文件已持久化
  - **When** 启动加载
  - **Then** 耗时 < 1s

**实现 / 测试**：`Tests/SearchTests/SearchBenchmarks.swift`

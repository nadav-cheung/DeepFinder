### REQ-0.4-01 DaemonMain ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望安装 DeepFinder 后 daemon 自动在后台运行，以便执行 CLI 查询时 daemon 已就绪、搜索即时返回，无需手动启动 daemon。

**验收标准（Given / When / Then）**

- **AC1 — 启动初始化**
  - **Given** daemon 进程启动
  - **When** 进入 ready 状态前
  - **Then** 长驻后台，加载 SQLite 索引、重建内存索引、启动 FSEventWatcher

- **AC2 — SIGTERM 清理**
  - **Given** daemon 接收到 SIGTERM
  - **When** 执行关闭流程
  - **Then** flush SQLite + 保存 FSEvents cursor + 移除 socket 文件 + 退出，全程在 2 秒内完成

- **AC3 — PID 文件管理**
  - **Given** daemon 启动 / 退出
  - **When** 写入 / 清理 PID
  - **Then** PID 文件位于 `~/.deep-finder/session/daemon.pid`，启动时写入 PID，退出时清理

- **AC4 — 状态机**
  - **Given** daemon 运行周期
  - **When** 状态流转
  - **Then** 经历 starting → ready → live → shutting_down，状态通过 IPC 暴露

- **AC5 — 单例检测**
  - **Given** daemon 已运行（PID 存在且进程存活）
  - **When** 再次尝试启动
  - **Then** 拒绝启动并输出友好提示

- **AC6 — 日志轮转**
  - **Given** daemon 运行
  - **When** 输出日志
  - **Then** 写入 `~/.deep-finder/log/daemon-YYYY-MM-DD.log`，日志文件保留最近 7 天自动轮转

- **AC7 — Stale PID 检测**
  - **Given** PID 文件存在但对应进程不存在
  - **When** 启动时检查
  - **Then** 自动清理 PID 文件并继续启动

**实现**：`Sources/Daemon/DaemonMain.swift` · **测试**：`Tests/DaemonTests/DaemonMainTests.swift`

### REQ-0.4-02 IPCServer ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望运行 `deepfinder "query"` 时 CLI 通过 IPC 连接 daemon 并获得搜索结果，以便 IPC 协议简洁高效且支持多客户端并发。

**验收标准（Given / When / Then）**

- **AC1 — Unix socket 服务器**
  - **Given** daemon 启动
  - **When** 监听 IPC
  - **Then** 在 `~/.deep-finder/session/ipc.sock` 提供 Unix domain socket 服务器

- **AC2 — 协议帧格式**
  - **Given** 一条 IPC 消息
  - **When** 编码为帧
  - **Then** 为 4 字节长度前缀（big-endian UInt32）+ JSON body（Codable-native）

- **AC3 — 请求类型**
  - **Given** CLI 发起请求
  - **When** 选择请求类型
  - **Then** 支持 query / cancel / stats / config_get / config_set / index_status

- **AC4 — 响应类型**
  - **Given** daemon 返回响应
  - **When** 选择响应类型
  - **Then** 支持 results / error / stats / ack / index_status

- **AC5 — 多客户端并发**
  - **Given** 多个 CLI 进程同时连接
  - **When** 并发请求
  - **Then** 基于 Foundation Socket / FileHandle 支持并发连接

- **AC6 — Socket 清理**
  - **Given** daemon 启动时存在残留 socket 文件且 daemon 未运行
  - **When** 启动
  - **Then** 清理残留 socket 文件

- **AC7 — 手动可调试**
  - **Given** daemon 运行中
  - **When** 通过 `nc -U ~/.deep-finder/session/ipc.sock` 发送 JSON 请求
  - **Then** 可手动调试协议

**实现**：`Sources/Daemon/IPCServer.swift` · **测试**：`Tests/DaemonTests/IPCServerTests.swift`

### REQ-0.4-03 IPCProtocol ✅ 🖥️ P0

**用户故事**

作为「系统」，我希望有定义清晰的 IPC 消息协议，以便 CLI 与 daemon 之间通信可靠且向前兼容（用户无直接感知）。

**验收标准（Given / When / Then）**

- **AC1 — Codable 消息类型**
  - **Given** IPC 协议定义
  - **When** 定义消息结构
  - **Then** 提供所有消息类型的 Codable 结构体：IPCRequest（带关联值的 enum）、IPCResponse（带关联值的 enum）、IPCError

- **AC2 — 往返无损**
  - **Given** 任意 IPC 消息
  - **When** JSON 编码后再解码
  - **Then** 零丢失（Codable 自动处理）

- **AC3 — 协议版本**
  - **Given** 新旧 daemon / CLI 共存
  - **When** 通信
  - **Then** 携带协议版本号字段，向前兼容（旧 CLI 可检测新 daemon 版本）

- **AC4 — 协议文档**
  - **Given** 第三方客户端开发者
  - **When** 查阅协议规范
  - **Then** 文档化协议规范，支持第三方客户端开发

- **AC5 — 错误类型细分**
  - **Given** IPC 错误场景
  - **When** 返回错误
  - **Then** 细分为 daemon_not_ready / query_error / invalid_request / permission_denied

**实现**：`Sources/Daemon/IPCProtocol.swift` · **测试**：`Tests/DaemonTests/IPCProtocolTests.swift`

### REQ-0.4-04 Daemon 生命周期管理 ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望开机后 daemon 自动启动（通过 LaunchAgent）或首次 CLI 查询时自动 spawn，以便无需关心 daemon 是否运行。

**验收标准（Given / When / Then）**

- **AC1 — Daemon 运行时的查询路径**
  - **Given** daemon 已运行
  - **When** 用户执行 `deepfinder "query"`，IPCClient 连接 socket
  - **Then** 直接发送查询并获得结果

- **AC2 — Daemon 未运行时自动 spawn**
  - **Given** daemon 未运行
  - **When** 用户执行 `deepfinder "query"`
  - **Then** IPCClient spawn daemon → 等待 ready → 发送查询

- **AC3 — 崩溃重连**
  - **Given** daemon 崩溃、CLI 检测到连接断开
  - **When** 重连
  - **Then** 自动重连（最多重试 3 次，间隔 1s），失败则提示用户

- **AC4 — LaunchAgent 集成**
  - **Given** 用户开机登录
  - **When** launchd 加载 plist
  - **Then** 通过 `~/Library/LaunchAgents/cn.com.nadav.deepfinder.daemon.plist` 自动启动 daemon（v0.7 `deepfinder install` 安装）

- **AC5 — CLI fallback spawn**
  - **Given** 未安装 LaunchAgent
  - **When** 首次查询时检测 daemon 未运行
  - **Then** CLI fallback：直接 spawn daemon 进程（自动启动不依赖 LaunchAgent）

- **AC6 — Stale PID 文件锁**
  - **Given** PID 文件锁存在但进程不存在
  - **When** 检测
  - **Then** 自动清理 stale PID 文件

**实现**：`Sources/Daemon/LaunchAgent.swift`, `Sources/Daemon/DaemonMain.swift` · **测试**：`Tests/DaemonTests/LifecycleTests.swift`

### REQ-0.4-05 ConfigStore ✅ 🖥️ P1

**用户故事**

作为「用户」，我希望通过 CLI `config` 命令管理配置（排除路径、AI 设置等），以便配置持久化且 daemon 重启后保留。

**验收标准（Given / When / Then）**

- **AC1 — 配置持久化**
  - **Given** 用户配置
  - **When** 保存
  - **Then** 持久化到 `~/.deep-finder/config.json`（权限 600）

- **AC2 — 配置项**
  - **Given** ConfigStore
  - **When** 读写配置
  - **Then** 覆盖索引排除路径、热键（v2.0 预留）、AI 设置（v3.0 预留）

- **AC3 — 原子写入**
  - **Given** 写入过程中可能中断
  - **When** 保存配置
  - **Then** 采用 write-to-temp + rename，避免写中断导致损坏

- **AC4 — Schema 版本管理**
  - **Given** 配置文件
  - **When** 读写
  - **Then** 通过 `config_version` 字段进行 schema 版本管理

- **AC5 — IPC 接口**
  - **Given** CLI 需要读写配置
  - **When** 通过 IPC 调用 config_get / config_set
  - **Then** daemon 暴露对应 IPC 接口

- **AC6 — 默认配置回退**
  - **Given** config.json 不存在
  - **When** 启动
  - **Then** 使用代码中内置的默认配置

**实现**：`Sources/Daemon/ConfigStore.swift` · **测试**：`Tests/DaemonTests/ConfigStoreTests.swift`

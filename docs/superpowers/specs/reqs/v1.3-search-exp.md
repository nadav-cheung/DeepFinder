### REQ-1.3-01 Bookmarks — 搜索书签保存与恢复 ✅ 🖥️ P0

**用户故事**

作为「经常执行同一组查询的用户」，我希望能把复杂查询（含排序与过滤）保存为命名书签并一键恢复，以便避免每次重复输入 `report ext:pdf;doc dm:thismonth size:>100kb` 这样的长查询。

**验收标准（Given / When / Then）**

- **AC1 — 添加书签到存储**
  - **Given** 一个 `BookmarkStore`（可选持久化路径 `filePath`）
  - **When** 调用 `add(SearchBookmark)`
  - **Then** 书签（`id`、`name`、`query`、`createdAt`）被加入内存列表；若提供 `filePath` 则原子写入 JSON 文件（权限 600）

- **AC2 — 按 ID 删除书签**
  - **Given** 一个已添加的书签（已知其 UUID）
  - **When** 调用 `remove(id:)`
  - **Then** 该书签被删除；ID 不存在时抛出 `BookmarkError.notFound`

- **AC3 — 列出 / 前缀查找**
  - **Given** 一个或多个已添加书签
  - **When** 调用 `getAll()` / `find(name:)`
  - **Then** `getAll()` 返回全部书签（按插入序）；`find(name:)` 返回名称以给定前缀开头的书签

- **AC4 — 持久化往返**
  - **Given** 通过 `store1`（带 `filePath`）写入书签
  - **When** 用同一 `filePath` 构造 `store2`
  - **Then** `store2.getAll()` 恢复出相同书签（`name`、`query` 字段一致）

- **AC5 — 同名书签允许共存**
  - **Given** 已存在名为 "Same Name" 的书签
  - **When** 再次 `add` 同名书签（不同 `query`）
  - **Then** 不报错；两条记录共存（相等性按 `id` 区分，非按 `name`）

- **AC6 — 损坏文件回退**
  - **Given** `filePath` 指向的 JSON 文件部分损坏（无法解码为 `[SearchBookmark]`）
  - **When** 构造 `BookmarkStore(filePath:)`
  - **Then** 静默回退为空列表（`?? []`），不抛出错误

- **AC7 — 上限**
  - **Given** 已存在 100 个书签（`Constants.Search.maxBookmarks`）
  - **When** 尝试 `add` 第 101 个
  - **Then** 抛出 `BookmarkError.limitExceeded`，不写入新条目

- **AC8 — 端到端 REPL/daemon（2026-06-15 接线）**
  - **Given** daemon 运行（`BookmarkStore` 持久化到 `~/.deep-finder/bookmarks.json`，经 `.bookmarkList`/`.bookmarkSave`/`.bookmarkDelete` IPC 暴露）
  - **When** REPL 执行 `:bm save NAME`（保存上次查询）/ `:bm`（列出，1-based 编号）/ `:bm delete N`（按索引删除）
  - **Then** 书签经 IPC 写入/读取/删除并跨 daemon 重启保留；超限时 daemon 返回 `.error`，未知 id 删除返回 `.error`

> 说明（2026-06-15 更新）：`BookmarkStore` 现已端到端接入——daemon 持久化 + IPC 暴露 + REPL `:bm` 命令（list/save/delete）。仍未实现：single-shot `--bookmark` flag、`:sort` 命令、`:filter` 宏（`SavedFilter` IPC 仍为 forward-compat `.ack`，无 daemon 存储）——未来增强。

**实现**：`Sources/Search/SearchBookmark.swift` · **测试**：`Tests/SearchTests/SearchBookmarkTests.swift`

### REQ-1.3-03 Result Sorting — 多维结果排序 ✅ 🖥️ P0

**用户故事**

作为「需要从大量结果中快速定位特定文件的用户」，我希望能按 name、size、date、extension、path、relevance 六种维度对结果排序，并支持自然排序（数字按数值而非字典序比较），以便在不同场景下以最合适的顺序浏览结果。

**验收标准（Given / When / Then）**

- **AC1 — 支持五种排序键**
  - **Given** 一组搜索结果
  - **When** 调用 `SearchSorter.sort(_:by:)`
  - **Then** 支持 `relevance`（默认）、`name`、`date`、`size`、`natural` 共 5 种 `SortCriterion`

- **AC2 — 自然排序（natural 维度）**
  - **Given** 文件名含数字段（如 `file1`、`file2`、`file10`、`file100`）
  - **When** 按 `.natural` 排序（`SearchSorter.naturalCompare`）
  - **Then** 数字段按数值比较：`file2` < `file10` < `file100`（而非字典序 `file10 < file2`）；输入先 NFC 统一化

- **AC3 — name 排序**
  - **Given** 一组搜索结果
  - **When** 按 `.name` 排序
  - **Then** 按 `localizedStandardCompare`（locale-aware）字母序升序

- **AC4 — size 与 date 排序**
  - **Given** 结果含不同大小或修改日期
  - **When** 按 `.size` / `.date` 排序
  - **Then** size 按数值降序（大到小），date 按时间降序（新到旧）

- **AC5 — relevance 排序策略**
  - **Given** 默认或显式选择 `.relevance` 排序
  - **When** 排序
  - **Then** 依次按：MatchType 优先级 > 文件名长度（短在前）> 修改时间（新在前）> 路径深度（浅在前）> FileRecord.ID（稳定兜底）

- **AC6 — Single-shot CLI 排序 flag**
  - **Given** Single-shot CLI 调用
  - **When** 指定 `--sort name|size|date`（`SortOption`，仅此 3 种）
  - **Then** 客户端按对应 `SortCriterion` 排序；`--reverse` 反转结果；未指定 `--sort` 时保持 daemon 返回顺序

- **AC7 — 无效排序值**
  - **Given** CLI `--sort <无效值>`
  - **When** 解析
  - **Then** 报错 `CLIError.invalidValue(flag: "--sort", value: <无效值>)`

- **AC8 — 边界情况**
  - **Given** 空结果或单条结果
  - **When** 排序
  - **Then** 空数组返回空数组；单条结果不变

- **AC9 — pathDepth 工具**
  - **Given** 任意路径
  - **When** 调用 `SearchSorter.pathDepth(_:)`
  - **Then** 返回路径深度（如 `/a/b/c/x.txt` → 4），供 relevance 排序使用

- **AC10 — 性能（手动目标）**
  - **Given** 大量搜索结果
  - **When** 排序
  - **Then** 报告基准测试 `Tests/SearchTests/SearchBenchmarks.swift`；< 100ms 为手动目标，非回归门禁

> 说明（2026-06-15 更新）：`SortCriterion` 无 `ext`、`path` 维度（仅 relevance/name/date/size/natural）。`--sort` 仅支持 name/size/date 三值。**REPL `:sort` 命令已实现**（in-session）：`:sort [relevance|name|date|size|natural|reverse|clear]`，作用于 REPL 后续查询的客户端排序；`:open`/`:reveal` 按排序后顺序索引。

**实现**：`Sources/Search/SearchSorter.swift` · **测试**：`Tests/SearchTests/SearchSorterTests.swift`、`Tests/SearchTests/NaturalSortTests.swift`

### REQ-1.3-04 Sort Persistence — 排序偏好持久化 ✅ 🖥️ P1

**用户故事**

作为「总是按修改日期降序查看结果的用户」，我希望设置一次排序偏好后，后续所有 REPL 会话和 single-shot 查询自动沿用，直到我再次更改，以便避免每次重复输入 `--sort date --reverse`。

**验收标准（Given / When / Then）**

- **AC1 — 按调用指定（无持久化）**
  - **Given** Single-shot CLI 调用
  - **When** 指定 `--sort <key>` 与可选 `--reverse`
  - **Then** 排序偏好仅在本次调用生效，由 `ArgParser` 解析为 `CLIOptions.sort` / `CLIOptions.reverse`

- **AC2 — 无 flag 时使用 daemon 默认顺序**
  - **Given** CLI 未指定 `--sort`
  - **When** 执行查询
  - **Then** 不进行客户端排序，使用 daemon 返回的（relevance）顺序

- **AC3 — reverse 反转**
  - **Given** 已指定 `--sort <key>`
  - **When** 同时指定 `--reverse`
  - **Then** 排序后结果反转（`sorted.reversed()`）

- **AC4 — 默认值**
  - **Given** 未指定 `--sort`
  - **When** 排序
  - **Then** 等同未设置，使用 daemon 默认 relevance 顺序

> 说明（2026-06-15 更新）：排序偏好的**跨会话持久化仍为未来增强**——`DaemonConfig` 无 `sort` 字段。REPL `:sort` 命令已实现，但偏好仅存于当前会话（进程退出即失效）；single-shot 经 `--sort`/`--reverse` flag 表达。故"跨会话保留 / 立即持久化 / 配置损坏回退"行为尚未实现。

**实现**：`Sources/Daemon/ConfigStore.swift` · **测试**：`Tests/DaemonTests/ConfigStoreTests.swift`

### REQ-1.3-05 Autocomplete — 查询自动补全 ✅ 🖥️ P0

**用户故事**

作为「希望减少输入量的 REPL 用户」，我希望输入部分查询后按 Tab 即时获得补全建议（来源为搜索历史与索引热门文件名），以便像 shell 一样快速完成查询，且整个过程本地完成无网络。

**验收标准（Given / When / Then）**

- **AC1 — Tab 触发补全**
  - **Given** 用户在 REPL 输入了部分文本
  - **When** 按 Tab
  - **Then** 触发自动补全（libedit `rl_attempted_completion_function` → `CompletionContext.complete`）

- **AC2 — 命令名补全（CompletionEngine）**
  - **Given** 当前输入以 `:` 开头（命令模式）
  - **When** 输入 `:st` + Tab
  - **Then** 候选含 `:stats`（及任何前缀匹配的命令；REPL 命令集为 help/quit/stats/config/daemon/open/reveal/explain/dataPreview/undo）

- **AC3 — 过滤器关键字补全（CompletionEngine）**
  - **Given** 当前输入是过滤器前缀
  - **When** 输入 `si` + Tab / `dm` + Tab / `ex` + Tab
  - **Then** 补全为 `size:` / `dm:` / `ext:`（关键字集见 `CompletionEngine.filterKeywords`，含 ext/size/type/dm/dc/path/file/folder/depth/case/regex/len/width/height/duration/pages 等）

- **AC4 — 文件名补全（CompletionEngine，基于上次结果）**
  - **Given** 当前输入是文件名片段，且 `lastResults` 非空
  - **When** 输入 `rep` + Tab
  - **Then** 返回 `lastResults` 中名称以该前缀开头（大小写不敏感）的文件名

- **AC5 — 文件名补全（AutocompleteProvider，基于索引）**
  - **Given** `AutocompleteProvider` 持有 `InMemoryIndex`
  - **When** 调用 `suggest(prefix:)`
  - **Then** 经索引 `search(query:)` 查询，按名称频率（同名不同路径计数）降序、字母序兜底排序，去重后返回至多 `limit`（默认 10）条

- **AC6 — 空前缀返回高频文件名（AutocompleteProvider）**
  - **Given** `AutocompleteProvider` 与非空索引
  - **When** 调用 `suggest(prefix: "")`
  - **Then** 遍历 `allRecords()`，按名称频率降序返回 Top-N 文件名

- **AC7 — 无匹配空数组（AutocompleteProvider）**
  - **Given** 索引中无匹配前缀的文件名
  - **When** 调用 `suggest(prefix:)`
  - **Then** 返回空数组（不报错、不插入）

- **AC8 — 大小写不敏感与 NFC（AutocompleteProvider）**
  - **Given** 索引中存在 `Report.pdf`
  - **When** 调用 `suggest(prefix: "REP")` / `suggest(prefix: "rep")`
  - **Then** 前缀先 NFC 统一化再 lowercased，两者返回相同结果

- **AC9 — :open / :reveal 索引补全（CompletionEngine）**
  - **Given** 用户输入 `:open ` 或 `:reveal `（带尾随空格），且 `lastResults` 非空
  - **When** 按 Tab
  - **Then** 候选为 `"<序号>  <文件名>"` 形式（1-based）

- **AC10 — 性能（手动目标）**
  - **Given** Trie 前缀查询
  - **When** 补全查询
  - **Then** 耗时 < 50ms（O(k)）为手动目标，非回归门禁

> 说明：过滤器关键字补全与命令补全位于 `CompletionEngine`（`Sources/CLI/REPL.swift`），非 `AutocompleteProvider`。`AutocompleteProvider` 仅提供基于索引的文件名补全，无"搜索历史优先"逻辑、无 100ms 超时（其方法为 `async`，无超时裁剪）。

**实现**：`Sources/Search/AutocompleteProvider.swift`、`Sources/CLI/REPL.swift` · **测试**：`Tests/SearchTests/AutocompleteTests.swift`

### REQ-1.3-06 Bookmark & Filter IPC Protocol ✅ 🖥️ P0

**用户故事**

作为「CLI 与 daemon 之间的集成层」（用户无直接感知），我希望有稳定的 IPC 协议在 CLI（single-shot 和 REPL）与 daemon 之间正确传输书签和过滤器操作，以便 CLI 通过 IPC 请求 daemon 执行 CRUD，同时宏定义在 CLI 端缓存以减少往返。

**验收标准（Given / When / Then）**

- **AC1 — Bookmark IPC 请求序列化往返**
  - **Given** `IPCRequest.bookmarkList` / `IPCRequest.bookmarkSave(SearchBookmark)` / `IPCRequest.bookmarkDelete(UUID)`（书签复用 `SearchBookmark`：id、name、query、createdAt；无独立 `IPCBookmark` 类型）
  - **When** Codable 编码后再解码
  - **Then** 字段零丢失

- **AC2 — Bookmark IPC 响应序列化往返**
  - **Given** `IPCResponse.bookmarks([SearchBookmark])`
  - **When** Codable 编码后再解码
  - **Then** 列表与每条 `SearchBookmark`（id、name、query、createdAt）零丢失

- **AC3 — Filter IPC 请求序列化往返**
  - **Given** `IPCRequest.filterList` / `IPCRequest.filterSave(name:expression:)` / `IPCRequest.filterDelete(name:)`
  - **When** Codable 编码后再解码
  - **Then** `name`、`expression` 字段零丢失

- **AC4 — Filter IPC 响应序列化往返**
  - **Given** `IPCResponse.filters([SavedFilter])`（`SavedFilter` 仅含 name、expression；无 createdAt）
  - **When** Codable 编码后再解码
  - **Then** 列表与每条 `SavedFilter` 的 `name`、`expression` 零丢失

- **AC5 — daemon 端处理（前向兼容 ack）**
  - **Given** daemon 收到 bookmark/filter CRUD 请求
  - **When** 处理（`IPCServer.handleRequest`）
  - **Then** 当前返回 `.ack`（前向兼容占位）；daemon 暂不读写 `bookmarks.json` / `filters.json`

> 说明：无独立 `IPCBookmarkRequest/Response/IPCBookmark` 类型——书签直接复用 `SearchBookmark`。`SavedFilter` 仅 `name`+`expression`，无 `createdAt`。无 CLI 端宏缓存加载/刷新逻辑（原 AC3/AC4 描述的"启动加载宏缓存"、"daemon 通知刷新"未实现）。并发安全由 IPC 层 actor 串行化保证（未对 bookmark/filter CRUD 单独验证文件并发写）。

**实现**：`Sources/Daemon/IPCProtocol.swift` · **测试**：`Tests/DaemonTests/IPCProtocolTests.swift`

### REQ-1.3-07 Search Suggestions — 搜索建议 ✅ 🖥️ P1

**用户故事**

作为「刚安装 DeepFinder、不知能搜什么的新用户」，我希望在 REPL 中输入空查询按 Enter 时看到搜索建议（最近搜索、热门文件、语法提示），以便快速上手了解产品能力。

**验收标准（Given / When / Then）**

- **AC1 — 空查询触发建议面板**
  - **Given** 用户在 REPL 中未输入任何内容（空行）
  - **When** 直接按 Enter
  - **Then** 显示搜索建议（"Recent searches" + "Tips"，非搜索结果）

- **AC2 — 最近搜索（本会话）**
  - **Given** 当前 REPL 会话内已执行过查询（`recentQueries` 非空）
  - **When** 生成建议
  - **Then** 取 `recentQueries.suffix(5)`（最近 5 条本会话查询）逐行列出；首行标题 "Recent searches:"

- **AC3 — 语法提示（静态）**
  - **Given** 内置静态示例
  - **When** 生成建议
  - **Then** 固定展示 5 条提示（`ext:pdf size:>1mb`、`"exact phrase"`、`path:~/Documents`、`dupe: | sizedupe:`、`:help`），首行标题 "Tips:"；非随机、每次相同

- **AC4 — 无最近搜索时省略区域**
  - **Given** 本会话 `recentQueries` 为空（如刚启动 REPL 即按 Enter）
  - **When** 生成建议
  - **Then** 不显示 "Recent searches:" 区域（仅显示 "Tips:"）

- **AC5 — 不影响正常搜索流程**
  - **Given** 建议已显示
  - **When** 用户随后输入实际查询
  - **Then** 正常执行搜索，建议不干扰

- **AC6 — 性能（手动目标）**
  - **Given** 建议生成
  - **When** 执行
  - **Then** 耗时 < 10ms（全部内存操作，仅切片 + 字符串拼接）为手动目标，非回归门禁

> 说明：`showSuggestions`（`Sources/CLI/REPL.swift`）仅展示本会话 `recentQueries` 与静态 Tips，无"热门文件 by openCount"区域（`openCount` 字段未实现），无"随机 3 条"逻辑（Tips 为固定 5 条）。"最近搜索"源自进程内 `recentQueries`，非持久化历史文件。

**实现**：`Sources/CLI/REPL.swift`（suggestions panel） · **测试**：`Tests/CLITests/REPLTests.swift`

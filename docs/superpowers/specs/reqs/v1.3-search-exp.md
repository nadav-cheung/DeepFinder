### REQ-1.3-01 Bookmarks — 搜索书签保存与恢复 ✅ 🖥️ P0

**用户场景**：用户经常执行同一组搜索——例如 `report ext:pdf;doc dm:thismonth size:>100kb`。每次手动输入完整查询很繁琐。用户输入 `:bm save monthly-report` 保存当前查询，下次输入 `:bm monthly-report` 一键恢复完整查询（含排序和过滤）。

**操作流程**：
1. 用户输入复杂查询 → 获得结果 → 输入 `:bm save <name>` 保存当前搜索状态
2. 下次进入 REPL → 输入 `:bm <name>` → 恢复完整查询并立即执行
3. 输入 `:bm list` → 查看所有已保存书签
4. 输入 `:bm delete <name>` → 删除不再需要的书签

**书签存储内容**：
- `queryText`：原始查询文本（如 `report ext:pdf;doc dm:thismonth size:>100kb`）
- `sortBy`：排序方式（如 `name`, `size`, `date`）
- `reverse`：是否逆序
- `limit`：结果数量限制
- `createdAt`：创建时间戳

**存储位置**：`~/.deep-finder/bookmarks.json`，权限 600。

**文件格式**：
```json
{
  "version": 1,
  "bookmarks": [
    {
      "name": "monthly-report",
      "queryText": "report ext:pdf;doc dm:thismonth size:>100kb",
      "sortBy": "date",
      "reverse": true,
      "limit": 100,
      "createdAt": "2026-05-30T10:15:00Z"
    }
  ]
}
```

**命名规则**：
- 书签名：字母、数字、`-`、`_` 组合，1-64 字符
- 大小写敏感：`Monthly-Report` 和 `monthly-report` 是不同书签
- 保留名：`save`, `list`, `delete`, `clear` 不可作为书签名

**REPL 命令**：
- `:bm save <name>` — 保存当前查询为书签
- `:bm <name>` — 恢复并执行书签查询
- `:bm list` — 列出所有书签（名称 + 查询摘要 + 创建日期）
- `:bm delete <name>` — 删除指定书签
- `:bm clear` — 删除所有书签（需确认）

**Single-shot CLI 支持**：
- `deepfinder --bookmark monthly-report` — 执行已保存的书签查询
- 书签查询结果与手动输入查询结果一致

**验收标准**：
- [ ] `:bm save <name>` 保存当前查询 + 排序 + 限制到 bookmarks.json
- [ ] `:bm <name>` 恢复书签查询并立即执行搜索
- [ ] `:bm list` 显示所有书签（名称、查询摘要、创建日期）
- [ ] `:bm delete <name>` 删除指定书签
- [ ] `:bm clear` 清空所有书签（需确认）
- [ ] 书签名验证：字母/数字/`-`/`_`，1-64 字符
- [ ] 保留名冲突：`:bm save save` 报错 "保留名不可用作书签名"
- [ ] 重复书签名覆盖旧书签（不报错，静默覆盖）
- [ ] bookmarks.json 损坏时跳过损坏部分，加载有效书签，warning 日志
- [ ] 文件权限 600
- [ ] CLI `--bookmark` flag 支持书签查询
- [ ] 最多 100 个书签，超出时提示删除旧书签

**测试覆盖**：

### REQ-1.3-02 Custom Filters — 自定义过滤器宏 ✅ 🖥️ P0

**用户场景**：用户频繁使用 `ext:pdf;doc;xlsx;pptx dm:thismonth` 组合过滤条件。输入 `:filter save workdocs ext:pdf;doc;xlsx;pptx` 保存为宏，之后查询 `report :workdocs` 等效于 `report ext:pdf;doc;xlsx;pptx`。类似 shell alias，减少重复输入。

**操作流程**：
1. 用户输入 `:filter save <name> <filter-text>` → 保存过滤器宏
2. 用户搜索时输入 `report :<name>` → 宏展开为完整过滤条件
3. 输入 `:filter list` → 查看所有自定义过滤器
4. 输入 `:filter delete <name>` → 删除过滤器

**存储位置**：`~/.deep-finder/filters.json`，权限 600。

**文件格式**：
```json
{
  "version": 1,
  "filters": [
    {
      "name": "workdocs",
      "expansion": "ext:pdf;doc;xlsx;pptx",
      "createdAt": "2026-05-30T10:20:00Z"
    }
  ]
}
```

**展开规则**：
- `:<name>` 在查询文本中被替换为宏展开文本
- 展开在 QueryParser 解析之前执行（文本替换阶段）
- 宏可包含任意有效的过滤语法：`size:>1mb ext:pdf dm:thisweek`
- 宏不可嵌套：`:<name>` 展开后如果包含 `:<other>`，不进行二次展开（防止无限递归）
- 未知的 `:<name>` 报错：`"未知的过滤器宏: ':<name>'。输入 :filter list 查看已定义的宏"`

**命名规则**：
- 过滤器名：字母、数字、`-`、`_` 组合，1-32 字符
- 大小写敏感
- 不可与内置修饰符冲突（`file`, `folder`, `ext`, `size`, `dm`, `dc`, `depth`, `case`, `path`, `parent`, `regex`, `audio`, `video`, `pic`, `doc`）

**REPL 命令**：
- `:filter save <name> <filter-text>` — 保存过滤器宏
- `:filter list` — 列出所有自定义过滤器（名称 + 展开文本）
- `:filter delete <name>` — 删除指定过滤器
- `:filter clear` — 清空所有自定义过滤器（需确认）

**Single-shot CLI 支持**：
- `deepfinder "report :workdocs"` — 宏在 CLI 端展开后发送到 daemon
- CLI 展开宏后打印展开后的查询文本（verbose 模式）

**验收标准**：
- [ ] `:filter save <name> <text>` 保存过滤器宏到 filters.json
- [ ] 查询中 `:<name>` 展开为保存的过滤文本
- [ ] 展开在 QueryParser 之前执行（文本替换）
- [ ] 宏不可嵌套展开（防止无限递归）
- [ ] 未知宏报错
- [ ] 命名冲突检测：内置修饰符名不可用作宏名
- [ ] 过滤器名验证：字母/数字/`-`/`_`，1-32 字符
- [ ] `:filter list` 显示所有过滤器
- [ ] `:filter delete <name>` 删除指定过滤器
- [ ] 文件权限 600
- [ ] 最多 50 个自定义过滤器
- [ ] CLI 查询中的宏展开

**测试覆盖**：
- [ ] 保存 → 查询展开 → 展开文本正确
- [ ] 宏 + 文本搜索组合：`report :workdocs` → `report ext:pdf;doc;xlsx;pptx`
- [ ] 宏包含多个过滤器：`size:>1mb ext:pdf`

### REQ-1.3-03 Result Sorting — 多维结果排序 ✅ 🖥️ P0

**用户场景**：用户搜索 `report` 得到 200 个结果。默认按相关度排序，但用户想按修改日期看最新的——输入 `:sort date` 或 CLI `deepfinder "report" --sort date`。想按文件大小找最大的——`:sort size`。排序支持 name、size、date、extension、path 五种维度。

**操作流程**：
1. 用户输入查询 → 默认按相关度排序输出结果
2. 用户输入 `:sort date` → 当前结果按修改日期重新排序并输出
3. 用户输入 `:sort size reverse` → 按大小降序排列
4. 后续查询自动使用上次选择的排序方式（排序持久化，见 REQ-1.3-04）

**排序维度**：

| 排序键 | 含义 | 排序规则 |
|--------|------|----------|
| `name` | 文件名 | 自然排序（Natural Sort，见下文） |
| `size` | 文件大小 | 数值升序（小到大） |
| `date` | 修改日期 | 时间升序（旧到新） |
| `ext` | 扩展名 | 字母序（无扩展名排在最前） |
| `path` | 完整路径 | 字母序 |
| `relevance` | 相关度（默认） | v0.3 排序策略：MatchType > 文件名长度 > 使用频率 > 修改时间 > 路径深度 |

**自然排序（Natural Sort）**：
- 数字按数值比较：`file2.txt` < `file10.txt` < `file100.txt`
- 与字典序不同：字典序 `file10.txt` < `file2.txt`（"1" < "2"）
- Unicode 数字段整体处理：`报告2.pdf` < `报告10.pdf`
- 相同数字段按后续字符字典序
- 实现使用 `std::strnum::cmp_natural` 等效逻辑或 `NSString.compare(.numeric)` 

**排序方向**：
- 默认升序
- `:sort <key> reverse` 或 `--reverse` 切换降序
- `relevance` 排序下 reverse 无效（相关度只有降序，最高相关在最前）

**REPL 命令**：
- `:sort` — 显示当前排序方式
- `:sort <key>` — 切换排序方式并重新显示当前结果
- `:sort <key> reverse` — 切换排序方式并逆序

**IPC 协议**：
```swift
enum IPSearchSort: String, Codable {
    case relevance    // 默认
    case name
    case size
    case date         // modifiedAt
    case ext          // extension
    case path
}
```

**验收标准**：
- [ ] 支持 6 种排序：relevance, name, size, date, ext, path
- [ ] `name` 排序使用自然排序（数字按数值比较）
- [ ] `size` 按数值排序，`date` 按时间排序
- [ ] `ext` 排序：无扩展名排在最前，扩展名字母序
- [ ] `path` 按完整路径字母序
- [ ] `relevance` 使用 v0.3 排序策略（MatchType > 名称长度 > 频率 > 时间 > 深度）
- [ ] 默认排序为 `relevance`
- [ ] `--reverse` flag 和 `reverse` 关键字切换降序
- [ ] `:sort` 命令显示当前排序方式
- [ ] `:sort <key>` 切换排序并重新输出结果
- [ ] 无效排序键报错：`"未知排序方式: '<key>'。可用: relevance, name, size, date, ext, path"`
- [ ] 性能：10 万条结果排序 < 100ms

**测试覆盖**：
- [ ] 自然排序：`file2 < file10 < file100`
- [ ] 自然排序 + Unicode：`报告2 < 报告10`
- [ ] 按大小排序：0 < 1024 < 1048576
- [ ] 按日期排序：旧日期 < 新日期
- [ ] 按扩展名排序：nil < "doc" < "pdf" < "txt"
- [ ] 按路径排序：`/a/` < `/b/` < `/z/`
- [ ] 相关度排序：exact > prefix > substring
- [ ] reverse 切换方向
- [ ] 无效排序键报错

### REQ-1.3-04 Sort Persistence — 排序偏好持久化 ✅ 🖥️ P1

**用户场景**：用户总是按修改日期降序查看搜索结果。每次搜索都输入 `--sort date --reverse` 很繁琐。用户设置一次 `:sort date reverse` 后，后续所有查询（REPL 和 single-shot）自动按此排序，直到用户再次更改。

**操作流程**：
1. 用户输入 `:sort date reverse` → 排序方式保存到配置
2. 后续查询自动使用 `date reverse` 排序
3. 用户输入 `:sort relevance` → 切换回默认排序，配置更新
4. 下次启动 REPL → 读取配置，恢复上次的排序偏好

**存储位置**：`~/.deep-finder/config.json` 的 `sort` 字段。

**配置格式**：
```json
{
  "sort": {
    "key": "date",
    "reverse": true
  }
}
```

**行为规则**：
- 排序偏好跨 REPL 会话持久化
- Single-shot CLI `--sort` flag 显式指定时覆盖持久化偏好（flag 优先）
- Single-shot CLI 无 `--sort` flag 时使用持久化偏好
- 持久化偏好为 `relevance`（默认）且无 reverse 时，等同未设置
- `:sort` 命令更改立即持久化，无需显式保存

**验收标准**：
- [ ] `:sort <key>` 命令更改后自动持久化到 config.json
- [ ] 排序偏好跨 REPL 会话保留
- [ ] CLI `--sort` flag 优先于持久化偏好
- [ ] CLI 无 `--sort` flag 时使用持久化偏好
- [ ] config.json 无 sort 字段时默认 `relevance` 升序
- [ ] 配置文件损坏时回退默认排序（`relevance`），warning 日志

**测试覆盖**：
- [ ] 设置排序 → 读取 config.json → 值一致
- [ ] 跨会话持久化：设置 → 退出 REPL → 重新进入 → 排序恢复
- [ ] CLI flag 覆盖持久化偏好
- [ ] CLI 无 flag 使用持久化偏好
- [ ] 配置损坏回退默认

### REQ-1.3-05 Autocomplete — 查询自动补全 🔨 🖥️ P0

**用户场景**：用户在 REPL 中输入 `rep` 按 Tab → 看到补全建议 `report`、`report_template`、`reports`。补全基于搜索历史和索引中的热门文件名，本地完成无需网络。减少输入量，提高搜索效率。

**操作流程**：
1. 用户输入部分查询文本 → 按 Tab
2. readline 补全函数查询两个来源：搜索历史 + 索引热门文件
3. 单个匹配 → 直接补全
4. 多个匹配 → 列出候选列表，再次 Tab 循环
5. 无匹配 → 无操作（不报错）

**补全来源**（优先级从高到低）：
1. **搜索历史**：最近使用的查询文本（来自 `~/.deep-finder/history`）
2. **文件名前缀**：索引中匹配前缀的文件名（通过 Trie 前缀查询，O(k)）

**补全规则**：
- 补全基于光标前的当前 token（空格分隔）
- 命令模式（以 `:` 开头）：补全 REPL 命令名（`:st` → `:stats`、`:sort`、`:stop`）
- 过滤器前缀：补全过滤器关键字（`si` → `size:`、`dm` → `dm:`、`ex` → `ext:`）
- 文件名 token：补全文件名（基于 Trie 前缀查询）
- 历史优先：历史中的完整查询优先于文件名补全

**readline 集成**：
- 使用 `rl_completion_matches()` 或 Swift readline wrapper 注册自定义补全函数
- 补全函数在 REPL 进程内执行（通过 IPC 查询 daemon 索引）
- 补全查询超时 100ms（超时返回空，不阻塞输入）

**补全显示**：
- 单个匹配：直接插入，光标移到补全后
- 多个匹配（<=10）：列出所有候选，每行一个
- 多个匹配（>10）：显示前 10 个 + "... (N more)"
- 候选列表不写入 readline 历史

**验收标准**：
- [ ] Tab 触发自动补全
- [ ] 命令名补全：`:st` → `:stats`、`:sort`、`:stop`
- [ ] 过滤器关键字补全：`si` → `size:`、`dm` → `dm:`
- [ ] 文件名补全：`rep` → 基于 Trie 前缀查询的文件名
- [ ] 搜索历史补全优先于文件名补全
- [ ] 单个匹配直接插入
- [ ] 多个匹配列出候选（最多显示 10 个 + 溢出提示）
- [ ] 无匹配时不报错、不插入
- [ ] 补全查询超时 100ms（不阻塞输入）
- [ ] 补全结果 NFC 统一化
- [ ] 性能：补全查询 < 50ms（Trie 前缀查询 O(k)）

**测试覆盖**：
- [ ] 命令补全：输入 `:st` + Tab → 候选含 `:stats`, `:sort`
- [ ] 过滤器补全：输入 `si` + Tab → `size:`
- [ ] 文件名补全：输入 `rep` + Tab → 候选含 "report", "reports"
- [ ] 历史优先：历史有 `report ext:pdf`，输入 `rep` + Tab → 优先补全历史

### REQ-1.3-06 Bookmark & Filter IPC Protocol ✅ 🖥️ P0

**用户场景**：用户无直接感知。书签和过滤器的 IPC 协议确保 CLI（single-shot 和 REPL）与 daemon 之间正确传输书签和过滤器操作。

**IPC 消息类型**：

```swift
// 书签操作
enum IPCBookmarkAction: String, Codable {
    case list
    case save
    case delete
    case clear
    case get       // 按名称获取单个书签
}

struct IPCBookmarkRequest: Codable {
    let action: IPCBookmarkAction
    let name: String?           // save/delete/get 时必填
    let bookmark: IPCBookmark?  // save 时必填
}

struct IPCBookmark: Codable {
    let name: String
    let queryText: String
    let sortBy: IPSearchSort?
    let reverse: Bool?
    let limit: Int?
    let createdAt: Date
}

struct IPCBookmarkResponse: Codable {
    let bookmarks: [IPCBookmark]  // list 时返回全部
    let bookmark: IPCBookmark?    // get 时返回单个
    let success: Bool
    let error: String?
}

// 过滤器操作
enum IPCFilterAction: String, Codable {
    case list
    case save
    case delete
    case clear
}

struct IPCFilterRequest: Codable {
    let action: IPCFilterAction
    let name: String?            // save/delete 时必填
    let expansion: String?       // save 时必填
}

struct IPCCustomFilter: Codable {
    let name: String
    let expansion: String
    let createdAt: Date
}

struct IPCFilterResponse: Codable {
    let filters: [IPCCustomFilter]
    let success: Bool
    let error: String?
}
```

**设计决策**：
- 书签和过滤器存储在 daemon 端（`~/.deep-finder/bookmarks.json` 和 `~/.deep-finder/filters.json`）
- CLI 通过 IPC 请求 daemon 执行 CRUD 操作
- 宏展开在 CLI 端执行（减少 daemon 往返），但 CLI 需先通过 IPC 获取宏定义
- CLI 启动时一次性加载所有宏定义到本地缓存（IPCFilterRequest list），后续展开使用缓存
- 宏定义变更时（save/delete）daemon 通知 CLI 刷新缓存

**验收标准**：
- [ ] IPCBookmarkRequest/Response Codable 序列化往返零丢失
- [ ] IPCFilterRequest/Response Codable 序列化往返零丢失
- [ ] CLI 启动时加载宏定义缓存
- [ ] 宏变更时 CLI 刷新缓存
- [ ] daemon 端 bookmarks.json 和 filters.json 正确读写
- [ ] 并发安全：多个 CLI 同时操作书签/过滤器不损坏文件（daemon actor 串行化）

**测试覆盖**：
- [ ] IPCBookmark Codable 往返
- [ ] IPCFilter Codable 往返
- [ ] daemon 端文件读写
- [ ] CLI 宏缓存加载和刷新
- [ ] 并发操作不损坏文件

**用户场景**：用户刚安装 DeepFinder，不知道能搜什么。输入空查询按 Enter → 显示搜索建议：最近打开的文件、热门查询、随机推荐。帮助新用户快速上手。

1. 用户在 REPL 中输入空查询（直接按 Enter）
2. REPL 显示搜索建议面板（非搜索结果）
3. 建议包含：最近搜索（历史）、热门文件（按 openCount 排序）、提示语法示例
4. 用户看到建议后输入实际查询

**建议内容**：

| 来源 | 数量 | 说明 |
|------|------|------|
| 最近搜索 | 5 条 | 搜索历史中最新的 5 条查询 |
| 热门文件 | 5 条 | 索引中 openCount 最高的 5 个文件（需 FileRecord.openCount 字段，v2.0 完善） |
| 语法提示 | 3 条 | 随机展示高级语法示例（如 `ext:pdf report`, `dm:today`, `size:>1gb`） |

**显示格式**：
```
最近搜索:
  1. report ext:pdf dm:thismonth
  2. *.xlsx size:>1mb
  ...

热门文件:
  1. ~/Documents/report-2026.pdf
  2. ~/Desktop/notes.md
  ...

语法提示:
  • ext:pdf;doc report — 在 PDF 和 DOC 中搜索 "report"
  • dm:today — 今天修改的文件
  • size:>1gb — 大于 1GB 的文件
```

**验收标准**：
- [ ] 空查询显示搜索建议面板
- [ ] 最近搜索从历史文件读取，最多 5 条
- [ ] 语法提示随机展示，每次不同
- [ ] 无历史记录时不显示"最近搜索"区域（而非空列表）
- [ ] 建议面板不影响正常搜索流程（下次输入查询正常执行）
- [ ] 性能：建议生成 < 10ms（全部内存操作）

**测试覆盖**：
- [ ] 空查询触发建议面板
- [ ] 历史读取正确
- [ ] 无历史时不显示该区域
- [ ] 语法提示内容合法
- [ ] 建议后正常搜索不受影响

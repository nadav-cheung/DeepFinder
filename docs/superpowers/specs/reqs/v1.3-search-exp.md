### REQ-1.3-01 Bookmarks — 搜索书签保存与恢复 ✅ 🖥️ P0

**用户故事**

作为「经常执行同一组查询的用户」，我希望能把复杂查询（含排序与过滤）保存为命名书签并一键恢复，以便避免每次重复输入 `report ext:pdf;doc dm:thismonth size:>100kb` 这样的长查询。

**验收标准（Given / When / Then）**

- **AC1 — 保存当前查询为书签**
  - **Given** 用户已在 REPL 执行过一次复杂查询并得到结果
  - **When** 输入 `:bm save <name>`
  - **Then** 当前查询的 `queryText`、`sortBy`、`reverse`、`limit`、`createdAt` 一并写入 `~/.deep-finder/bookmarks.json`（权限 600）

- **AC2 — 恢复并执行书签**
  - **Given** 一个已存在的书签 `monthly-report`
  - **When** 输入 `:bm monthly-report`
  - **Then** 恢复完整查询（含排序和限制）并立即执行搜索，结果与手动输入一致

- **AC3 — 列出 / 删除 / 清空**
  - **Given** 一个或多个已保存书签
  - **When** 输入 `:bm list` / `:bm delete <name>` / `:bm clear`
  - **Then** list 显示所有书签（名称 + 查询摘要 + 创建日期）；delete 删除指定条目；clear 在用户确认后清空全部

- **AC4 — 书签名验证**
  - **Given** 用户尝试保存书签
  - **When** 名称为字母/数字/`-`/`_` 组合且 1-64 字符
  - **Then** 接受；非法名称（含空格、特殊字符或过长）被拒绝

- **AC5 — 保留名冲突**
  - **Given** 用户尝试保存名为 `save`/`list`/`delete`/`clear` 的书签
  - **When** 执行 `:bm save save`
  - **Then** 报错 "保留名不可用作书签名"

- **AC6 — 重复书签名静默覆盖**
  - **Given** 已存在同名书签
  - **When** 再次 `:bm save <同名>`
  - **Then** 不报错，静默覆盖旧书签

- **AC7 — 损坏文件恢复**
  - **Given** `bookmarks.json` 部分内容损坏
  - **When** 加载
  - **Then** 跳过损坏部分，加载有效书签，并输出 warning 日志

- **AC8 — Single-shot CLI 支持**
  - **Given** 已保存书签 `monthly-report`
  - **When** 执行 `deepfinder --bookmark monthly-report`
  - **Then** 执行该书签查询，结果与 REPL 手动输入一致

- **AC9 — 上限**
  - **Given** 已存在 100 个书签
  - **When** 尝试保存第 101 个
  - **Then** 提示删除旧书签，不写入新条目

**实现**：`Sources/Search/SearchBookmark.swift` · **测试**：`Tests/SearchTests/SearchBookmarkTests.swift`

### REQ-1.3-02 Custom Filters — 自定义过滤器宏 ✅ 🖥️ P0

**用户故事**

作为「频繁使用固定过滤条件的用户」，我希望能把诸如 `ext:pdf;doc;xlsx;pptx` 的过滤组合保存为命名宏（类似 shell alias），以便在查询中用 `report :workdocs` 一次性展开，减少重复输入。

**验收标准（Given / When / Then）**

- **AC1 — 保存过滤器宏**
  - **Given** 用户输入 `:filter save <name> <filter-text>`
  - **When** 执行命令
  - **Then** 宏（name、expansion、createdAt）写入 `~/.deep-finder/filters.json`（权限 600）

- **AC2 — 宏展开**
  - **Given** 已保存宏 `workdocs` → `ext:pdf;doc;xlsx;pptx`
  - **When** 用户查询 `report :workdocs`
  - **Then** 查询文本在 QueryParser 解析之前的文本替换阶段展开为 `report ext:pdf;doc;xlsx;pptx`

- **AC3 — 宏可含任意有效过滤语法**
  - **Given** 宏 `bigpdf` → `size:>1mb ext:pdf dm:thisweek`
  - **When** 查询 `report :bigpdf`
  - **Then** 正确展开为多个过滤器组合

- **AC4 — 宏不可嵌套**
  - **Given** 宏 A 展开后包含 `:B` 形式
  - **When** 展开 A
  - **Then** 不对结果中的 `:B` 二次展开（防止无限递归）

- **AC5 — 未知宏报错**
  - **Given** 查询文本中存在未定义的 `:<name>`
  - **When** 展开阶段
  - **Then** 报错：`"未知的过滤器宏: ':<name>'。输入 :filter list 查看已定义的宏"`

- **AC6 — 命名规则与冲突检测**
  - **Given** 用户尝试保存宏
  - **When** 名称为字母/数字/`-`/`_`、1-32 字符，且不与内置修饰符（`file`/`folder`/`ext`/`size`/`dm`/`dc`/`depth`/`case`/`path`/`parent`/`regex`/`audio`/`video`/`pic`/`doc`）冲突
  - **Then** 接受；否则拒绝（命名冲突或非法名称）

- **AC7 — 列出 / 删除 / 清空**
  - **Given** 一个或多个已保存宏
  - **When** 输入 `:filter list` / `:filter delete <name>` / `:filter clear`
  - **Then** list 显示所有过滤器（名称 + 展开文本）；delete 删除指定；clear 在确认后清空

- **AC8 — Single-shot CLI 支持**
  - **Given** 已保存宏 `workdocs`
  - **When** 执行 `deepfinder "report :workdocs"`
  - **Then** 宏在 CLI 端展开后发送到 daemon；verbose 模式下打印展开后的查询文本

- **AC9 — 上限**
  - **Given** 已存在 50 个自定义过滤器
  - **When** 尝试保存第 51 个
  - **Then** 不写入新条目

**实现**：`Sources/Search/SearchFilter.swift` · **测试**：`Tests/SearchTests/SearchFilterTests.swift`

### REQ-1.3-03 Result Sorting — 多维结果排序 ✅ 🖥️ P0

**用户故事**

作为「需要从大量结果中快速定位特定文件的用户」，我希望能按 name、size、date、extension、path、relevance 六种维度对结果排序，并支持自然排序（数字按数值而非字典序比较），以便在不同场景下以最合适的顺序浏览结果。

**验收标准（Given / When / Then）**

- **AC1 — 支持六种排序键**
  - **Given** 一组搜索结果
  - **When** 切换排序方式
  - **Then** 支持 `relevance`（默认）、`name`、`size`、`date`、`ext`、`path` 共 6 种

- **AC2 — 自然排序（name 维度）**
  - **Given** 文件名含数字段（如 `file2.txt`、`file10.txt`、`file100.txt`）
  - **When** 按 `name` 排序
  - **Then** 数字按数值比较：`file2.txt` < `file10.txt` < `file100.txt`（而非字典序 `file10 < file2`）；Unicode 数字段整体处理：`报告2.pdf` < `报告10.pdf`

- **AC3 — size 与 date 数值排序**
  - **Given** 结果含不同大小或修改日期
  - **When** 按 `size` / `date` 排序
  - **Then** size 按数值升序（小到大），date 按时间升序（旧到新）

- **AC4 — ext 排序**
  - **Given** 结果中部分文件无扩展名
  - **When** 按 `ext` 排序
  - **Then** 无扩展名排在最前，其余按扩展名字母序

- **AC5 — path 排序**
  - **Given** 结果含完整路径
  - **When** 按 `path` 排序
  - **Then** 按完整路径字母序

- **AC6 — relevance 排序策略**
  - **Given** 默认或显式选择 relevance 排序
  - **When** 排序
  - **Then** 使用 v0.3 策略：MatchType > 文件名长度 > 使用频率 > 修改时间 > 路径深度

- **AC7 — 排序方向**
  - **Given** 任意排序键
  - **When** 添加 `reverse` 关键字或使用 `--reverse` flag
  - **Then** 切换为降序；但 `relevance` 排序下 reverse 无效（相关度只有降序，最高相关在最前）

- **AC8 — REPL 命令**
  - **Given** 用户在 REPL 中
  - **When** 输入 `:sort`（显示当前排序）/ `:sort <key>`（切换并重新输出结果）/ `:sort <key> reverse`（切换并逆序）
  - **Then** 对应行为生效

- **AC9 — 无效排序键**
  - **Given** 用户输入未知的排序键
  - **When** 执行 `:sort <无效键>`
  - **Then** 报错：`"未知排序方式: '<key>'。可用: relevance, name, size, date, ext, path"`

- **AC10 — 性能**
  - **Given** 10 万条搜索结果
  - **When** 排序
  - **Then** 耗时 < 100ms

**实现**：`Sources/Search/SearchSorter.swift` · **测试**：`Tests/SearchTests/SearchSorterTests.swift`、`Tests/SearchTests/NaturalSortTests.swift`

### REQ-1.3-04 Sort Persistence — 排序偏好持久化 ✅ 🖥️ P1

**用户故事**

作为「总是按修改日期降序查看结果的用户」，我希望设置一次排序偏好后，后续所有 REPL 会话和 single-shot 查询自动沿用，直到我再次更改，以便避免每次重复输入 `--sort date --reverse`。

**验收标准（Given / When / Then）**

- **AC1 — 立即持久化**
  - **Given** 用户在 REPL 中执行 `:sort <key>` 或 `:sort <key> reverse`
  - **When** 命令完成
  - **Then** 排序偏好（key + reverse）自动写入 `~/.deep-finder/config.json` 的 `sort` 字段，无需显式保存

- **AC2 — 跨会话保留**
  - **Given** 上次会话设置了排序偏好
  - **When** 下次启动 REPL
  - **Then** 读取 config.json 恢复上次的排序偏好

- **AC3 — CLI flag 优先**
  - **Given** 已持久化排序偏好
  - **When** Single-shot CLI 显式指定 `--sort` flag
  - **Then** flag 覆盖持久化偏好（flag 优先）

- **AC4 — CLI 无 flag 时使用持久化偏好**
  - **Given** 已持久化排序偏好，且 CLI 未指定 `--sort`
  - **When** 执行查询
  - **Then** 使用持久化偏好

- **AC5 — 默认值**
  - **Given** config.json 无 `sort` 字段，或持久化偏好为 `relevance` 且无 reverse
  - **When** 排序
  - **Then** 等同未设置，默认 `relevance` 升序

- **AC6 — 配置损坏回退**
  - **Given** config.json 损坏
  - **When** 加载
  - **Then** 回退默认排序（`relevance`），输出 warning 日志

**实现**：`Sources/Daemon/ConfigStore.swift` · **测试**：`Tests/DaemonTests/ConfigStoreTests.swift`

### REQ-1.3-05 Autocomplete — 查询自动补全 ✅ 🖥️ P0

**用户故事**

作为「希望减少输入量的 REPL 用户」，我希望输入部分查询后按 Tab 即时获得补全建议（来源为搜索历史与索引热门文件名），以便像 shell 一样快速完成查询，且整个过程本地完成无网络。

**验收标准（Given / When / Then）**

- **AC1 — Tab 触发补全**
  - **Given** 用户在 REPL 输入了部分文本
  - **When** 按 Tab
  - **Then** 触发自动补全

- **AC2 — 命令名补全**
  - **Given** 当前 token 以 `:` 开头（命令模式）
  - **When** 输入 `:st` + Tab
  - **Then** 候选含 `:stats`、`:sort`、`:stop`

- **AC3 — 过滤器关键字补全**
  - **Given** 当前 token 是过滤器前缀
  - **When** 输入 `si` + Tab / `dm` + Tab / `ex` + Tab
  - **Then** 补全为 `size:` / `dm:` / `ext:`

- **AC4 — 文件名补全**
  - **Given** 当前 token 是文件名片段
  - **When** 输入 `rep` + Tab
  - **Then** 基于 Trie 前缀查询（O(k)）返回匹配文件名（如 `report`、`reports`）

- **AC5 — 历史优先**
  - **Given** 搜索历史中存在 `report ext:pdf`，索引中也有以 `rep` 开头的文件名
  - **When** 输入 `rep` + Tab
  - **Then** 历史中的完整查询优先于文件名补全

- **AC6 — 单匹配直接插入**
  - **Given** 补全仅有一个匹配
  - **When** 按 Tab
  - **Then** 直接插入补全文本，光标移到补全后

- **AC7 — 多匹配候选列表**
  - **Given** 补全有多个匹配
  - **When** 按 Tab
  - **Then** ≤10 个时逐行列出；>10 个时显示前 10 个 + "... (N more)"；候选列表不写入 readline 历史

- **AC8 — 无匹配空操作**
  - **Given** 没有任何匹配
  - **When** 按 Tab
  - **Then** 不报错、不插入

- **AC9 — 超时不阻塞**
  - **Given** 补全查询
  - **When** 超过 100ms
  - **Then** 返回空结果，不阻塞输入

- **AC10 — NFC 统一化**
  - **Given** 补全候选
  - **When** 生成
  - **Then** 结果 NFC 统一化

- **AC11 — 性能**
  - **Given** Trie 前缀查询
  - **When** 补全查询
  - **Then** 耗时 < 50ms（O(k)）

**实现**：`Sources/Search/AutocompleteProvider.swift`、`Sources/CLI/REPL.swift` · **测试**：`Tests/SearchTests/AutocompleteTests.swift`

### REQ-1.3-06 Bookmark & Filter IPC Protocol ✅ 🖥️ P0

**用户故事**

作为「CLI 与 daemon 之间的集成层」（用户无直接感知），我希望有稳定的 IPC 协议在 CLI（single-shot 和 REPL）与 daemon 之间正确传输书签和过滤器操作，以便 CLI 通过 IPC 请求 daemon 执行 CRUD，同时宏定义在 CLI 端缓存以减少往返。

**验收标准（Given / When / Then）**

- **AC1 — IPCBookmark 序列化往返**
  - **Given** `IPCBookmarkRequest` / `IPCBookmarkResponse`（含 `IPCBookmark`：name、queryText、sortBy、reverse、limit、createdAt）
  - **When** Codable 编码后再解码
  - **Then** 字段零丢失

- **AC2 — IPCFilter 序列化往返**
  - **Given** `IPCFilterRequest` / `IPCFilterResponse`（含 `IPCCustomFilter`：name、expansion、createdAt）
  - **When** Codable 编码后再解码
  - **Then** 字段零丢失

- **AC3 — CLI 启动加载宏缓存**
  - **Given** CLI 启动
  - **When** 初始化
  - **Then** 一次性通过 IPC（IPCFilterRequest list）加载所有宏定义到本地缓存，后续展开使用缓存

- **AC4 — 宏变更刷新缓存**
  - **Given** 宏定义发生变更（save/delete）
  - **When** daemon 通知 CLI
  - **Then** CLI 刷新本地缓存

- **AC5 — daemon 端文件读写**
  - **Given** daemon 收到书签/过滤器 CRUD 请求
  - **When** 处理
  - **Then** 正确读写 `~/.deep-finder/bookmarks.json` 和 `~/.deep-finder/filters.json`

- **AC6 — 并发安全**
  - **Given** 多个 CLI 进程同时操作书签/过滤器
  - **When** 并发请求
  - **Then** daemon actor 串行化处理，文件不损坏

**实现**：`Sources/Daemon/IPCProtocol.swift` · **测试**：`Tests/DaemonTests/IPCProtocolTests.swift`

### REQ-1.3-07 Search Suggestions — 搜索建议 ✅ 🖥️ P1

**用户故事**

作为「刚安装 DeepFinder、不知能搜什么的新用户」，我希望在 REPL 中输入空查询按 Enter 时看到搜索建议（最近搜索、热门文件、语法提示），以便快速上手了解产品能力。

**验收标准（Given / When / Then）**

- **AC1 — 空查询触发建议面板**
  - **Given** 用户在 REPL 中未输入任何内容
  - **When** 直接按 Enter
  - **Then** 显示搜索建议面板（非搜索结果）

- **AC2 — 最近搜索**
  - **Given** 搜索历史文件存在记录
  - **When** 生成建议
  - **Then** 从历史读取最近 5 条查询并展示

- **AC3 — 热门文件**
  - **Given** 索引中文件含 openCount（v2.0 完善该字段）
  - **When** 生成建议
  - **Then** 按 openCount 排序取前 5 个文件展示

- **AC4 — 语法提示**
  - **Given** 内置语法示例池
  - **When** 生成建议
  - **Then** 随机展示 3 条高级语法示例（如 `ext:pdf;doc report`、`dm:today`、`size:>1gb`），每次不同

- **AC5 — 无历史记录时省略区域**
  - **Given** 搜索历史为空
  - **When** 生成建议
  - **Then** 不显示"最近搜索"区域（而非空列表）

- **AC6 — 不影响正常搜索流程**
  - **Given** 建议面板已显示
  - **When** 用户随后输入实际查询
  - **Then** 正常执行搜索，建议面板不干扰

- **AC7 — 性能**
  - **Given** 建议生成
  - **When** 执行
  - **Then** 耗时 < 10ms（全部内存操作）

**实现**：`Sources/CLI/REPL.swift`（suggestions panel） · **测试**：`Tests/CLITests/REPLTests.swift`

### REQ-0.2-01 FileSystemEventStream 协议 ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望文件创建 / 删除 / 重命名后搜索结果立即反映变化，以便无需手动刷新或等待就能看到最新状态。

**验收标准（Given / When / Then）**

- **AC1 — 协议接口**
  - **Given** FileSystemEventStream 协议
  - **When** 调用其方法
  - **Then** 提供 `start(paths:handler:)`、`stop()`、`isRunning` 三个成员

- **AC2 — 生产实现**
  - **Given** 生产环境
  - **When** 使用 `FSEventStreamImpl`
  - **Then** 内部包装 `FSEventStreamCreate` 提供真实事件流

- **AC3 — 测试替身**
  - **Given** 测试环境
  - **When** 使用 `MockEventStream`
  - **Then** 可编程注入任意事件序列以驱动测试

- **AC4 — 事件类型**
  - **Given** 文件系统发生变化
  - **When** 事件送达 handler
  - **Then** 覆盖创建、删除、重命名、修改四类事件

- **AC5 — 延迟目标**
  - **Given** 文件发生变更
  - **When** 变更被 FSEvents 捕获
  - **Then** 在 ≤ 2 s 内反映在搜索结果中

**实现**：`Sources/FS/FileSystemEventStream.swift`、`Sources/FS/MockEventStream.swift`、`Sources/FS/FSEventStreamImpl.swift` · **测试**：`Tests/FSTests/FileSystemEventStreamTests.swift`

### REQ-0.2-02 FileScanner 全量扫描 ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望首次启动或索引损坏后扫描全部文件时能看到进度条，且扫描过程中已可搜索（边扫边建索引），以便不必等待扫描完成才能开始使用。

**验收标准（Given / When / Then）**

- **AC1 — 遍历所有卷**
  - **Given** 全量扫描启动
  - **When** 使用 `FileManager.enumerator`
  - **Then** 遍历所有可见卷

- **AC2 — 按卷并行扫描**
  - **Given** 多个卷挂载
  - **When** 通过 TaskGroup 调度
  - **Then** 按卷并行扫描，外置卷并行不阻塞主卷

- **AC3 — 边扫边建索引**
  - **Given** 扫描正在进行
  - **When** 已扫描 N 条记录
  - **Then** 该 N 条立即可被搜索命中

- **AC4 — 排除目录**
  - **Given** 可配置的排除规则
  - **When** 扫描
  - **Then** 跳过 `/System`、`/Library`、`.Trash`、`.git`、`node_modules`、`.Spotlight-V100`（可配置）

- **AC5 — 隐私排除**
  - **Given** 隐私敏感目录
  - **When** 扫描
  - **Then** 跳过 `~/Library/Caches`、`~/Library/Cookies`、`~/Library/Keychains`

- **AC6 — 索引范围**
  - **Given** 索引范围约束
  - **When** 扫描
  - **Then** 只索引当前用户 home + 系统共享目录

- **AC7 — 性能目标**
  - **Given** 50 万文件、M4 内置 SSD
  - **When** 全量扫描
  - **Then** 耗时 < 30 s；外置卷 / 网络卷按 I/O 速度比例延长，外置卷以低优先级后台扫描，daemon 日志记录各卷独立进度

- **AC8 — 进度反馈**
  - **Given** 扫描进行中
  - **When** 持续扫描
  - **Then** daemon 日志输出「正在索引... 已扫描 12,345 / ~500,000 文件」，CLI 查询可命中已扫描文件，扫描完成后索引状态转为 live

- **AC9 — 权限拒绝处理**
  - **Given** 某目录无访问权限
  - **When** `FileManager.enumerator` 抛出 `Swift.EncodingError` 或 POSIX EACCES
  - **Then** 捕获并跳过该目录，受限目录计数 +1，继续扫描其他目录

- **AC10 — 受限目录汇总**
  - **Given** 扫描结束
  - **When** 受限目录计数 N > 0
  - **Then** daemon 日志记录「跳过 N 个受限目录」（含路径列表），CLI `:stats` 命令可查询受限目录计数

- **AC11 — Full Disk Access 未授予检测**
  - **Given** 启动时检测受限目录
  - **When** 受限目录 ≥ 3（如 ~/Documents、~/Desktop、~/Downloads 等）
  - **Then** daemon 日志输出警告「部分目录因权限限制无法索引」，IPC `index_status` 返回权限状态；v2.0 GUI 展示提示与「打开系统设置」按钮（跳转 `x-apple.systempreferences:com.apple.preference.security?Privacy_AllFiles`）

- **AC12 — 错误日志策略**
  - **Given** 扫描跳过 / 失败
  - **When** 写入日志
  - **Then** 路径与原因写入 `~/.deep-finder/log/scan-YYYY-MM-DD.log`，格式 `[HH:mm:ss] [WARN] SKIP: {path} — {reason}`，日志文件保留最近 7 天自动轮转

**实现**：`Sources/FS/FileScanner.swift`、`Sources/FS/VolumeManager.swift` · **测试**：`Tests/FSTests/FileScannerTests.swift`、`Tests/FSTests/VolumeManagerTests.swift`

### REQ-0.2-03 FSEventWatcher ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望后台监控持续运行而自己无感知，以便在 Finder 中新建文件后切回 DeepFinder 时新文件已在结果中。

**验收标准（Given / When / Then）**

- **AC1 — 事件映射**
  - **Given** 收到 FSEvents 事件
  - **When** 处理该事件
  - **Then** 创建 → 插入索引；删除 → 移除；重命名 → 删除旧 + 插入新；修改 → 更新元数据

- **AC2 — 启动衔接**
  - **Given** daemon 启动
  - **When** 依次加载持久化索引、启动 FSEvents、补齐 cursor 差距
  - **Then** 索引状态依次经过 stale → verifying → live

- **AC3 — cursor 失效重建**
  - **Given** FSEvents cursor 已失效
  - **When** 检测到失效
  - **Then** 触发全量重建，且不阻塞搜索线程

- **AC4 — 状态机 IPC 暴露**
  - **Given** 索引状态机
  - **When** 通过 IPC 暴露
  - **Then** stale =「结果可能不完整」/ verifying =「验证中...」/ live = 无提示 / error =「索引可能不完整，实时监控已停止」；CLI `:stats` 命令显示该状态

- **AC5 — 启动失败重试**
  - **Given** `FSEventStreamCreate` / `FSEventStreamScheduleWithRunLoop` / `FSEventStreamStart` 返回错误
  - **When** 启动失败
  - **Then** 指数退避重试（初始 2 s，最大 60 s，抖动 ±20%），最多 5 次

- **AC6 — 降级轮询**
  - **Given** 重试全部失败
  - **When** 放弃 FSEvents
  - **Then** 降级为定时轮询：每 30 s 对已索引目录执行增量扫描（对比 mtime 变化），daemon 日志记录「实时监控不可用，使用轮询模式」，IPC `index_status` 返回降级状态

- **AC7 — 运行中异常停止**
  - **Given** 收到 `kFSEventStreamEventFlagUserDropped` / `kFSEventStreamEventFlagKernelDropped`
  - **When** stream 异常停止
  - **Then** 记录丢失事件数并自动重启 stream；若 10 分钟内重启 ≥ 3 次则降级为轮询

- **AC8 — FSEvents 日志策略**
  - **Given** FSEvents 生命周期事件（启动 / 停止 / 重试 / 降级）
  - **When** 写入日志
  - **Then** 写入 `~/.deep-finder/log/fsevents-YYYY-MM-DD.log`，格式 `[HH:mm:ss] [LEVEL] {message}`（LEVEL: INFO/WARN/ERROR），日志文件保留最近 7 天自动轮转

**实现**：`Sources/FS/FSEventWatcher.swift` · **测试**：`Tests/FSTests/FSEventWatcherTests.swift`

### REQ-0.2-04 IndexPersistence ✅ 🖥️ P0

**用户故事**

作为「用户」，我希望退出 app 再打开后搜索即时可用、无需重新扫描，以便获得 < 1 秒的启动体验。

**验收标准（Given / When / Then）**

- **AC1 — 存储配置**
  - **Given** SQLite 数据库
  - **When** 打开
  - **Then** 处于 WAL 模式，路径为 `~/.deep-finder/cache/index.db`，权限 600

- **AC2 — 持久化对象**
  - **Given** 持久化内容
  - **When** 写入
  - **Then** 持久化 `FileRecord[]`（不含索引结构，启动时在内存中重建）

- **AC3 — 增量批量写入**
  - **Given** 待写入变更
  - **When** 积累变更
  - **Then** 每 5 秒或每 100 条变更批量写入一次（避免 I/O 抖动）

- **AC4 — 启动性能**
  - **Given** 100 万条记录
  - **When** 启动加载
  - **Then** 加载耗时 < 1 s，索引重建 < 2 s

- **AC5 — 退出 flush**
  - **Given** daemon 退出
  - **When** 收到退出信号
  - **Then** flush 全部未写入变更并保存 FSEvents cursor

- **AC6 — 完整性校验**
  - **Given** 启动验证
  - **When** 检查数据库完整性
  - **Then** 使用 `PRAGMA integrity_check`（非自定义 checksum）作为主完整性校验

- **AC7 — 磁盘空间不足**
  - **Given** 磁盘空间不足
  - **When** 批量写入触发失败
  - **Then** 停止批量写入，daemon 日志记录警告，继续从内存提供搜索服务（不中断搜索体验），IPC `index_status` 返回磁盘空间不足状态

- **AC8 — Schema 版本管理**
  - **Given** 数据库 schema
  - **When** 应用更新
  - **Then** 通过 `PRAGMA user_version` 跟踪 schema 版本，支持自动迁移

- **AC9 — 迁移事务化**
  - **Given** 执行迁移
  - **When** 在事务内操作
  - **Then** `BEGIN` → 迁移 DDL → 更新 `user_version` → `COMMIT`；失败时 `ROLLBACK` 回退到迁移前状态

- **AC10 — 迁移失败回退**
  - **Given** 迁移事务 ROLLBACK
  - **When** 标记索引为损坏
  - **Then** 触发全量重建（同 REQ-0.2-05 恢复流程），daemon 日志记录「数据库迁移失败，正在重建索引...」，IPC `index_status` 返回重建状态

- **AC11 — 降级场景检测**
  - **Given** `PRAGMA user_version` 高于当前应用版本
  - **When** 检测到该情况
  - **Then** 拒绝自动迁移，daemon 日志记录「索引由更新版本创建，建议删除索引重建」，IPC `index_status` 返回降级检测状态；v2.0 GUI 提供「重建索引」与「退出」按钮

- **AC12 — 单进程访问约束**
  - **Given** index.db 访问模型
  - **When** 文档化其约束
  - **Then** 明确标注仅由单个 DeepFinder 进程访问，不支持并发多进程写入（WAL 模式下多进程写入可能导致锁冲突）

**实现**：`Sources/Persist/IndexPersistence.swift` · **测试**：`Tests/PersistTests/IndexPersistenceTests.swift`

### REQ-0.2-05 索引恢复 ✅ 🖥️ P1

**用户故事**

作为「用户」，我希望异常断电或磁盘错误导致索引损坏时能看到「索引需要修复」并自动修复，以便尽快恢复搜索能力。

**验收标准（Given / When / Then）**

- **AC1 — 启动校验**
  - **Given** 启动时加载 SQLite
  - **When** 执行校验（行数 + checksum）
  - **Then** 校验通过则正常启动

- **AC2 — 校验失败重建**
  - **Given** 校验失败
  - **When** 判定索引损坏
  - **Then** daemon 日志记录「索引损坏，正在重建...」，自动触发全量重建（无需用户手动操作）

- **AC3 — 重建进度**
  - **Given** 全量重建进行中
  - **When** 持续扫描
  - **Then** daemon 日志输出进度（已扫描文件数 / 预估总数），IPC `index_status` 返回重建进度

- **AC4 — 重建完成**
  - **Given** 重建完成
  - **When** 索引就绪
  - **Then** 自动进入 live 状态

- **AC5 — WAL 损坏恢复**
  - **Given** WAL 文件损坏而主 DB 完好
  - **When** 删除 WAL + SHM 文件（`index.db-wal`、`index.db-shm`）后重试加载
  - **Then** 成功则正常启动（无需全量重建）

- **AC6 — Checkpoint 失败**
  - **Given** 写入过程中 WAL 合并（checkpoint）失败
  - **When** 回退策略
  - **Then** 从主 DB 文件单独加载（忽略未 checkpoint 的 WAL 内容），丢失最近未持久化变更，触发增量 FSEvents 补齐

- **AC7 — Schema 版本不兼容**
  - **Given** 应用更新后数据库版本过旧
  - **When** Schema 版本不兼容
  - **Then** 执行迁移事务（见 REQ-0.2-04），迁移失败则标记索引为损坏并触发全量重建

- **AC8 — 残留锁清理**
  - **Given** 恢复流程假设无其他进程正在写入 index.db
  - **When** 检测到 SHM 文件被其他进程持有（flock / POSIX lock）
  - **Then** 等待超时 5 s 后判定为残留锁并强制清理

**实现**：`Sources/Persist/IndexRecovery.swift` · **测试**：`Tests/PersistTests/IndexRecoveryTests.swift`

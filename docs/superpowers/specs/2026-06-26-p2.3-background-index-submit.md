# P2.3 — 索引构建迁移到 daemon 后台 + df-watch 增量热替换 + 进度经 status

> **状态:** 草稿(2026-06-26)。代码级细节待 4 路侦察(daemon-ipc / index-job / watcher-swap / tests-infra)确认后校准。设计意图依据 `docs/architecture.md`、`docs/decisions.md`(ADR-0009)、`docs/tech-selection.md`(F1/F4)、`docs/superpowers/plans/2026-06-24-oss-release.md`(Task 2.1–2.3)。

## 1. 目标

把索引构建从「CLI 前台阻塞」迁移到「daemon 后台执行」:
1. 前台 `deepfind index` 改为向 daemon **提交任务**(立即返回),不再就地阻塞构建。
2. 构建在 daemon 后台跑(df-watch 已有的 `rebuild_and_swap` + ArcSwap 热替换路径),**搜索中途不中断**(已由 ArcSwap 快照 + rename-aside 保证)。
3. 构建进度(已扫文件数 / 已索引字节 / shard 数)**通过 `deepfind status` 查询**。

## 2. 现状(已交付,无需重做)

| 能力 | 现状 | 依据 |
|---|---|---|
| df-watch 文件 watcher(`notify`/FSEvents,去抖) | ✅ 已交付(`mod watch`) | ADR-0009、tech-selection F4 |
| `rebuild_and_swap`:变更 → 全根重扫(复用 `build_content_index`)→ **ArcSwap 热替换** | ✅ 已交付 | ADR-0009、architecture §5 |
| 旧 shard 退役:**rename-aside + drain-then-delete**(映射中绝不 truncate/unlink → 无 SIGBUS) | ✅ 已交付 | ADR-0009、decisions.md |
| DbSet 持于 `Arc<ArcSwap<DbSet>>`;每连接 `load_full()` 快照(在飞搜索持旧 `Arc<DbSet>` 存活) | ✅ 已交付 | OSS 计划 Task 2.1 |
| 启动时对「有 root 无 index」的注册 DB 后台构建 + 热替换(`index_job::spawn_if_missing`) | ✅ 已交付 | OSS 计划 Task 2.3 |
| `is_indexing(db_path)` = 标记文件 `<db>.indexing` 存在;`status` 报 `indexing/missing/fresh/stale` | ✅ 已交付 | OSS 计划 Task 2.3/2.5 |

**结论:** 「搜索中途不中断」这条**已经满足**(ArcSwap 快照 + rename-aside)。P2.3 的真实工作量集中在缺口 §3。

## 3. 缺口(本次要补的 delta)

- **G1 — `deepfind index` 仍就地前台构建。** `crates/deepfind/src/main.rs:358` 直接调用 `df_index::build_content_index(...)`(architecture 附录 A:「Index (cold) … no daemon needed」)。→ 改为**提交**给 daemon;daemon 不可达时回退到就地构建(对齐 search 的 `--direct` 回退、「永不阻塞」原则)。
- **G2 — IPC 无「构建」请求变体。** df-ipc proto 目前只有 `SearchRequest`(architecture §8)。→ 新增 `IndexRequest`(+ ack 响应)。
- **G3 — `index_job` 只会 `spawn_if_missing`(已有 index 就跳过),没有「按需/强制重建」入口,也不写进度。** → 增加 `spawn_build(force)`,并把进度数值周期写进标记文件。
- **G4 — status 进度是布尔级**(`indexing` 标记文件存在与否),无数值。→ 读标记文件里的进度行并展示。
- **G5 — 进度数据源缺失。** `build_content_index` 的 consumer/walker 在内部,外部看不到计数。→ 复用上一轮设计的 `IndexProgress` 原子计数,经 `build_content_index_with_progress` 注入(旧函数保留为委托包装,现有调用方零改动)。

## 4. 设计

### 4.1 进度管线(df-index)— 复用既有原子计数风格

`build_content_index` 内部**已经**用 `Arc<AtomicU32>` 统计 `denied/skipped_binary/skipped_large`。沿用同一风格:

- 新增 `pub struct IndexProgress { files_scanned: AtomicU64, content_bytes: AtomicU64, shards_written: AtomicU64 }` + `Snapshot(Copy)` + `Default`。
- 新增 `build_content_index_with_progress(root, out_db, content_dir, opts, progress: &IndexProgress)`:walker 每条目 `files_scanned.fetch_add(1, Relaxed)`;读入内容时累加 `content_bytes`;consumer 每次 flush shard 累加 `shards_written`。
- **保留** `build_content_index(4 参)` 作为委托包装(throwaway `IndexProgress`)→ 现有 12 个调用方(benches/tests/`spawn_if_missing`)**零改动**。df-core 依旧纯。

### 4.2 index_job 增加按需构建 + 进度写盘(deepfindd)

- `spawn_build(root, db_path, content_dir, opts, dbset, default_db_path, force)`:
  - **并发守卫**:标记文件存在且 `!force` → 返回 `AlreadyIndexing`(不重复起任务)。
  - 起线程:写标记文件 → `build_content_index_with_progress(..., &progress)` → 起一个轻量 reporter(每 ~250ms 把 `progress.snapshot()` 覆写进标记文件,如 `scanning 12345 files · 4.2MB · 3 shards`)→ 构建完成删除标记 → `dbset.store(Arc::new(DbSet::open(&default_db_path)))`(与现有启动路径同一条安全热替换)。
  - 错误路径也要删标记 + 记日志(避免泄漏、避免永久 `indexing`)。
- `spawn_if_missing` 改为 `spawn_build(force=false)` 的薄封装(去重,行为不变)。
- `is_indexing` 不变(标记文件存在);新增 `read_progress(db_path) -> Option<String>` 给 status 用(读标记文件内容,空/无则降级为「indexing」)。

### 4.3 IPC proto(df-ipc)

新增(全部 `#[serde(default)]`,旧/新端可互操作):

```
IndexRequest { root: Option<PathBuf>, force: bool, skip: Vec<String>,
               max_file_size: u64, no_content: bool, one_file_system: bool,
               hidden: bool, db: Option<String> }
IndexResponse { accepted: bool, state: String }   // accepted=false ⇒ "already indexing" 等
```

帧化沿用 `LengthDelimitedCodec`(4B 长度前缀)+ bincode。

### 4.4 daemon 分发(deepfindd lib.rs `handle_conn`)

`handle_conn` 按请求类型分发:收到 `IndexRequest` → 解析 root/db(与 CLI 同样的默认 DB / `--root` / `--db` 解析)→ `index_job::spawn_build(force)` → 回 `IndexResponse{accepted, state}`。复用既有 `dbset.store` 热替换,**不引入第二条 swap 路径**。

### 4.5 CLI `deepfind index` 改写(deepfind main.rs)

- 默认:**连 daemon socket → 发 `IndexRequest` → 打印「已提交,后台索引中;用 `deepfind status` 看进度」→ exit 0**。
- **daemon 不可达 → 回退就地构建**(当前行为),打一行提示。保留 `--foreground` 显式强制就地(给脚本/测试/无 daemon 场景;改动极小)。
- root 解析与今天一致(default_db / `--root` / `--db`),把解析结果放进 `IndexRequest`。

### 4.6 status 进度展示(deepfind main.rs `cmd_status`)

构建在飞时:调 `index_job::read_progress(db_path)` 读标记文件里的进度行,展示如 `indexing: scanning 12345 files · 4.2MB · 3 shards`(无数值则仍显示 `indexing`)。其余状态(missing/fresh/stale + db 统计)不变。**status 读文件,不新增 socket 往返**(与现有 `is_indexing` 跨进程读标记文件一致)。

## 5. 测试计划(TDD,先红后绿)

1. **df-index 单测(确定性):** `build_content_index_with_progress` over tempdir(已知 N 文件)→ `progress.snapshot().files_scanned >= N`、`shards_written` 与最终 report 一致。先红(结构/函数不存在)。
2. **进度格式纯函数单测(确定性):** `render_progress(Snapshot) -> String` 给定快照 → 确定串。
3. **deepfindd 集成(serve.rs,复用既有 harness):**
   - `submit_index_build_populates_index`:发 `IndexRequest` → 标记文件出现 → 轮询查询直到命中 → 断言收敛(镜像现有 `background_build_populates_missing_index`,但由**请求**触发而非启动)。
   - `submit_while_indexing_rejected`:已 in-flight 时再发请求 → `accepted=false`。
4. **deepfind CLI 集成:** daemon 在线 → `deepfind index` 返回提交信息 + daemon 开始构建;daemon 离线 → 回退就地构建(走子进程或抽 dispatch helper 单测)。
5. **在飞搜索不中断:** 复用既有 SIGBUS/热替换覆盖(若已有,引用;否则加一个:起慢查询 → 触发 swap → 断言查询以旧结果完成)。**注意避免 timing flake**:用足量 fixture + 宽松 deadline,失败时按隔离重跑确认(见 memory `df-watch-serve-test-flake`)。

## 7. 风险

- **并发构建守卫**:标记文件存在判断与写入之间有 TOCTOU 窗口 → 用「先写标记(独占 create_new)再起线程」:`OpenOptions::new().write(true).create_new(true)` 成功者才构建,否则 `AlreadyIndexing`。原子,无锁。
- **标记文件竞态**:reporter 覆写 vs status 读 → 进度行用整行覆写 + 读端容忍残缺(读失败降级为 `indexing`),非关键路径。
- **回退确定性**:daemon 离线回退必须与旧行为字节级一致(回归测试守住最终 summary 行)。
- **热替换与在飞搜索**:复用既有 `dbset.store` + rename-aside;**不新增 swap 路径**是关键约束(避免第二条可能不安全的路径)。

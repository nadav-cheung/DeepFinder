## Design Philosophy

**"Speed That Feels Invisible"** — the fastest UI is one the user doesn't notice. Every interaction under 100ms, keyboard-first navigation, zero cognitive overhead for discovery. Visual polish (glow, animations) enhances the feeling of speed rather than detracting from it — Apple's Liquid Glass aesthetic meets Raycast's keyboard efficiency.

**Reference apps** (priority-ordered for influence):
1. **Windows Everything** — sub-second raw search, massive result handling
2. **Raycast** — keyboard-first, action panel, minimal but polished
3. **macOS Spotlight** — Liquid Glass, native macOS integration
4. **Alfred** — fuzzy search, rich feature depth

## Key Architecture Decisions

### 1. REQ-3.2-06 merged into REQ-3.2-37

The original REQ-3.2-06 (keyboard hint tags in empty state) is subsumed by REQ-3.2-37 (bottom action bar). The action bar is always visible (not just empty state), follows the Raycast pattern, and provides better discoverability. The empty state instead shows search history or zero-result suggestions.

### 2. IntelligenceGlow deferred to Phase 3

The 4-layer glow upgrade is visually stunning but functionally cosmetic. With "Speed & Efficiency" as the priority, keyboard navigation and result list performance ship first. The current single-layer glow works acceptably.

### 3. LazyVStack trust — no manual windowing

Per best-practices research: macOS 26 LazyVStack has bidirectional lazy loading with built-in recycling. Manual windowing ("buffer ±5 rows") fights the framework. Instead:
- Use `.equatable()` on row views for fast diff
- Pre-compute AttributedString highlights on background thread
- Fixed 40pt row height for consistent layout

### 4. Liquid Glass — navigation layer only

Apple WWDC25 guidance: `.glassEffect()` is for navigation chrome only. Never applied to result rows (content layer). The search panel background gets glass; rows use standard list styling.

### 5. Keyboard stack priority

```
↑/↓              → selection (primary)
Ctrl+N/Ctrl+P    → selection (alias, Emacs compat)
Type-to-select   → press letter jumps to first matching filename, same-letter cycles
⌥↑/⌥↓           → page jump (~20 lines)
⌘↑/⌘↓           → group jump (when categories active)
Home/End         → first/last result
```

### 6. Search history interaction model

Empty search box + ↑ enters history (Raycast pattern). History and results share the same list area — no overlapping layers. History items show timestamp and are deletable individually.

### 7. Action Panel (⌘K) isolation

When the action panel is open, keyboard focus is trapped within it. ↑↓ navigates actions, not results. Unavailable actions are hidden (not grayed) to keep the panel compact. Each action row shows its keyboard shortcut on the right side.

## Component Architecture

```
  ToastView.swift          ← inline toast for copy confirmation
```

### New Files (Phase 2-3)

```
Sources/GUI/
  ActionPanelView.swift    ← ⌘K action panel (Phase 2)
  ResultCategory.swift     ← UTI-based grouping (Phase 2)
  BottomActionBar.swift    ← fixed footer with shortcuts (Phase 2)
  SearchHistory.swift      ← history persistence + dropdown (Phase 2, may already exist)
  SearchFilterBar.swift    ← filter chips (Phase 3)
  FileDetailView.swift     ← ⌘I detail panel (Phase 3, may already exist)
  AccessHistory.swift      ← usage tracking (Phase 3, may already exist)
```

### Data Flow

```
SearchPanelView
  ├── SearchBarView (text input, spinner, clear button)
  ├── SearchFilterBar (Phase 3)
  ├── ResultsListView
  │   ├── LazyVStack
  │   │   ├── ResultCategory (sticky header, Phase 2)
  │   │   └── ResultRowView × N (equatable, highlight, hover)
  │   ├── BottomActionBar (Phase 2)
  │   └── Result count footer (Phase 2)
  ├── ActionPanelView (⌘K overlay, Phase 2)
  ├── QuickLookPreview (Space/⌘Y)
  ├── FileDetailView (⌘I, Phase 3)
  └── ToastView (inline feedback)

SearchViewModel
  ├── Selection state (selectedIndex, hoverIndex)
  ├── Keyboard routing (↑↓⌥⌘ combinations)
  ├── Search state (query, results, loading)
  └── History/Filter state (Phase 2-3)
```

## Frontend-Design Integration

Each phase uses the `frontend-design` skill iteratively:

1. **Pre-implementation**: Design each component for distinctive, non-generic visuals
2. **Post-implementation**: Review rendered output against design intent
3. **Refinement**: Polish pixel-level details, animation curves, color accuracy

Applied to: ResultRowView, ActionPanelView, EmptyStateView, IntelligenceGlow, SearchPanelView animations, ToastView, BottomActionBar.

## Verification Strategy

Per phase:

**Phase 1**: Keyboard flow end-to-end — open panel → type → navigate → open → close. Timing: all interactions <100ms. Instruments: FPS ≥55fps with 10000 results.

**Phase 2**: History persistence, category grouping accuracy, action panel fuzzy search, result count accuracy.

**Phase 3**: IntelligenceGlow 60fps verified via Instruments. Hover transitions smooth. Detail panel metadata correct.

All phases: reduceMotion graceful degradation, VoiceOver labels, CJK input stability.
